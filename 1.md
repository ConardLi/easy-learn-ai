## 这个仓库是什么？

`garden-skills` 是我整理的一组 Agent Skills。

你可以把 Skill 理解成给 AI Agent 装上的一套专业工作流。它不是一个简单 prompt，而是一份更完整的执行说明：什么时候该问问题、什么时候该产出草稿、什么时候必须让用户 Review、代码怎么写、效果怎么验收、遇到问题怎么返工。

目前仓库里主要有这几个：

- `web-video-presentation`：用来做网页视频、演示视频、课程视频、产品介绍视频
- `web-design-engineer`：用来做网页设计、落地页、Dashboard、交互原型、视觉稿
- `gpt-image-2`：用来做图片生成、图片编辑、海报、产品图、信息图、UI Mockup
- `kb-retriever`：用来做本地知识库检索

这篇主要讲前三个，因为它们也是最近几篇文章里大家问得最多的。

## 视频制作 Skill

先说 `web-video-presentation`。

这个 Skill 主要解决的是：怎么让 Agent 帮你把一篇文章、一个脚本、一个课程内容，做成可以录屏的视频级网页演示。

它不是让 AI 直接给你吐一个 PPT，也不是随便生成几个页面。它的目标更明确：做一个 16:9 的网页视频舞台。

整个页面会按章节和步骤推进，每一屏对应一段讲稿或者一个视觉节奏。你可以用点击或键盘控制进度，录屏时就像在播放一个精心设计过的视频。

适合做这些东西：

- 技术文章视频化
- 产品功能演示
- 课程讲解
- 发布会式演示
- 科普内容
- 知识类短视频
- 公众号文章转视频

我自己最喜欢它的一点是，它不会一上来就闷头写完。它会先把脚本、主题、分镜、章节结构拆出来，让你 Review。确认之后再进入实现。

这件事很重要。因为视频类内容最怕前面方向错了，后面做得再精致也没用。

### 最近更新了什么？

这次更新里，视频 Skill 加了几块比较实用的东西。

第一，新增了多套主题模板。

之前很多人用这个 Skill 的时候，容易做出“能看，但风格不够明确”的东西。现在内置了更多主题方向，比如偏技术架构的、偏杂志排版的、偏课程演示的、偏发布会视觉的。你可以一开始就告诉 Agent 想走哪种风格，它会围绕这个方向生成视觉系统。

第二，做了在线效果预览网站：

[https://mmh1.top/#/ai-application/web-video-presentation](https://mmh1.top/#/ai-application/web-video-presentation)

这个页面可以提前看不同主题大概长什么样。以前大家要先安装、跑起来、生成一遍，才知道效果。现在可以先看预览，再决定要不要用某个主题。

第三，支持自定义 TTS。

视频页面做好以后，很多人下一步就是配音。新版里把 TTS 做成了可插拔的方式。你可以接 MiniMax、OpenAI TTS，也可以按自己的需要接 ElevenLabs、Azure、Google Cloud、edge-tts，甚至 macOS 自带的 `say`。

这块我没有把它做死，因为每个人的声音偏好、预算、语言场景都不一样。Skill 只负责把接口和流程留出来。

### 几个使用建议

如果你想用这个 Skill 做出比较稳定的效果，我建议注意三件事。

第一，模型很关键。

我目前试下来效果最好的是 Opus 4.7。这个 Skill 里有很多审美判断、章节规划、代码实现和返工决策，模型能力不够的话，很容易前面看着还行，越往后越散。

第二，第一轮 Review 一定要认真看。

尤其是脚本拆分、章节结构、视觉方向这几步。不要觉得“先让它做完再说”。视频制作不是普通问答，前期结构会直接影响后面每一屏的质量。

如果第一轮 outline 不满意，就直接改。该删章节删章节，该换风格换风格，该调整节奏就调整节奏。

第三，最后不满意的地方可以局部返工。

很多人第一次用 Agent 做这种复杂页面，会有一个误区：生成完了，不满意，就觉得失败了。

其实不是。

你可以直接告诉 Agent：“第三章的节奏太平了”“这一屏太像 PPT 了”“这里我想更像发布会大屏”“这个转场太生硬”。让它围绕某一章、某一屏、某个组件做几轮调优，效果通常会明显变好。

Skill 本身也是这样迭代出来的。你用得越多，越知道自己想要什么，也越容易把它改成适合自己内容风格的版本。

## 网页设计 Skill

第二个是 `web-design-engineer`。

它和视频制作 Skill 很像，但目标不一样。

视频制作 Skill 关注的是“时间线”和“叙事节奏”：一屏接一屏，像视频一样推进。

网页设计 Skill 关注的是“页面本身”：它要帮你做一个更像真实产品、更有设计判断的网页。

它适合做这些：

- 官网首页
- 产品落地页
- Dashboard
- 活动页
- 作品集
- HTML Slide
- 交互原型
- 数据可视化页面
- 前端 Demo
- UI 概念稿

很多 AI 生成网页最大的问题不是不能用，而是太像 AI 生成的网页。

比如大渐变背景、玻璃拟态卡片、满屏圆角、几张漂浮卡片、一个看起来很热闹但没什么信息密度的 Hero。你一眼就能看出它是模板。

`web-design-engineer` 想解决的就是这个问题。

它会要求 Agent 先理解产品语境，再确定设计系统，然后做 v0，再进入完整实现。中间还会限制一些常见的 AI 审美套路，让它不要每次都掉进同一个坑里。

### 最近更新了什么？

这次更新里，网页设计 Skill 也新增了很多主题模板。

比如偏 Linear 的工具型界面、偏 Aesop 的产品页面、偏 Bloomberg Terminal 的高密度数据界面、偏 Stripe Press 的出版物风格、偏 Mid-Century 的复古海报感，还有一些更适合杂志、仪表盘、艺术展、技术产品的方向。

在线预览地址在这里：

[https://mmh1.top/#/ai-application/web-design-engineer](https://mmh1.top/#/ai-application/web-design-engineer)

这个预览站对我自己也挺有用。因为很多时候你跟 Agent 说“高级一点”“有设计感一点”，它其实不知道你要什么。但你给它一个具体方向，比如“用 aesop 风格做一个护肤品页面”“用 bloomberg-terminal 风格做一个数据看板”，它就更容易收敛。

另外还有一个小坑，也顺手说一下。

Skill 的 `description` 里尽量不要写冒号。

这个坑挺隐蔽。有些 Agent 或 Skill 加载器在解析描述时，对冒号处理不太稳定。你写的时候觉得没问题，但实际触发 Skill 时可能表现很奇怪。现在我会尽量把 description 写得简单一点，少用容易引起解析问题的符号。

## 图片生成 Skill

第三个是 `gpt-image-2`。

这个 Skill 主要是给 GPT Image 2 和兼容 OpenAI 图片接口的环境用的。

图片生成这件事，大家现在都不陌生。但真正用起来会发现，很多失败不是模型不行，而是提示词太随意、需求拆得不够细、没有明确画面结构，也没有考虑后续编辑。

所以我做这个 Skill 的时候，没有只写一份“万能出图 prompt”。

它更像一个图片生成工作台。

它会先判断当前环境能不能直接生成图。如果可以，就走本地生成流程；如果宿主 Agent 自带图片工具，就把提示词整理好交给宿主；如果两者都没有，它就退化成 prompt 顾问，帮你写出更适合拿去出图的提示词。

它适合做这些：

- 海报
- 产品图
- UI Mockup
- 信息图
- 学术图
- 技术架构图
- 漫画分镜
- 头像
- 品牌视觉
- 电商图
- 图片编辑
- 背景替换
- 视觉方案探索

目前里面整理了很多结构化模板，覆盖不同视觉类型。比起“帮我生成一张高级感海报”，我更推荐你让它先帮你拆：画面主体是什么、构图怎么安排、文字区域在哪里、风格参考是什么、哪些元素不能出现、是否需要后续编辑。

图片生成最怕一句话全靠模型猜。猜对了你会觉得很神，猜错了你会觉得它不听话。Skill 的作用就是尽量少让模型猜。

## 为什么我要把这些 Skill 放到一个仓库里？

一开始每篇文章发一个 Skill，其实也能用。

但后来我发现，单个 Skill 分散在文章里，对用户不太友好。

有人想装最新版，不知道去哪里找；有人想看更新记录，只能翻文章；有人想同时用视频、网页、图片几个能力，还要自己复制来复制去。

所以我把它们合到了 `garden-skills`。

现在你可以直接通过仓库安装，也可以只安装其中某一个 Skill。仓库里会持续更新文档、模板、案例和下载链接。

我也会继续把自己真实使用中踩过的坑写进去。

因为 Skill 这种东西，如果只停留在“看起来很强”的 prompt，其实没什么意义。它真正有价值的地方，是把那些反复出现的问题、返工方式、验收标准、模型偏好，慢慢沉淀成一套稳定流程。

## 接下来会更新什么？

后面我会继续沿着几个方向更新：

第一，继续补模板。

视频和网页这两个 Skill，模板越多，越容易让 Agent 找到明确的视觉方向。尤其是那些“不是很常见，但一用就很有辨识度”的风格，我会继续整理进去。

第二，补更多真实案例。

光看文档没什么感觉，案例更直接。一个 Skill 能做出什么效果，最好还是用结果说话。

第三，继续优化安装和使用体验。

现在 Skill 生态还在快速变化，不同 Agent 对 Skill 的加载方式、触发方式、目录结构支持都不完全一样。我会尽量让这个仓库适配更多环境，让大家少踩一点安装层面的坑。

第四，把用户反馈吸收进来。

这几周我看到不少人已经基于这些 Skill 改出了自己的版本。有些改得挺好，我也会参考这些真实用法，把通用部分合回来。
