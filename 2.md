后来我把这些 Skill 整理到一个合集仓库里：

[ConardLi/garden-skills](https://github.com/ConardLi/garden-skills)

写这篇文章时，GitHub 页面显示已经有 6.6K Star，离 7K 也就一步之遥。这个速度有点超出我的预期。

更有意思的是，很多人并没有只把它当成“收藏夹”点个 Star 就走。

有人拿它做课程视频，有人拿它做产品介绍页，有人把它装进自己的 Claude Code、Cursor、Codex 工作流里。也有一些大 V 转发推荐，带来了不少真实反馈。

这些反馈让我更确定了一件事：

Skill 这东西，真正有价值的地方，往往不在提示词写得多漂亮。

它的价值在于，把一套可重复的工作方法交给 Agent。

## Skill 到底解决了什么问题？

很多人第一次用 Agent 做复杂任务，都会遇到同一个问题：

第一次效果很惊艳，第二次就开始飘。

你让它做网页，它可能一会儿像 SaaS 官网，一会儿像课程海报，一会儿又变成一堆渐变卡片。

你让它做视频，它可能前 30 秒节奏很好，后面突然开始堆字、乱切画面、字幕也跟不上。

你让它生成图片，它可能能听懂“赛博朋克”“高级感”“科技感”，可到了可复用的设计场景里，经常缺少结构。

原因很简单。

Agent 默认接到的是一个“任务”，可复杂产物需要的是一条“生产线”。

一个好 Skill 要提供几样东西：

第一，明确的工作流程。

什么时候先问需求，什么时候出草稿，什么时候停下来让用户 Review，什么时候进入实现。

第二，明确的质量标准。

什么算好，什么算普通，什么算 AI 味很重，哪些套路要避开。

第三，明确的可迭代接口。

用户觉得不满意时，要知道该反馈什么，Agent 也要知道该改哪一层。

第四，明确的产物结构。

生成的是 React 项目、网页视频、图片 Prompt、TTS 音频，还是可继续编辑的工程目录。

这几个点组合起来，Skill 才会从“提示词片段”变成“工作系统”。

这也是 garden-skills 想做的事。

它目前收录了视频制作、网页设计、图片生成、知识库检索等几个方向。今天先重点介绍前三个。

## 1. 视频制作 Skill：把文章做成可录制的网页视频

第一个是 `web-video-presentation`。

它适合把文章、脚本、课程、产品 Demo、技术分享，做成 16:9 的网页演示视频。

注意，这里说的“视频”，并不等于直接生成一个 mp4 文件。

它生成的是一个可运行的 Vite + React + TypeScript 项目。页面本身就是一个视频舞台。你可以点击或用键盘推进章节，每一步对应一段旁白、一组画面、一段节奏。

这种方式有几个好处。

画面清晰，适合录屏。

每一帧都能编辑，适合反复打磨。

结构可控，适合长内容。

如果你做过 AI 视频，大概率知道长视频最麻烦的地方：

开头能看，后面崩掉。

画风一变，节奏一乱，前面铺好的东西全散了。

网页视频这条路的优势在于，它把“视频”拆成了网页工程。章节、步骤、旁白、画面、主题、进度控制，全都在代码里。Agent 生成后，你还能继续让它局部修改。

比如你可以说：

“第三章节奏太平了，把画面做得更像发布会 Keynote。”

“这一页文字太多，改成三段式视觉解释。”

“这个概念太抽象，换成系统架构图。”

这类反馈很适合 Agent 执行。

### 最近更新了什么？

视频制作 Skill 最近加了几类重要能力。

首先是主题模板。

现在内置了 23 套主题，覆盖演讲、技术架构、数据报告、科普、终端风、杂志风等不同方向。

以前你让 Agent 做视频，它很容易默认进入一种“通用科技蓝”的风格。看一两个还行，看多了就审美疲劳。

主题模板的作用，是先给 Agent 一套明确的设计坐标。

颜色怎么用，字体怎么选，章节怎么切，画面密度多高，都提前约束好。这样做出来的东西更稳定，也更容易继续迭代。

我也做了一个在线效果预览网站：

[web-video-presentation 在线预览](https://mmh1.top/#/ai-application/web-video-presentation)

你可以先看主题，再决定把哪套风格用到自己的内容里。

其次是支持自定义 TTS。

之前很多人反馈，画面做出来了，旁白还想接自己的语音服务。

现在 Skill 提供了更开放的 TTS 接口。它内置了 MiniMax 和 OpenAI TTS 的示例，同时也给 ElevenLabs、edge-tts、Azure、Google Cloud、macOS say 留了接入方式。

这对团队协作很重要。

因为不同场景对声音的要求差别很大。

课程视频希望稳定清楚，产品视频希望更有情绪，技术报告希望克制一点。TTS 能换，整个工作流就更灵活。

### 使用时最容易踩的坑

第一个坑，是模型选择。

目前我自己测试下来，效果最好的还是 Opus 4.7。尤其是做长内容拆解、章节规划、画面节奏时，它更稳。

视频制作 Skill 里有很多需要判断的地方。

哪些内容适合做成一页？

哪些内容应该拆成两步？

画面该抽象，还是该具象？

这些判断会直接影响最终质量。模型能力差一点，前期结构就容易歪。

第二个坑，是第一轮 Review。

很多人一上来就让 Agent 直接做完整视频，结果跑完发现不满意。

视频类任务一定要重视第一轮 Review。

脚本、主题、章节大纲、视觉方向，在前面定得越清楚，后面返工越少。

尤其是章节大纲，它决定了后续每一页的质量。

如果第一轮大纲已经偏了，后面再怎么修局部，都很难完全舒服。

第三个坑，是别期待一次到位。

更推荐的方式是：

先让 Agent 做完整版本。

整体跑通后，挑不满意的章节继续调。

比如第二章太平、第四章信息太密、结尾没有记忆点，都可以单独反馈。

Agent 很擅长这种局部迭代。

你越能描述自己想要的风格，Skill 后续也越容易被调成适合你的版本。

## 2. 网页设计 Skill：让 Agent 像设计工程师一样工作

第二个是 `web-design-engineer`。

它适合做网页、Landing Page、Dashboard、交互原型、HTML Slide、动画页面、UI Mockup、数据可视化、设计系统探索。

很多人会问：

它和视频制作 Skill 有什么区别？

可以这样理解。

视频制作 Skill 关注的是“按时间推进的表达”。

它要考虑章节、节奏、旁白、镜头感、录制体验。

网页设计 Skill 关注的是“可交互的页面体验”。

它要考虑信息架构、视觉系统、布局、响应式、交互状态、组件细节。

一个偏叙事，一个偏产品。

一个更像导演加前端工程师，一个更像设计师加前端工程师。

所以它们会有交集，比如都能生成网页，都能用 React，都能做视觉表达。

可任务目标不同，工作流也会不同。

`web-design-engineer` 里最关键的设计，是把 Agent 的工作拆成六步：

需求理解。

上下文分析。

设计系统声明。

v0 初稿。

完整实现。

视觉验证。

这套流程看起来普通，实际非常有用。

因为网页设计最怕 Agent 直接开写。

一旦它没有先定义设计系统，就很容易进入“想到哪写到哪”的状态。组件样式不统一，间距忽大忽小，按钮状态缺失，移动端也容易出问题。

设计系统声明能提前锁住几个关键变量：

页面气质是什么？

颜色和字体怎么用？

哪些设计套路要避开？

组件密度多高？

动效要克制还是更有表现力？

这一步做好，后面的实现质量会明显提升。

### 新增主题模板和在线预览

网页设计 Skill 最近也加了很多主题模板。

目前包含 25 套风格配方，比如 Linear、Aesop、Pentagram、Bloomberg、Stripe Press、Mid-Century 等。

这些模板并非简单写一句“高级感”就完事。

每套模板都会包含更具体的设计规则：颜色、字体、版式、标志性动作、适合场景、需要避开的套路。

比如做开发者工具页面，可以用 Linear 风格。

做产品图文叙事，可以参考 Aesop。

做交易台、数据工作站，可以走 Bloomberg Terminal 方向。

做书籍、报告、内容型页面，可以参考 Stripe Press 或 Tufte Data-Ink。

我也放了在线效果预览：

[web-design-engineer 在线预览](https://mmh1.top/#/ai-application/web-design-engineer)

建议使用前先去看一眼。

很多时候，你以为自己要的是“高级科技感”，看完模板会发现，其实你更需要的是信息密度、排版秩序、交互状态和克制的视觉语言。

### 一个很小但很坑的细节

这次更新里还修了一个 Skill 描述的小坑：

Skill 描述里不能有冒号。

这个问题看起来非常小，可它会影响某些 Agent 对 Skill 元信息的解析。

很多人写 Skill 时会把 description 写得很自然：

`description: xxx: yyy`

在部分环境下，这种写法可能带来解析歧义。

所以写 Skill 时，描述字段尽量保持简单、平直、无特殊符号堆叠。尤其是面向多个 Agent 分发时，元信息越朴素越稳。

这类问题也提醒了我：

Skill 工程化之后，提示词质量只是一部分。

目录结构、元数据、版本分发、兼容性、安装方式，同样会影响用户体验。

## 3. 图片生成 Skill：把“会写 Prompt”变成可复用流程

第三个是 `gpt-image-2`。

它面向 GPT Image 2 和 OpenAI 兼容图像 API。

适合做海报、UI Mockup、产品图、信息图、论文图、技术架构图、漫画、头像、分镜、品牌板，以及图片编辑工作流。

很多人对图片生成的理解还停留在一句 Prompt。

比如：

“生成一张科技感海报。”

“做一个高级的产品图。”

“画一个 AI Agent 架构图。”

这种方式能出图，但稳定性有限。

真正做项目时，你很快会遇到更多细节：

尺寸是多少？

主体放在哪里？

文字区域要不要留白？

能不能安全裁切？

风格参考是什么？

要不要分层？

要不要可复用？

要不要适配公众号封面、PPT、官网首屏？

`gpt-image-2` 解决的就是这些问题。

它把图像任务拆成了不同类别，并提供结构化模板。

目前包含 18 大类、79 个结构化 Prompt 模板，还覆盖生成和编辑两类工作流。

它还有三种运行模式：

Garden 本地模式，可以直接调接口出图并落盘。

宿主工具模式，可以把整理好的 Prompt 交给当前 Agent 自带的图像工具。

顾问模式，在没有图像工具时，也能先帮你把 Prompt 写到可执行水平。

这个设计很实用。

因为不同用户的 Agent 环境差异很大。

有人在本地跑，有人用 Codex，有人用 Claude Code，有人只需要 Prompt。Skill 先判断环境，再决定怎么工作，能减少很多隐性失败。

### 图片生成最值得关注的点

我自己最看重的，还是“结构化”。

一个好的图片 Prompt，通常需要同时描述：

画面目标。

主体与关系。

构图。

材质。

光线。

字体与文字限制。

输出尺寸。

后续编辑空间。

很多失败图片，并不因为模型听不懂风格词。更多时候，是任务本身缺少结构。

比如你要做一张系统架构图，只说“现代、清晰、科技感”远远不够。

你要说明有哪些模块，模块之间怎么连接，主次层级是什么，哪些文字必须准确，哪些区域需要留白，画面最终用于哪里。

这就是 Skill 的价值。

它让 Agent 先把任务拆清楚，再进入生成阶段。

## 为什么这些 Skill 会被这么多人收藏？

我复盘了一下，原因大概有几个。

第一，它们都解决了真实高频问题。

视频、网页、图片，都是 AI Agent 用户每天会遇到的产物类型。

第二，它们没有停留在“给你一段提示词”。

每个 Skill 都尽量给到完整流程、检查点、模板、示例和安装方式。

第三，它们对新手友好。

你不需要先理解所有实现细节，也能直接让 Agent 按 Skill 工作。

第四，它们对熟手也有用。

如果你已经会写 Prompt，这些 Skill 仍然能帮你节省流程设计时间。你可以直接改主题、改模板、改约束，把它变成自己的生产线。

这也是我对 Skill 的一个判断：

未来很多 Agent 能力，会以 Skill 的形式沉淀下来。

以前我们复用代码。

后来我们复用组件。

现在开始复用工作流。

一个团队里，优秀成员怎么做需求分析、怎么设计页面、怎么拆解视频、怎么写图片 Prompt，都可以慢慢沉淀成 Skill。

这比单次对话更有长期价值。

## 怎么安装？

仓库里提供了几种安装方式。

最简单的是用 `skills` CLI：

```bash
npx skills add ConardLi/garden-skills
```

只安装某一个 Skill：

```bash
npx skills add ConardLi/garden-skills -s web-design-engineer
```

也可以指定到某个 Agent：

```bash
npx skills add ConardLi/garden-skills -s gpt-image-2 -a claude-code
```

如果你更关注稳定版本，也可以使用 Release 里的 zip 包，或者通过 tag 固定版本。
