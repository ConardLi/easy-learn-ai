# Create Knowledge Site

让一个完全没听过 XXX 的人，读完真的能跟同事讲清这个概念。**这是唯一目标。** 别的规则（反模板 / 交互密度 / 视觉花活）都为它服务 —— 服务不了就让步。

---

## 唯一红线：初学者视角

写每一段、加每一个交互前自问：

- 这段话里的名词，前文铺垫过吗？没有 → 先补一句解释，或者整段重写
- 一个完全没听过 XXX 的人读到这里，会卡住吗？会 → 拆得更碎
- 这个交互能让他「摸到」概念吗？还是只是好看？只是好看 → 砍

新概念第一次出现，**必须**先用一句话「主语 = 谓宾」的陈述句说清。后文才能引用。

---

## 永远不做（工程 + 内容 Hard No）

- **不分 TAB / 不要 Header / Footer / 不要"模块导览"** —— 一长页向下滚就是全部
- **不写任何关于网站本身的话术** —— 「为什么选我们 / 学习路线 / 全面深度 / 让我们一起开启 X 之旅 / 本站包含 N 章」全砍
- **不用 framer-motion / recharts / NextUI / emoji 装饰 / 渐变文字** —— 依赖只留 `react / react-dom / lucide-react`
- **不写产品白皮书腔**（详见下「人话纪律」）
- **不假设读者已知** —— 矩阵 / 梯度 / Transformer / Token / Embedding，除非这站就是讲这些。否则要么补一句铺垫，要么换说法

工程约束见 [`infrastructure.md`](infrastructure.md)。

## 开工流程（按顺序，跳一步等于流水线）

### 1 · 收集资料

数据可靠性优先级：

1. **用户在 prompt 里给的资料**（贴的文档 / 链接 / 课件）—— 主源
2. **WebSearch** 验证关键数据 + 补 2026 时效
3. **自查训练知识** —— 仅作骨架，**所有数字 / 版本号 / 产品状态都要查证**

| 场景 | 做法 |
|---|---|
| 用户给了完整资料 | 以用户资料为主，WebSearch 只补 2026 时效 |
| 用户给了片段 | 用户资料为骨架，WebSearch 补缺 + 验证 |
| 用户完全没给 | WebSearch 收集：1) 定义 2) 主要用法 3) 2026 现状 4) 经典 demo / 数据 |

**任何模型版本号 / benchmark 分数 / 产品发布状态 / 协议标准 → WebSearch，不查不写。**

### 2 · 设计学习路径（先输出给用户对齐！）

动手写代码前，先输出一份「为这个概念设计的循序渐进路径」给用户看一眼。形如：

```
> RAG 学习路径草稿（请你看一眼是否合理）：
>
> 1. RAG 是什么 —— 一句话定义 + 白话解释
> 2. 没有 RAG 时，大模型会怎样（具体翻车例子，引出动机）
> 3. RAG 怎么工作 —— 拆成「检索 + 拼回 + 生成」三步
> 4. 走一遍完整 demo（用户输问题 → 看检索结果 → 看最终回答）
> 5. 关键参数和坑（chunk size、top-K、相似度阈值）
> 6. RAG vs 微调 —— 什么时候选哪个
>
> 你看 OK 吗？或者有要加 / 要砍的？
```

**每一节都要回答：为了听懂下一节，这一节必须讲清什么？** 这是「循序渐进」的本质。

用户对路径有意见 → 改路径再开工，**不要先动代码再回头改结构**。

### 3 · Hero 开场纪律

进站第一秒，用户看到的必须是「XXX 是什么？」的**直白答案** —— 不是钩子、不是反差、不是比喻。

#### Hero 六段骨架（顶到下）

1. **eyebrow**：`概念全称（英文）· 中文` 形式
2. **H1**：直接「XXX 是什么？」3-5 字 + 问号
3. **一句话定义**：display 字体加粗 + 高亮底。15-35 字，完整陈述句
4. **白话 2-3 段**：每段 ≤ 2 句，解释定义里出现的名词
5. **过渡句**：一句话告诉用户下方/右侧的交互卡在玩什么
6. **滚动提示**

#### H1 禁用句型（好钩子但**降到 Section 02 开头**）

| 类型 | 反例 |
|---|---|
| 戏剧反差 | 「它知道很多，但只知道昨天的事」 |
| 反直觉断言 | 「模型从不调用任何函数」 |
| 比喻型主角 | 「老师只说『是金毛』」 |
| 数字震撼 | 「671B 参数只用 5.5%」 |
| 行业感叹 | 「AI 现在不光回答问题，它能做事了」 |

#### 一句话定义的标尺

读完用户应能补全：「啊，原来 XXX = ___」

- 主语 = 概念本身（「知识蒸馏 = …」「MoE = …」）
- 完整陈述句，不带感叹号、反问、省略号
- 不以「想象一下 / 其实 / 不只是 / 不是…而是 / 它只是 / 让我们」开头
- 15-35 字最佳

### 4 · 一节一节写（每写完一节自检 3 问）

- 这节用到了什么名词？前文铺垫过吗？没有 → 补
- 这节读完，用户对 XXX 的理解多了什么？说不出来 → 删
- 这节有 ≥ 1 个可动的东西吗（hover 不算）？没有就想想能不能加

### 5 · 交互设计：辅助理解 ≠ 堆数量

交互只有一个价值：**让用户摸到概念**。所以对号入座：

| 概念形态 | 选什么交互 |
|---|---|
| 有可调参数（学习率 / batch / top-K / chunk / temperature） | slider + 实时曲线 |
| 一个流程 / 算法 / 循环 | 单步 trace（next / prev / reset） |
| 两种东西对比（A vs B） | pill 切换，同输入两边并行 |
| 多个组件组合解锁场景 | 勾选 / 拼装编辑器 |
| 让用户用自己内容看效果 | 输入框 + live preview |

**反过来**：

- 一个无用的 pill 切换不如一个静态 SVG + 准确文字
- 但整站只有 0-1 个真可动元素 = 重做
- 整站 6-8 节，**至少 3 节到 L3+**（拖拽编辑 / 实时反映），**至少出现 4 种不同交互**

完整范式表见 [`patterns.md`](patterns.md)。

### 6 · 反模板：是 nice-to-have，不是红线

如果这个站的最佳教学路径跟某个旧站类似（比如都是讲微调参数，结构相近很自然），**允许接近**。但：

- section 命名用本概念的语言（`SectionWhyChunk` / `SectionRetrievalDance`），不要用通用 `SectionOpening / SectionContrast / SectionCoreLoop`
- 视觉锚（hero 主图色 / 主图元素）尽量跟相邻已建站不同
- 别把上个站的 hero 配色 + 主图布局直接平移过来

> 提示：建议开工前扫一眼 `ai-sites/` 同类目下 2-3 个旧站的 Hero + 整体节奏，避免无意识抄袭。

## 人话纪律

判断一段话是不是 AI 味，自问三个问题：

1. 产品发布会上会说吗？会 → 砍
2. 大词解释小事吗？是 → 砍
3. 口语版差很多吗？差很多 → 用口语那版

**对照感觉**：

> 「赋能企业级智能化决策」 → 「让公司不用每次都翻文档」
> 「探索 X 的核心原理与应用价值」 → 「X 到底咋工作的」
> 「RAG 是一种检索增强生成技术，旨在...」 → 「问之前先翻一眼资料 —— 就这」

句式：短句 > 长句、20 字内最好、一段 ≤ 4 句、多用具体动词（找 / 翻 / 切 / 拼）少用抽象动词（实现 / 赋能 / 驱动）。

完整禁词清单见 [`pitfalls.md`](pitfalls.md)「AI 味写作」段。

## 收尾自检（build 前必做）

### A · 初学者视角（最重要 · 全过才能 build）

1. **Hero 第一秒能让用户答出「XXX = ___」吗？** 答不上 → 重写 Hero 定义
2. **随机挑 3 个名词倒查铺垫**：在站内随手挑 3 个出现的术语，往前找它的定义/解释。找不到 → 补铺垫，或换说法
3. **找个完全没听过 XXX 的同事（或想象一个）读完，能跟别人讲清概念吗？** 在脑子里走一遍他的复述 —— 如果他卡在第 3 段 = 第 3 段重写

### B · 内容质量

4. 哪段是删掉用户也不少学一个核心点？→ 删
5. 数据有没有都能追到来源？没追到 → `WebSearch` 补
6. 有没有 AI 味词？`rg "赋能|重塑|颠覆|范式|探索|旅程|全面|深度|一站式|旨在|致力于|智能化" src/`

### C · 体验

7. 有几个真正可动的元素（hover 不算）？整站 < 3 → 加；section 内连续静态 ≥ 5 句没图 → 改造
8. 相邻 section 是不是用了同种交互？连续两个 pill / 两个 trace / 两个 slider → 至少改一个
9. Hero 主图 / 颜色重心跟相邻已建站撞了吗？撞了 → 换
