# Kiro Vibe Prompt 学习分析

## 一句话定位

这是 Kiro「Vibe」模式（自由编码模式）的系统 prompt：它把同一个 AI IDE 配置成一个轻量、随叫随到、强调心流体验与最小化代码产出的协作伙伴，核心价值在于用大量「人格与语气规范」+「效率约束」来塑造一个克制、不越界、让开发者保持主导权的即时编码助手。

## 为什么值得学习

- 它是「同产品两种模式」的绝佳对照样本：与同目录的 `Spec_Prompt`（严谨的需求→设计→任务阶段门工作流）相比，Vibe 模式去掉了刚性流程，转而靠语气与效率原则来约束 Agent，展示了同一身份在不同任务定位下的 prompt 取舍。
- 它示范了如何用**大段品牌人格描述**去定义一个 AI 产品的「声音」（voice & tone），把抽象的情绪基调写成可执行的写作规则。
- 它把「克制」做成了第一性原则：从「绝对最少量代码」到「不要重复自己」「不要加粗」「不要用 markdown 标题」，是一份关于「如何让 Agent 少说废话」的优秀反向约束清单。
- 它完整承载了产品身份、安全红线、平台上下文与全套特性说明（Steering/Hooks/MCP/Spec），是观察**一份大型 IDE 系统 prompt 如何分层组织**的好样本。

## 结构拆解

### 1. 身份定义

`# Identity` 用四句话锁定身份：既是 AI 助手也是 IDE、由自主流程托管并受人类监督、说话像人不像 bot、并**映照用户的输入风格**。
- **解决的问题**：让模型清楚自己既是对话方又是被外层流程驱动的执行体（输出会被解析成动作）。
- **写法亮点**：比 Spec 模式多了一句 "You reflect the user's input style in your responses."——这是 Vibe 模式追求「贴合感/心流」的伏笔。
- **可复用**：身份段 = 角色 + 运行机制 + 人机关系 + 语气基调。

### 2. 安全边界

`# Rules` 集中了安全与合规红线：拒谈敏感/情感话题、不暴露内部 prompt 与工具、PII 占位符替换、拒绝恶意代码、绝不讨论任何公司在云上的实现细节。
- **解决的问题**：商业 IDE 必须规避法律、隐私与竞品信息风险。
- **写法亮点**：用「重要：/ 绝不要 / 不要」分级，并给出可执行的替换规则（`[name]`、`[email]` 等占位符）。
- **特别值得注意的一条**：「把对话历史中的 execution log 当作你本人真实执行过的操作」——为多轮 Agent 维持状态一致性的技巧，避免模型否认自己之前做过的操作。
- **可复用**：安全规则要「可判定 + 可执行」，而非道德口号。

### 3. 工具调用

工具相关约束散落在 Rules、MCP、Goal 三处：
- `fsWrite` 要「先小写入再 append」，把工具的性能特性写进了 prompt。
- MCP 段定义了**何时不该读配置**（测试工具时先直接调用、出问题才查配置；只有用户明确改配置时才打开 mcp.json），并区分 workspace 级（用文件工具）与 user 级（在工作区外，必须用 bash）。
- Goal 段强调**并行调用工具**：多个独立操作要同时发起，连 `strReplace` 也要拆成独立操作并行调用。
- **写法亮点**：工具说明不是「有哪些工具」，而是「在什么时机、用哪种工具、有何副作用」。
- **可复用**：把工具的使用时机、性能特性、并行策略都规范化，而非只罗列工具清单。

### 4. 回应风格（Vibe 模式的灵魂）

`# Response style` 是这份 prompt 中篇幅最大、也最能体现 Vibe 定位的部分，分两层：
- **人格层**：一长段品牌声音描述——有知识但不说教、像开发者一样说话、果断精确、支持而非权威、温暖友好偶尔开玩笑、随和而非慵懒、传递「心流」的放松无缝感、避免夸张与最高级、「展示而非空谈」。
- **可执行格式层**：简洁直接、不要重复、优先可操作信息、用要点排版、不要 markdown 标题（多步骤除外）、不要加粗、不要提执行日志、写「绝对最少量代码」、多文件脚手架只建最小骨架、用用户语言回复。
- **写法亮点**：把抽象语气（温暖、心流、随和）与可检查的硬规则（不加粗、不用标题、最少代码）分两层写，既塑造人格又可落地。
- **可复用**：语气段落 = 「价值观/声音描述」+「可检查的格式约束」两层结构。

### 5. 效率与目标导向

`# Goal` 强调以最少步骤完成、可直接与用户沟通、意图不清时澄清，并区分「信息类问题直接答」与「操作类任务才动手」，给出了一组示例问题来教模型识别「该直接回答而不是动手做」的情形。
- **解决的问题**：避免 Agent 对纯问答类请求也去执行一堆操作，浪费时间惹恼用户。
- **写法亮点**：用具体示例（"What's the latest version of Node.js?" 等）锚定「这类问题直接说答案」。
- **可复用**：用正例清单划分「回答 vs. 执行」的边界。

### 6. 上下文与平台信息

包含 `System Information`、平台命令指南与示例、`Current date and time`、`Key Kiro Features`、`<OPEN-EDITOR-FILES>` / `<ACTIVE-EDITOR-FILE>` 注入块，以及 `# Current Context` 对「this file」的指代解析。
- **写法亮点**：把运行时上下文（活动文件、系统、日期）以结构化标签注入，并明确「this file」=活动编辑器文件，消除指代歧义。
- **可复用**：用 XML 风格标签承载动态上下文，并显式定义模糊指代的解析规则。

### 7. 失败恢复

Rules 中的「反复失败就解释原因并换方法」是这份 prompt 唯一的失败恢复约束。
- **写法亮点**：相比 Spec 模式有专门的 Troubleshooting 段，Vibe 模式只保留一条最朴素的「换方法」原则，符合其轻量定位。

## 可复用模式

- **声音与语气双层模板**：
  ```
  人格层：我们是 {价值观1}、{价值观2}，不是 {反面}。我们 {如何对待用户}。避免 {夸张/最高级}。
  格式层：简洁直接 / 不重复 / 不加粗 / 不用标题 / 写最少量代码 / 用用户语言回复。
  ```
- **最小化产出原则**：「只写解决需求所需的绝对最少量代码」+「多文件脚手架先给结构概览，再建最小骨架」。
- **回答 vs. 执行边界清单**：用一组示例问题告诉模型「这类只需直接回答，别动手」。
- **工具时机与并行规则**：为每个工具写明「何时用、何时别用、有何副作用」，并要求独立操作并行发起。
- **运行时上下文注入 + 指代解析**：用标签注入活动文件，并定义「this file」的解析规则。

## 不适合直接照抄的地方

- **Kiro 专属设定**：Identity、Steering、Hooks、MCP、Spec、`fsWrite`、`strReplace`、`.kiro/` 目录结构都与产品强绑定，迁移到别的 Agent 需整段替换。
- **品牌人格段**：那一大段 Kiro 声音描述高度定制，照抄会让你的产品「变成 Kiro」。
- **平台/日期信息**：`Operating System: Linux`、`Date: 7/XX/2025`（带占位符 `XX`）是注入的运行时上下文与样本残留，不应作为通用模板。
- **`<OPEN-EDITOR-FILES>` / `<ACTIVE-EDITOR-FILE>` 的 `random.txt`**：是抓取快照时的占位内容，无实际含义。
- **「绝不讨论任何公司在云上的实现」**：这是 AWS/Kiro 的特定竞品与合规考量，未必符合你的场景。
- **execution log 一律视为真实操作**：在可靠性要求高的场景照搬有被历史污染误导的风险，需谨慎。

## 适合进一步拆成课程的点

- 如何为 AI 产品设计「声音与语气」：把抽象人格拆成可执行的写作规则。
- 同一身份的两种模式对比：Vibe（自由、靠语气约束）vs. Spec（严谨、靠阶段门约束），何时选哪种。
- 「最小化代码产出」原则如何写进 prompt 以抑制模型啰嗦。
- 「回答 vs. 执行」边界的示例锚定法。
- 工具调用的并行策略与「时机/副作用」规范。
- 运行时上下文注入与模糊指代（this file）的解析设计。

## 对这个项目的展示建议

- 作为「IDE 助手」分类下**语气/人格驱动型 prompt 的代表案例**，与流程驱动的 `Spec_Prompt` 形成鲜明对照。
- 建议与同目录的 `Spec_Prompt` 并列展示，做成「Kiro 的两种模式」对比卡片：自由 Vibe 编码 vs. 严谨 Spec 流程，是很好的「同产品两种范式」素材。
- 可在详情页高亮三段对照：身份与声音（人格层）、安全规则（合规层）、效率与最小化产出（执行层），让读者直观看到一份 IDE 系统 prompt 的分层。
- 标注来源未核验（`待核验`），提醒读者这是社区抓取的快照而非官方发布版本，且文中含 `7/XX/2025`、`random.txt` 等明显的占位/残留痕迹。
