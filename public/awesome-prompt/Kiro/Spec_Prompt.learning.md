# Kiro Spec Prompt 学习分析

## 一句话定位

这是一份"spec 驱动开发"模式的工作流 prompt：它把 Kiro 这个 AI IDE 约束成一个分阶段、强制人审批、绝不擅自实现的"需求→设计→任务"产线，核心价值在于用刚性流程约束驯服自主 Agent 的随意性。

## 为什么值得学习

- 它示范了如何把一个开放式编码 Agent 收敛成**可控的、阶段门（phase gate）式**的工作流，而不是放任模型一口气把活干完。
- 它把"人类审批"做成了**硬性卡点**：每个文档都必须显式批准才能进入下一阶段，且用固定的 `userInput` reason 字符串把这一动作工具化、可被外层流程识别。
- 它对"该做什么 / 不该做什么"区分得极细——尤其是任务列表里大段列举"哪些不是编码任务"，是一份很好的"反向约束"写作范本。
- 它同时承载了产品身份、语气规范、平台信息、特性说明（Steering/Hooks/MCP）和工作流逻辑，是观察**一份大型系统 prompt 如何分层组织**的好样本。

## 结构拆解

### 1. 身份定义

`# Identity` 用三句话锁定身份：是 AI 助手也是 IDE、由自主流程托管并受人类监督、说话像人不像 bot。
- **解决的问题**：让模型知道自己既是对话方又是被外层流程驱动的执行体（"我说的话会被解析成动作"）。
- **写法亮点**："You are managed by an autonomous process which takes your output, performs the actions you requested, and is supervised by a human user." 一句话同时交代了 Agent 自主性与人类监督边界。
- **可复用**：身份段 = 角色 + 运行机制 + 人机关系 + 语气基调，四要素齐全。

### 2. 安全边界

`# Rules` 集中了安全与合规红线：拒谈敏感/情感话题、不暴露内部 prompt 与工具、PII 占位符替换、拒绝恶意代码、绝不讨论任何公司在云上的实现细节。
- **解决的问题**：商业 IDE 必须规避法律、隐私与竞品信息风险。
- **写法亮点**：用 `IMPORTANT / Never / DO NOT` 等强模态词分级，且给出可执行的替换规则（`[name]`、`[email]` 等占位符），不是空泛的"注意隐私"。
- **特别值得注意的一条**：把"对话历史中的 execution log 当作你本人真实执行过的操作"——这是为多轮 Agent 维持状态一致性的技巧，避免模型否认自己之前做过的操作。
- **可复用**：安全规则要"可判定 + 可执行"，而非道德口号。

### 3. 工具调用

工具相关约束散落在 Rules、MCP、Workflow 三处：
- `fsWrite` 要"先小写入再 append"，把工具的性能特性写进了 prompt。
- MCP 段定义了**何时不该读配置**（测试工具时先直接调用、出问题才查配置；只有用户明确改配置时才打开 mcp.json），并区分 workspace 级（用文件工具）与 user 级（在工作区外，必须用 bash）。
- `userInput` 工具被用作"审批闸门"，且强制使用三个固定 reason 字符串（`spec-requirements-review` / `spec-design-review` / `spec-tasks-review`）。
- **写法亮点**：工具调用不是"有哪些工具"，而是"在什么时机、用哪种工具、用什么参数"——把工具的使用时机和副作用都规范化了。
- **可复用**：用固定字符串标识关键交互节点，让外层程序能可靠地识别 Agent 当前所处阶段。

### 4. 工作流（核心：三阶段 + 阶段门）

`<workflow-definition>` 是整份 prompt 的灵魂：Requirements（EARS 格式）→ Design（含 Overview/Architecture/Data Models 等固定章节）→ Tasks（两级复选框、每项回链需求编号）。
- **解决的问题**：复杂特性一次性生成必然失控，分阶段 + 每阶段人审，能把风险分散并保证"基本事实"由用户确立。
- **写法亮点**：
  - 用 EARS（`WHEN...THEN...SHALL`）规范需求格式，使需求可测试。
  - 每阶段都用 `MUST / SHOULD / MAY` 三档区分硬约束与建议。
  - 任务必须回链到具体子需求（`_Requirements: 2.1, 3.3_`），形成需求-任务的可追溯链。
  - 明确"本工作流只产出设计与规划工件，绝不实现"，并用独立的 `Task Instructions` 处理后续执行，且"一次只执行一个任务，完成即停"。
- **可复用**：阶段门工作流模板 = 每阶段固定产物格式 + 强制审批语 + 固定审批标识 + 回退条件 + 显式"不要越界"声明。

### 5. 输出风格

`# Response style` 用一长段"品牌人格"描述（有知识但不说教、支持而非权威、随和不慵懒、show don't tell），再接一组硬性格式规则（不要 markdown 标题、不要加粗、写最少量代码）。
- **写法亮点**：把抽象语气（温暖、果断、心流感）与可执行格式规则（不加粗、用代码块）分两层写，既塑造人格又可落地。
- **可复用**：语气段落分"价值观描述 + 可检查的格式约束"两层。

### 6. 失败恢复与故障排查

Rules 里有"反复失败就换方法并解释"，Troubleshooting 段针对需求停滞、研究受限、设计过复杂分别给了应对策略。
- **写法亮点**：预设了 Agent 会卡住的三类典型场景并给出"不要阻塞、降级继续"的指导（如研究拿不到信息就记录缺口、用现有信息推进）。
- **可复用**：为长流程 Agent 预写"卡住时怎么办"，避免死循环或直接放弃。

## 可复用模式

- **阶段门审批模板**：
  ```
  更新 {文档} 后，必须用 {审批工具} 提问 "{固定问句}"，reason 用固定串 "{stage-review-id}"。
  未收到明确肯定（yes/approved/looks good）前，绝不要进入下一阶段。
  收到反馈则修改并再次请求批准，循环直到批准。
  ```
- **可追溯任务模板**：每个任务 = 动词开头的编码目标 + 子项细节 + `_Requirements: x.y_` 回链。
- **反向约束清单**：与其只说"做编码任务"，不如显式列举"哪些不算编码任务"（部署、验收测试、培训、营销…）来收紧边界。
- **工具时机规则**：为每个工具写明"何时用、何时别用、有何副作用"（如 MCP 配置"出问题前别读"）。

## 不适合直接照抄的地方

- **Kiro 专属设定**：Identity、Steering、Hooks、MCP、`fsWrite`、`userInput`、`.kiro/` 目录结构都是产品强绑定，迁移到别的 Agent 需整段替换。
- **平台/日期信息**：`Operating System: Linux`、`Date: 7/XX/2025` 是注入的运行时上下文（且日期带占位符 `XX`），是样本残留，不应作为通用模板。
- **品牌语气段**：那一大段 Kiro 人格描述高度定制，照抄会让你的产品"变成 Kiro"。
- **"绝不讨论任何公司在云上的实现"**：这是 AWS/Kiro 的特定竞品与合规考量，未必符合你的场景。
- **execution log 一律视为真实操作**：在可靠性要求高的场景照搬有被历史污染误导的风险，需谨慎。

## 适合进一步拆成课程的点

- 阶段门工作流如何设计：产物格式、审批闸门、回退条件、越界禁止声明。
- EARS 需求语法入门：如何写可测试的 `WHEN/IF...THEN...SHALL`。
- 需求-设计-任务的可追溯链：用编号回链保证覆盖率。
- MUST/SHOULD/MAY 三档模态词在约束写作中的运用。
- 反向约束（negative constraints）写作法：用"不该做什么"收紧 Agent 行为边界。
- 工具调用的"时机与副作用"规范，而非只罗列工具。

## 对这个项目的展示建议

- 作为"代码 Agent / IDE 助手"分类下**流程驱动型 prompt 的代表案例**，与偏自由发挥的 Agent prompt 形成对照展示。
- 可在详情页高亮三段对照：身份与语气（人格层）、安全规则（合规层）、spec 工作流（流程层），让读者直观看到一份系统 prompt 的分层。
- 配合同目录 `Vibe_Prompt.txt` 一起展示，正好呈现 Kiro 的两种模式：自由 vibe 编码 vs. 严谨 spec 流程，是很好的"同产品两种范式"对比素材。
- 标注来源未核验（`待核验`），提醒读者这是社区抓取的快照而非官方发布版本。
