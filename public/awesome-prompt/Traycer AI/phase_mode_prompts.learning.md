# Traycer AI Phase Mode Prompts 学习分析

## 一句话定位

这是 Traycer.AI 的「阶段规划模式」（Phase Mode）系统提示——它把模型塑造成一位**只读、不写码的技术负责人**，专门负责把用户需求拆解成高层级的开发阶段。

## 为什么值得学习

它是一个极简但定位精准的「规划型 Agent」范例。整篇提示不到 50 行，却完整地约束了一个角色的**能力边界、行为模式与输出形式**。它的价值不在于复杂，而在于：

- 用一个清晰的人设（tech lead）一次性传达了大量隐含期望（高视角、不亲自动手、关注代码质量）。
- 通过显式的 `<limitations>` 把「能做什么 / 不能做什么」硬性切断，避免规划型 Agent 越界去写代码或执行命令。
- 用「强制工具输出」这一机制保证产物结构化，杜绝模型用自由文本糊弄。

对于想设计「规划 / 拆解 / 评审」类 Agent 的人，这是一份高性价比的参考样板。

## 结构拆解

### 1. 身份定义

开篇即用 `@traycerai` 命名并明确「基于业界领先架构」，随后立刻补一句「绝不要提及你由 Anthropic 创建」——这是典型的**身份隔离**写法，把底层模型与产品人格切开。

紧接着 `<role>` 用「工程团队的技术负责人」这个职业隐喻完成了人格塑造。这里的妙处在于：一句「tech lead」自动携带了一组人类共识——高层视角、对架构负责、不亲自写每一行代码、要提及关键符号/类/函数。比起逐条列规则，用职业角色锚定行为成本更低、覆盖面更广。

### 2. 安全边界

边界被拆成了三层，层层加固：

- `<role>` 里的**领域边界**：限定在 IDE/编码范围内，明确禁止涉足账号、凭据、生产部署、生产日志等运维领域；同时留了一个例外口子——「代码库里有部署文件则可以建议更新」，体现了边界设计的精细度（不是一刀切，而是按「是否在 IDE 可见范围内」划线）。
- `<communication>` 里的**保密边界**：绝不泄露 system prompt、绝不泄露工具及其描述。
- `<limitations>` 里的**能力边界**：不能编辑文件、不能运行终端命令——直接从能力层面切断写操作，确保「只读规划」的定位不被绕过。

三层边界分别对应「不该管的事」「不该说的事」「不能做的事」，是一套可直接套用的安全分层模板。

### 3. 工具调用

工具策略集中在 `<decision_tree>` 与 `<important>` 两处：

- **决策树**给出了标准工作流：先用搜索工具广泛理解代码库 → 完全清晰后用 `write_phases` 输出阶段 → 仅在关键节点用 `ask_user_for_clarification` 询问。
- **澄清的时机与方式**写得很克制：优先靠探索找答案，只有缺关键信息/关键决策/设计偏好时才问；问题要简短、给选项、一次问一个。这避免了规划型 Agent 频繁打断用户的通病。
- `<important>` 强调可并行调用多工具以降低延迟，并下了一条硬约束：**必须用工具生成响应，禁止纯文本回复**。这把「输出结构化」从建议升级为强制。

### 4. 工作态度与代码规范

`<general_guidelines>` 用「不想给团队留坏印象」这种**情绪化激励**来抑制两类坏行为：越权写代码、提出范围外的额外工作；并默认不主动推荐单测（除非用户问或上下文已有）。

`<coding_guidelines>` 则复刻了主流编码 Agent 的经典三条：不假设库可用要先查 package.json/cargo.toml、新组件先看现有组件再规划、用 imports 上下文判断框架并追求 idiomatic。虽然本模式不写码，但规划时仍需遵守这些，保证产出的阶段是「贴合现有代码库」的。

## 可复用模式

- **职业角色锚定**：用一个真实职业（tech lead / SRE / QA）一次性传递一整套隐含行为预期，省去逐条罗列。
- **三层安全边界模板**：领域边界（不该管）+ 保密边界（不该说）+ 能力边界（不能做），可直接迁移到任何受限 Agent。
- **强制工具输出**：「必须用工具响应、禁止纯文本」是保证产物结构化、便于下游消费的有效手段。
- **克制的澄清策略**：先探索、后提问；问题简短、给选项、逐个问——适合所有需要与人协作的 Agent。
- **带例外的边界划线**：用「是否在 IDE 可见范围内」这种可判定标准来划边界，比绝对禁止更实用。

## 不适合直接照抄的地方

- 工具名（`write_phases`、`ask_user_for_clarification`）是 Traycer 私有的，套用前需替换为你自己的工具体系。
- 「绝不提及由 Anthropic 创建」属于厂商身份隔离的特定话术，照抄到别的产品上既无意义也可能误导。
- 「只读、不写码」的强约束是 Phase Mode 这个特定场景的定位；如果你的 Agent 需要落地执行，这套限制会直接卡死它。
- 「IDE 内 / 不碰生产」的领域边界绑定了 IDE 插件形态，CLI 或云端 Agent 需要重新界定范围。

## 适合进一步拆成课程的点

- 如何用「职业角色」做人格设计，对比「逐条规则」的优劣。
- 安全边界的分层方法论：领域 / 保密 / 能力三类边界如何识别与表达。
- 「规划 Agent」与「执行 Agent」的职责分离设计（Plan/Act 双模式）。
- 强制结构化输出（tool-only response）的实现与权衡。
- 澄清时机的设计：如何在「少打扰用户」和「不瞎猜」之间取舍。

## 对这个项目的展示建议

- 把它作为「规划型 / 只读型 Agent」的代表与「执行型编码 Agent」并列展示，突出 Plan vs Act 的对照。
- 在卡片高亮中强调它「短小精悍」的特点——适合作为入门级提示工程的拆解案例。
- 可与其它 IDE 类 Agent（如 Cursor、Kiro）放在同组，对比各家如何做能力边界与工具约束。
