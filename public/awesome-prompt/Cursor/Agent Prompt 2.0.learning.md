# Cursor Agent Prompt 2.0 学习分析

## 一句话定位

这是 Cursor 编辑器内置 AI 编码 Agent（"2.0" 版本，标注由 GPT-4.1 驱动）的完整系统提示词，核心在于用一套 TypeScript 风格的工具签名 + 大量行为约束，把一个通用大模型塑造成"在 IDE 里结对编程、自主推进到任务完全解决"的代码 Agent。

## 为什么值得学习

- **工具描述即文档**：它没有把工具说明单独抽出来，而是直接用 `namespace functions { type tool = (_: {...}) => any; }` 的 TS 声明承载工具语义，注释里写满"何时用 / 何时不用 / 正反例"。这是一种把工具契约和使用指南合二为一的成熟写法。
- **决策边界写得极细**：例如 `codebase_search` vs `grep` vs `read_file` vs `file_search` 的取舍，用四条"何时不要用"+ 多个 `<example><reasoning>` 把模型的工具选择问题前置消解掉，而不是寄希望于模型临场判断。
- **IDE 场景独有的约束**：相比命令行 Agent，它体现了 IDE 特有的能力——光标/打开文件/编辑历史作为隐式上下文、`edit_file` 交给"较弱模型"快速应用、`read_lints` 诊断、notebook 单元格编辑、代码引用的 UI 渲染规则等。
- **代码引用格式规范**是少见的"为前端 UI 服务的输出协议"范例：`startLine:endLine:filepath` 语法配大量 good/bad 对照，本质是在约束模型输出以适配宿主软件的渲染组件。

## 结构拆解

### 1. 身份定义

文末才出现核心身份段：`You are an AI coding assistant, powered by GPT-4.1. You operate in Cursor.`，并强调"结对编程""你是一个 agent——持续推进直到查询被完全解决"。把模型定义为**自主、不轻易交还回合**的 Agent，是 2.0 版的关键基调。

### 2. 工具调用契约（functions / multi_tool_use）

占据全文绝大篇幅。每个工具是一段带详尽注释的 TS 类型声明。亮点：
- 工具按用途分簇并互相对照（搜索类、编辑类、终端、记忆、诊断、任务管理）。
- `multi_tool_use.parallel` 显式鼓励并行调用，即使提示语暗示按顺序也要并行——这是性能/成本优化导向的设计。

### 3. 安全与协作边界（tool_calling / making_code_changes）

- 绝不调用未提供的工具、绝不在对话中提及工具名、不确定时读文件而非猜测。
- `run_terminal_cmd` 要求对交互式命令传非交互标志、长任务后台运行。
- `edit_file` 强调"会被较弱模型应用"，因此必须用 `// ... existing code ...` 标示未改动区，否则可能误删——这是多模型协作流水线的安全约定。

### 4. 探索方法论（maximize_context_understanding）

把"语义搜索"定为主探索手段，要求宽泛起手、拆分子查询、换措辞多搜几遍，倾向自助而非求助用户。

### 5. 任务管理（todo_write / task_management）

用极多正反例界定何时该建待办列表（3+ 步、多任务、重构），并明确"绝不把 linting/测试/搜索代码库放进 todos"，避免待办被操作性琐事污染。

### 6. 输出风格（communication / citing_code）

为宿主 UI 定制的代码块渲染协议，区分"引用现有代码"和"提议新代码"两套格式，禁止混用。

## 可复用模式

- **工具签名 + 内联注释规范**：用目标语言的类型声明承载工具契约，注释里塞"何时用/何时不用/正反例"，比把这些散落在别处更聚合、更难被模型忽略。
- **正反例 + reasoning 三段式**：`<example> … <reasoning>Good/BAD: …</reasoning></example>`，对易错的判断点（工具选择、待办边界、输出格式）逐条钉死。
- **互斥工具的决策表**：对功能相近的工具（语义搜索 vs grep vs 读文件）显式写出取舍规则，消除模型的选择困难。
- **面向宿主 UI 的输出协议**：当 Agent 输出要被特定软件渲染时，把渲染要求写成强约束 + 大量 bad-example。
- **"持续推进直到完成"的 Agent 基调**：明确何时才能交还回合，减少半途而废。

## 不适合直接照抄的地方

- **强绑定 Cursor 工具集**：`edit_file`/`edit_notebook`/`read_lints`/`update_memory`/代码引用渲染等都依赖 Cursor 宿主能力，离开该环境无意义。
- **"较弱模型应用编辑"的假设**：`edit_file` 的 `// ... existing code ...` 约定是 Cursor 双模型架构的产物，单模型方案不需要。
- **代码引用 `startLine:endLine:filepath` 语法**：是 Cursor UI 专用渲染契约，别的产品照搬会渲染异常。
- **知识截止/模型名**（GPT-4.1、2024-06）是该快照的具体信息，不应当作通用模板硬写。
- **篇幅极长**：工具注释非常详尽，token 成本高，小项目不必照此规模。

## 适合进一步拆成课程的点

1. **"工具选择决策树"专题**：以 `codebase_search`/`grep`/`read_file`/`file_search` 四者取舍为案例，讲如何用正反例消除工具误用。
2. **多模型协作的编辑协议**：`edit_file` 的 `// ... existing code ...` 机制，讲"主模型生成 diff、弱模型应用"的工程权衡。
3. **面向 UI 的输出契约设计**：代码引用格式如何服务前端渲染组件。
4. **Agent 自主性与回合控制**：何时持续推进、何时停下询问用户。
5. **待办列表的"反污染"边界**：为什么 linting/测试/搜索不该进 todos。
6. **并行工具调用**：`multi_tool_use.parallel` 的性能/成本动机。

## 对这个项目的展示建议

- 与已有的 "Cursor Agent CLI Prompt" 条目**并列对照展示**：同为 Cursor，但本条是 IDE 内 Agent（含 notebook、lints、代码引用渲染、隐式编辑上下文），CLI 版偏终端环境，可做"同厂商不同形态 Agent 提示词差异"的对比专栏。
- 在卡片高亮"工具签名 + 正反例"这一最具教学价值的写法，作为"如何写工具描述"的范本。
- 提供原文/中文对照阅读，便于学习者逐条体会 MUST/NEVER 等强约束的措辞密度。
- 因来源为社区流出快照，建议在页面标注 `待核验`，不渲染为官方文档。
