# Cursor Agent CLI System Prompt 学习分析

## 一句话定位

这是一份面向**终端（CLI）形态的自主编程 Agent** 的系统提示词，核心命题是：在一个交互式命令行工具里，让模型既能“自己干完活”（autonomy），又能把沟通、代码编辑和工具调用都约束在工程团队可接受的规范内。

## 为什么值得学习

- 它是少见的**把“沟通规范”与“工程行为”平级对待**的系统提示词：communication / summary / markdown 的篇幅几乎和 tool_calling 一样多，说明在 CLI 产品里“输出体验”本身就是产品质量。
- 它示范了如何用**多个并列 XML 标签段落**把一份长指令拆成可维护、可单独引用的模块（`<communication>`、`<tool_calling>`、`<code_style>` 等），而不是写成一大段散文。
- 它对“自主推进”和“何时停下”给出了明确边界——这是 agent 类提示词最容易写崩的地方。
- 它把**代码风格规范（Clean Code 级别）直接内嵌进系统提示**，是“让模型产出可被人类 review 的代码”这一目标的范本。

## 结构拆解

### 1. 身份定义

开篇三句话定调：`AI coding assistant, powered by GPT-5` → `interactive CLI tool` → `pair programming with a USER`。
- **解决的问题**：让模型同时知道“我是什么形态（CLI）”“我和谁协作（USER 结对）”“我的行为模式（agent）”。
- **写得好在哪**：用“pair programming”一词同时传达了协作姿态与责任边界，比“你是助手”信息量大得多。
- **可复用**：身份段落 = 形态 + 协作对象 + 行为模式三要素，缺一不可。

### 2. 安全边界 / 自主性边界

`keep going until the query is completely resolved` + `Only terminate when you are sure the problem is solved` + `State assumptions and continue; don't stop for approval unless you're blocked`。
- **解决的问题**：CLI agent 最怕“做一半就来问用户”或“假装做完”。这里用对称的两句话框定上下限：能自己查就别问，但没真正解决就别收尾。
- **写得好在哪**：把“何时继续”和“何时停下”成对出现，避免模型偏向任一极端。
- **不要照抄**：在高风险/需要人工审批的场景（生产数据库、付费操作），这种“默认继续、不停下确认”的策略要反向收紧。

### 3. 工具调用

`<tool_calling>` + `<maximize_parallel_tool_calls>` 两段。
- **解决的问题**：性能（并行）与正确性（依赖排序）的平衡，以及“编辑后必须跑测试/构建”的工程闭环。
- **写得好在哪**：用“DEFAULT TO PARALLEL”给出强默认值，再用“只有 A 的输出是 B 的输入时才串行”给出唯一例外，规则可执行、无歧义；还附上“快 3-5 倍”的动机解释，提升模型遵循意愿。
- **可复用**：`默认并行 + 单一串行例外 + 量化收益` 是工具编排提示词的黄金模板。
- 第 8/9 条把“substantive edit → run tests/build → 绿色才收尾”写成硬规则，是 agent 自我验证的关键。

### 4. 沟通与输出风格（communication / status_update / summary / markdown）

这是本提示词篇幅最大的板块。
- **解决的问题**：CLI 终端里没有富 UI，文本即界面。所以它细到“只在语义正确处用 markdown”“别整段套代码块”“文件名用反引号”“别用 `#` 一级标题（太夺目）”。
- **写得好在哪**：把“进展更新（过程中）”和“总结（回合末）”分成两个 spec，明确各自时机与格式，避免模型在过程里写长总结、在结尾写流水账。
- **可复用**：`status_update（叙事式、短、说了就立刻做）` + `summary（高层、要点式、不重复计划）` 的双轨输出规范。

### 5. 代码规范（code_style）

直接把命名、类型标注、控制流、注释、格式五类规则写进系统提示，并给出“Bad → Good”对照。
- **解决的问题**：保证产出的代码是“能被人类 review、可立即运行”的，而不是能跑就行。
- **写得好在哪**：用具体反例（`n` → `numSuccessfulRequests`）而非抽象原则，模型更容易对齐。“HIGH-VERBOSITY code 即使被要求简洁沟通”一句解决了“沟通简洁”与“代码详尽”之间的潜在冲突。
- **不要照抄**：这套风格偏向大型协作代码库；写一次性脚本、竞赛代码或受限上下文时会显得啰嗦。

### 6. 上下文理解（context_understanding）

把 Grep 定为主探索工具，强调“宽泛起步 + 并行多模式 + 逐步收窄”。
- **可复用**：`广撒网（并行多 pattern）→ 确认无遗漏 → 收窄精读` 的检索方法论，适用于任何带代码检索工具的 agent。

### 7. 元规则（citing_code / inline_line_numbers / 文件 @ 简写）

处理与编辑器/工具链交互的细节，如行号是元数据、`@file` 要去掉前缀、引用代码用 `start:end:filepath` 格式。
- **价值**：这类“协议层约定”常被忽略，但它们决定了模型输出能否被宿主程序正确解析。

## 可复用模式

- **XML 标签分段**：用 `<communication>`、`<tool_calling>` 等语义标签切分长提示，便于维护与定向引用。
- **成对边界表达**：对“继续 vs 停下”“并行 vs 串行”这类二选一，用一对对称句子同时框定上下限。
- **强默认 + 唯一例外**：先给一个强默认行为，再明确唯一允许的例外条件。
- **Bad → Good 对照**：用具体正反例代替抽象规则来传递风格标准。
- **过程/收尾双轨输出**：把“进行中的状态更新”和“回合末总结”拆成两套独立规范。

## 不适合直接照抄的地方

- **“默认不停下确认、自主推进到底”**：在涉及生产环境、资金、不可逆操作时必须反向加审批闸口。
- **HIGH-VERBOSITY 代码风格**：不适合脚本、原型、token 受限或追求极简的场景。
- **GPT-5 等具体模型名与 CLI 形态绑定**：迁移到其他模型/IDE/Web 形态时，身份段与工具段都需重写。
- **`ApplyPatch` 相关的“五条消息 / 三次连续”等数字约束**：这些是针对特定工具链的经验值，换工具即失效。

## 适合进一步拆成课程的点

- 「agent 自主性边界设计」：怎么让模型既不半途而废、也不假装完成。
- 「工具并行编排」：默认并行策略、依赖判定与量化动机的写法。
- 「CLI/终端 Agent 的输出体验设计」：把文本当 UI 的 markdown 与总结规范。
- 「把代码规范写进系统提示」：用反例对照对齐风格的方法。
- 「检索方法论」：广撒网到收窄精读的 Grep 探索流程。

## 对这个项目的展示建议

- 作为「代码 Agent / CLI 形态」的代表条目展示，与 IDE 内联助手、Web 端 agent 形成形态对比。
- 重点高亮三个模块卡片：**自主性边界**、**默认并行的工具编排**、**内嵌代码规范**，这三处最具迁移价值。
- 在学习页可做一个「沟通规范 vs 工程行为」的篇幅占比可视化，直观说明 CLI 产品中“输出即界面”的设计取向。
- 适合配一组「成对边界表达」「强默认 + 唯一例外」的可复制模板片段，方便学习者直接套用。
