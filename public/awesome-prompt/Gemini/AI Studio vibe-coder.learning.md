# Gemini AI Studio Vibe Coder 学习分析

## 一句话定位

这是 Google AI Studio「vibe coding」（用自然语言生成完整可运行 Web 应用）背后的系统提示词，它把一个通用大模型约束成一个**只产出单文件 XML、严格遵守 React/TypeScript/Tailwind 规范、并内置一整套 `@google/genai` SDK 用法的前端代码生成器**。

## 为什么值得学习

- 这是**生产级代码生成 Agent**的真实样本，不是教学演示。它要解决的核心矛盾是：让一个泛化模型在「无人工 review、直接编译运行」的环境里，产出可立即跑起来的代码。
- 它示范了如何用提示词**对抗模型的"坏习惯"**：模型训练数据里混杂着大量过时 API、错误写法、易踩坑的范式，提示词用「正确/错误对照」逐条把这些倾向掰回来。
- 它把**运行环境的硬约束**（沙箱不能改 URL path、API key 由外部注入、视频需用户自选 key）显式写进规则，是「提示词即运行时契约」的典型案例。
- 它的 SDK 文档部分本质上是一份**面向模型的 API 速查手册（knowledge injection）**，展示了如何在上下文里固化"唯一正确写法"以压制幻觉。

## 结构拆解

整篇提示词可分成两大块：**前半段是工程规范（如何写 React 应用）**，**后半段是 `@google/genai` SDK 用法库（如何调用 Gemini）**。

### 1. 身份定义

开篇用一句话锚定角色：「world-class senior frontend React engineer，精通 Gemini API 与 UI/UX」，并立刻给出首要目标——生成完整可运行、视觉美观的 React 应用。

- 解决的问题：把模型的输出风格从"解释/教学"切换到"直接交付工程产物"。
- 写法亮点：身份 + 技术栈 + 审美要求三合一，结尾还用全大写 `AESTHETICS ARE VERY IMPORTANT` 再次强调，形成首尾呼应的"情绪锚点"。
- 可复用：角色设定要绑定**可验证的产出标准**（完整、可运行、好看），而非空泛的"你是专家"。

### 2. 输出格式契约（最硬的边界）

要求**整个响应必须是单一合法 XML 块**，用 `<changes>/<change>/<file>/<content>` 结构，且内容用 `<![CDATA[...]]>` 包裹；并明令「只返回 XML，不要任何解释」。

- 解决的问题：下游是程序解析而非人阅读，任何寒暄/解释都会破坏自动落盘流程。
- 写法亮点：先给完整 XML 模板，再补"XML rules"，最后规定第一个文件必须是 `metadata.json`——把**机器可解析性**当成第一约束。
- 可复用：当 Agent 输出要被程序消费时，应给出**精确 schema + 反例（不要加解释）+ 转义规则**三件套。

### 3. 工程规范与"正确/错误对照"

第 1–9 节是 React/TypeScript/Tailwind 的硬性规则：目录结构、`import` 规则、枚举不能用 `const enum`、只能用 Tailwind、移动优先、`createRoot`、模板字面量转义、泛型箭头函数尾逗号、Hook 无限循环、组件作用域……

- 解决的问题：约束模型避开它"统计上爱犯"的错误（如用 `ReactDOM.render`、`import type` 取枚举值、在父组件内定义子组件导致失焦）。
- 写法亮点：几乎每条规则都配**Correct / Incorrect 成对代码**，并解释*为什么*错（例如 useEffect+useCallback 的死循环推导链）。这种"对照 + 因果"远比单纯"禁止 X"更能压住模型。
- 可复用：要纠正模型的具体坏习惯，最有效的不是抽象禁令，而是**贴出错误代码 + 正确代码 + 一句话原因**。

### 4. 运行环境约束

明确写出沙箱特性：不能创建 `src/` 前缀路径、不能改 URL path（禁 `BrowserRouter`、可用 `HashRouter`）、不能用独立 CSS 文件、占位图用 `picsum.photos`。

- 解决的问题：模型不知道它运行在受限沙箱里，必须把环境事实灌进上下文。
- 可复用：凡是"环境决定的不可违反事实"，都应在提示词里列成显式清单，而不是寄望模型推断。

### 5. SDK 知识注入（API Key / 模型选择 / 各能力示例）

后半段是一份结构化的 `@google/genai` 用法库：初始化、API key 来源、模型别名映射表、文本/JSON/函数调用/流式/图像/语音/视频/Live/Chat/Grounding/错误处理，每节都带可直接抄的 `ts` 示例和"弃用 API 黑名单"。

- 解决的问题：模型记忆里的 SDK 可能停留在旧版本（`GoogleGenerativeAI`、`getGenerativeModel`），直接生成会跑不起来。提示词用"唯一正确写法 + 明确禁用列表"覆盖模型的过时知识。
- 写法亮点：
  - **别名→全名映射表**（如 "nano banana" → `gemini-2.5-flash-image`），把用户口语翻译成 SDK 真实模型 ID。
  - **按任务类型选默认模型**的决策表，减少模型乱选。
  - 大量 `IMPORTANT` / `CRITICAL` / `Do not` 标注关键约束（如 API key 只能来自 `process.env.API_KEY`、绝不生成索取 key 的 UI）。
- 可复用：当 Agent 要稳定调用某个快速演进的 SDK 时，最可靠的做法是**把"当前唯一正确用法"作为知识块固化进系统提示**，并配一份显式弃用清单。

### 6. 安全与密钥边界

API key 一节是典型的安全约束：key **只能**从 `process.env.API_KEY` 读取，**绝不**生成任何输入/管理 key 的 UI 或代码，**绝不**要求用户在代码里填 key。Veo 视频场景则相反——必须让用户通过 `window.aistudio.openSelectKey()` 自选 key，并要求附计费文档链接。

- 解决的问题：防止把密钥管理逻辑泄漏到生成代码里，统一由平台外部注入。
- 可复用："凭证由宿主环境注入、Agent 不得自行索取或硬编码"是值得照搬的安全默认。

## 可复用模式

- **输出契约模式**：精确 schema + "只返回 X 不要解释" + 转义/CDATA 规则，用于让输出被程序消费。
- **Correct/Incorrect 对照模式**：纠正模型坏习惯的最强武器，附因果说明。
- **知识注入模式**：把快速演进的 SDK 的"唯一正确用法 + 弃用黑名单"写进提示，压制版本幻觉。
- **别名映射表 / 任务→默认值决策表**：把用户口语和模糊需求映射到确定的技术参数。
- **环境契约清单**：把沙箱/运行时的不可违反事实显式列出。
- **首尾情绪锚点**：开头定身份、结尾用全大写重申核心目标（美学+功能）。

## 不适合直接照抄的地方

- **模型 ID 与版本是时效性内容**：`gemini-2.5-flash`、`veo-3.1-*`、`imagen-4.0-*`、各种 `-preview-09-2025` 日期会随时间失效，照抄会指向过期或不存在的模型。
- **强绑定 AI Studio 沙箱**：`window.aistudio.*`、单文件 XML 输出、`src/` 前缀禁令、不能改 URL path 等都是平台特有约束，迁到其他环境需要重写。
- **极端"无解释"输出策略**：只适用于"输出被程序解析"的场景；在需要人机协作、解释推理的 Agent 上照搬会损害体验。
- **提示词体量巨大**：把整份 SDK 文档塞进系统提示会显著占用上下文、抬高成本，普通项目应按需裁剪而非全量复制。
- **个别格式瑕疵**（如响应式一节的 `*.` 列表符号）说明它是工程产物而非精雕教材，不必模仿其排版。

## 适合进一步拆成课程的点

- 「如何为代码生成 Agent 设计机器可解析的输出契约」——以这里的单文件 XML 为案例。
- 「用 Correct/Incorrect 对照纠正模型坏习惯」——可抽取 React Hook 死循环、组件作用域、模板字面量转义做成练习。
- 「SDK 知识注入与弃用黑名单」——讲解如何对抗模型对快速演进库的版本幻觉。
- 「凭证安全：注入 vs 索取」——对比文本场景（外部注入）与 Veo 场景（用户自选 key）两种相反策略。
- 「别名映射 / 默认值决策表」——把模糊需求确定化的提示工程技巧。

## 对这个项目的展示建议

- 在卡片上突出它的双重身份：既是**代码生成 Agent 系统提示**，又是一份**`@google/genai` SDK 速查**，对学习者有"提示工程 + 实用 API 手册"双重价值。
- 把"Correct/Incorrect 对照"作为该条目的招牌看点，可在详情页摘录 1–2 组（如 Hook 死循环）作为预览。
- 用醒目标注提醒读者：模型 ID/版本号具有强时效性、且强绑定 AI Studio 环境，定位为「待核验」更稳妥。
- 难度建议标为「进阶」：读者需具备 React/TypeScript 与 LLM API 基础才能充分吸收。
