# Notion AI System Prompt 学习分析

## 一句话定位

这是一个"嵌入式生产力 Agent"的系统提示：它运行在 Notion 应用内部，通过工具循环操作页面、数据库与搜索，核心挑战不是"会不会写"，而是"如何在用户的真实工作区里克制、安全、贴合产品规范地行动"。

## 为什么值得学习

大多数公开的 Agent 提示词关注代码或浏览器场景，而 Notion AI 展示了一类很少被讲透的形态：**在结构化内容产品（页面 + 数据库）中运行的 Agent**。它的价值集中在三点：

1. 把"产品的数据模型"（Workspace / Page / Database / Data Source / View / 属性类型）以**领域本体（domain ontology）**的形式喂给模型，让模型能精确生成符合产品约束的内容。
2. 用大量"不要过度执行 / 不要主动提议 / 不要假设"的负向约束，把一个强能力模型**收敛到一个克制的协作者**，这是企业级助手最难也最关键的部分。
3. 把输出格式细化到了一种**自定义标记语言（Notion-flavored Markdown）**，是"如何让 LLM 稳定产出结构化富文本"的优秀范本。

## 结构拆解

### 1. 身份定义

开篇四句话就把边界钉死：你是谁（Notion 内部 agent）、在哪交互（chat / sidebar）、如何运转（工具循环，直到无 tool call 结束）、能力上限（"除了工具能做的，你什么都做不了；除了用户消息触发的循环，你不能行动"）。

- **解决的问题**：防止模型幻想出自己并不具备的能力，并明确"行动只发生在循环内"这一执行模型。
- **写法为何有效**：先定义"运行时模型"再定义"人格"，比单纯说"你是一个助手"信息密度高得多。
- **可复用**：`你是 X，运行在 Y 内部 / 你通过 Z 循环行动 / 除工具外你无法行动` 这一三段式模板适用于任何嵌入式 Agent。

### 2. 安全边界与克制（本提示的灵魂）

这一部分占了相当篇幅，且高度独特，分散在多个小节：

- **Refusals（拒绝）**：明确列出"应当拒绝"的任务清单（评论、表单、模板、权限、计费、布局……），并规定话术——宁可说"我没有这个工具"，也不要声称功能"损坏/不支持"。
- **Avoid offering to do things（不要主动提议）**：禁止在任务完成后追加"要不要我再帮你做 X"，尤其禁止提议自己做不到的事。
- **IMPORTANT: Avoid overperforming（避免过度执行）**：核心铁律——"除非明确要求，绝不修改用户内容"，并用大量正例锚定（让你头脑风暴≠让你改页面；查错别字≠改格式；翻译≠加解释）。

- **解决的问题**：强模型天然倾向"多做一点显得有用"，但在用户真实工作区里，多做一点 = 破坏数据 = 信任崩塌。
- **写法为何有效**：用"行为对照例子"而非抽象原则，把模糊的"克制"变成可判定的规则。
- **可复用模板**：`当用户要求 A 时，不要做 B`（A 是请求，B 是看似相关但越界的动作）——这是约束 Agent 最有效的句式。

### 3. 工具调用与搜索策略

- **tool calling spec**：要求"能用工具解决就立即用，不要请求许可"，并规定"第一次工具调用通常应是默认 search"。
- **Search strategy**：极其细致——搜索便宜要多用，但"同一信息连续搜不超过两次"；默认搜索是 web+internal 的超集，是最安全默认；并行搜索要在一次 call 里塞多个问题；附带 8 个"该搜/不该搜"的决策示例。

- **解决的问题**：在"信息可能藏在用户工作区"的前提下，平衡"答得准"与"别瞎搜浪费时间"。
- **写法为何有效**：用**量化阈值**（最多两次连续搜索）+ **决策示例表**，把策略变得可执行、可对齐。
- **可复用**：决策示例表（query → 应采取的检索类型）是训练/对齐检索行为的高性价比手段。

### 4. 领域数据模型（Domain Ontology）

提示词花大量篇幅描述 Page / Database / Data Source / View / 属性类型 / 属性值格式。包括精确到字段级的格式规范，例如 `date` 属性必须 `start` 与 `end` 同提交、`place` 必须带经纬度，并配"失败/正确"的 JSON 对照。

- **解决的问题**：让模型生成的结构化操作能直接通过后端校验，减少无效 tool call。
- **写法为何有效**："反例 + 正例"的 JSON 对照（`Fails:` / `Correct:`）比任何文字描述都更能防止具体的校验错误。
- **可复用**：凡是 Agent 要产出结构化数据（API 参数、配置、schema），都应内置"易错点的正反例对照"。

### 5. 输出格式：Notion-flavored Markdown

定义了一整套自定义标记：颜色属性 `{color="Color"}`、提及 `<mention-page>`、表格 XML、callout、columns、synced block、压缩 URL 等。还区分了**两套风格**：直接聊天回复（简短、中性、可引用）vs. 在页面里写作（丰富排版、无元评论、一次成稿）。

- **解决的问题**：让一个 LLM 稳定产出能被产品渲染引擎解析的富文本。
- **写法为何有效**：把"聊天文风"与"页面文风"显式分离，避免模型把聊天腔带进文档。
- **可复用**：`同一个 Agent 在不同输出通道用不同文风`是一个值得借鉴的分层思路。

### 6. 本地化与表达规范

- **Language**：必须用最贴合用户问题的语言回答；绝不假设用户在用"蹩脚英语"；回复前先输出 `<lang primary="..."/>`。
- **Gender neutral**：仅针对英语输出——不得据名字猜性别，用 they / 改写 / 用名字，并配会议记录行动项的正反例。

- **解决的问题**：跨国产品里语言误判与性别冒犯都是高频投诉点。
- **可复用**：把"敏感表达规范"用正反例固化，而不是泛泛要求"礼貌"。

## 可复用模式

1. **运行时身份三段式**：身份 + 交互界面 + 行动循环模型。
2. **越界对照句式**：`当用户要求 A 时，不要做 B`。
3. **检索决策示例表**：query → 检索类型/不检索，附理由。
4. **结构化输出正反例**：`Fails:` / `Correct:` JSON 对照。
5. **双文风分层**：聊天通道 vs. 内容产出通道，规则各异。
6. **量化护栏**：用具体数字（如"连续搜索≤2次"）替代"适度"这类模糊词。

## 不适合直接照抄的地方

- **领域本体高度绑定 Notion**：Page/Database/Data Source/View 这套模型、属性类型、Notion-flavored Markdown 全部是 Notion 专有，换产品需整体替换。
- **工具名是占位约定**：`update-page`、`query-data-sources`、`view` 等工具在你的系统里并不存在，照抄会指向空工具。
- **示例 context 是占位脏数据**：源文件结尾的 `<context>`（2075 年、Phobos 时区、用户邮箱填成 obsidian.md、工作区名 "Donald Trump's Notion"）显然是脱敏/测试占位，绝不能当真实配置参考。
- **压缩 URL 机制**：依赖 Notion 后端的解压管线，脱离该环境无意义。

## 适合进一步拆成课程的点

- **「如何让 Agent 学会克制」**：以 Refusals + Avoid overperforming + Avoid offering 三节为案例，讲负向约束设计。
- **「领域本体注入」**：以属性类型与属性值格式为例，讲如何把产品数据模型喂给 LLM 并保证可校验。
- **「检索策略工程」**：以 Search strategy 的量化阈值与决策示例表为案例。
- **「结构化富文本生成」**：以 Notion-flavored Markdown 为案例，讲自定义标记语言的设计。
- **「多通道文风分层」**：聊天 vs. 页面写作的风格切换。

## 对这个项目的展示建议

- 归类到「效率工具 / 通用助手」，与代码类、浏览器类 Agent 形成对照，凸显"结构化内容产品 Agent"这一独特形态。
- 在亮点里强调**克制设计**与**领域本体注入**两个差异化卖点。
- 展示时建议把"越界对照例子"与"JSON 正反例"作为可截图的精华片段，最直观。
- 在 caution 中提醒读者：结尾 `<context>` 为占位脏数据、工具名为产品私有，避免学习者误以为是可复用配置。
