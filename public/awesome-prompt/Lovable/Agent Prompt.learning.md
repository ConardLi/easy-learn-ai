# Lovable Agent Prompt 学习分析

## 一句话定位

这是一份面向「非技术用户、实时预览、固定技术栈」的 Web 应用生成 Agent 系统提示词，核心在于用强约束把「设计质量、工具效率、范围克制」三件事同时管住。

## 为什么值得学习

- **产品形态高度具体**：它不是一个泛化助手，而是绑定了固定技术栈（React + Vite + Tailwind + TypeScript + Supabase）和固定界面（左聊天 / 右 iframe 预览）的垂直 Agent。学习它能看到「如何把产品约束写进提示词」。
- **设计质量被当成一等公民**：大多数代码 Agent 只关心「能跑」，而这份提示词花了近一半篇幅强制「设计系统优先、语义化 token、绝不写内联样式」，是把审美标准工程化的范本。
- **效率与克制并存**：既要求并行批量调用工具、绝不重复读上下文，又反复强调「不要过度工程、不要范围蔓延」。两条看似矛盾的目标通过明确规则被同时落地。
- **面向非技术用户的协作设计**：明确「不要让用户手动改文件 / 贴日志，因为大多数用户是非技术人员」，这是把目标用户画像直接转化为行为约束的好例子。

## 结构拆解

### 1. 身份定义

开篇用四段把「我是谁、界面长什么样、能做什么、不能做什么」讲透：

- 身份：实时修改 Web 应用代码的 AI 编辑器。
- 能力边界：能上传图片、读 console 日志、实时预览。
- 技术栈白名单：明确列出支持什么、并显式列出**不支持**的框架（Angular/Vue/Svelte/Next.js/原生 App）。
- 后端边界：不能跑后端语言，只能通过 Supabase 集成。

**价值**：先用「白名单 + 黑名单」双向锁定能力范围，避免模型承诺做不到的事。可复用模板：

```
你是 {产品}，{一句话职责}。
你可以：{能力清单}。
技术栈固定为：{白名单}。因此你无法支持 {黑名单}。
{关键限制}，但可通过 {替代方案} 实现。
```

### 2. 安全边界与范围控制

安全在这里更多体现为「范围安全」而非「攻击安全」，集中在 General Guidelines、Required Workflow 和 Common Pitfalls 三处：

- 默认进入「讨论模式」，只有出现明确动词（implement/code/create/add）才动手写代码。
- 编码前先确认功能是否已存在、范围是否清晰，不清晰必须先问。
- 反复点名禁止：过度工程、范围蔓延、单体文件、一次改太多、使用 `VITE_*` 环境变量。

**价值**：通过「默认不动手 + 显式触发词」把误操作风险降到最低，对非技术用户尤其重要。可复用模式：用「触发词清单」区分讨论与执行，而不是让模型自行揣测意图。

### 3. 工具调用

工具策略非常工程化：

- **基本规则（CARDINAL RULES）**：绝不读已在上下文的文件；尽量批量并行；绝不做可合并的顺序调用。
- **侵入性分级**：search-replace > write-file（仅新文件/重写）> rename-file > delete-file，明确告诉模型「优先用最小侵入工具」。
- **调试优先于改码**：先用 read-console-logs / read-network-requests 看现象，再动代码。

**价值**：把 token 浪费（重复读文件）和动作低效（顺序调用）作为显式反模式来禁止，是控制成本和延迟的实用手段。可复用模板：

```
工具使用基本规则：
1. 绝不重复获取已在上下文中的信息
2. 独立操作必须并行/批量发起
3. 优先选择侵入性最小的工具：{工具优先级排序}
```

### 4. 工作流程（8 步法）

从「检查上下文 → 工具审查 → 默认讨论 → 思考规划 → 提问澄清 → 收集上下文 → 实现 → 验证总结」形成闭环。亮点是把「重述用户真实意图」「界定改什么不改什么」写进规划步骤，强制模型先对齐再动手。

### 5. 设计质量规范

这是全篇最有特色的模块，把「好看」拆成可执行规则：

- 设计系统就是一切，绝不写组件内联样式。
- 仅在 `index.css` / `tailwind.config.ts` 中定义 token，颜色统一用 HSL。
- 用 shadcn 的 variant 机制扩展，而非覆盖。
- 显式禁止 `text-white`/`bg-white` 等裸类。
- 甚至提示了高频 bug：暗/亮模式撞色、outline 变体不透明导致白字不可见、rgb 被 hsl() 包裹导致颜色错乱。

**价值**：把「审美」转化为「约束 + 反例清单」，让模型在不可控的创意空间里有确定性底线。

### 6. 输出风格与首条消息特例

- 全局要求：少于 2 行、少 emoji、改完代码不写长解释。
- 支持 Mermaid 图表说明架构/流程。
- **首条消息特殊处理**：因为是模板初始化，直接进入「写代码」模式，要求先描述设计灵感、列出第一版功能、定义设计系统，并强调「less is more」。

**价值**：区分「常规交互」与「冷启动」两种语境，给出不同行为策略，是上下文敏感型提示词的好示范。

## 可复用模式

1. **能力白名单 + 黑名单双向声明**：避免模型承诺超出产品边界的功能。
2. **讨论模式默认 + 触发词执行**：用关键词显式切换「聊」与「做」。
3. **工具侵入性分级**：给工具排优先级，逼模型选最小代价方案。
4. **审美工程化**：用「设计系统唯一来源 + 裸类黑名单 + 常见 bug 清单」把主观审美变成可校验规则。
5. **反模式清单（Common Pitfalls）**：与正向规则配对，正反夹击降低偏离。

## 不适合直接照抄的地方

- **技术栈与工具名强绑定**：search-replace、read-console-logs、imagegen、`lov-` 标签等都是 Lovable 私有工具/约定，迁移到别的产品需整体替换。
- **`VITE_*` 禁用、Supabase-only** 等限制是平台特性，不通用。
- **「少于 2 行」「绝不写解释」** 的极简输出风格适合实时预览产品，不适合需要教学或评审的场景。
- **首条消息直接写代码** 的策略基于「模板已存在」假设，普通对话照搬会显得武断。

## 适合进一步拆成课程的点

- 「如何把产品约束（技术栈/界面/用户画像）翻译成提示词规则」。
- 「审美工程化：用设计系统 token + 反例清单约束 AI 生成 UI」。
- 「讨论模式 vs 执行模式：用触发词管理 Agent 主动性」。
- 「工具效率规则：批量并行 + 侵入性分级 + 禁止重复读上下文」。
- 「冷启动 vs 常规交互的差异化行为设计」。

## 对这个项目的展示建议

- 归类为「设计生成 / 代码 Agent」交叉案例，重点展示其**设计质量约束**章节，这是与 Claude Code、Cursor 等纯代码 Agent 最大的差异点。
- 可与其他代码 Agent 提示词做横向对比，突出 Lovable「面向非技术用户 + 审美优先」的定位。
- 在学习卡片上标注 highlights：设计系统强约束、工具侵入性分级、讨论/执行双模式；同时用 cautions 提醒私有工具名和平台限制不可照搬。
