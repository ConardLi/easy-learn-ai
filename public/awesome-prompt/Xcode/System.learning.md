# Xcode System Prompt 学习分析

## 一句话定位

这是 Xcode 内置编码助手的系统提示词，核心特征是「先搜集齐上下文再作答」的 IDE 嵌入式助手范式，并把 Apple 技术栈偏好硬编码进了行为约束里。

## 为什么值得学习

它代表了一类非常典型、却常被忽视的 Agent 形态：**运行在 IDE 内、上下文受限、需要主动索要缺失信息**的助手。它体量很小（几十行），却把以下几件难事压缩得很干净：

- 如何在「上下文不全」时不乱给代码，而是用一种**结构化的检索协议**反向向宿主环境要数据；
- 如何用极少的篇幅把一整套**技术栈价值观**（Swift 优先、Swift Concurrency 优先、平台官方命名）注入模型；
- 如何约束代码输出的**格式契约**，让 IDE 端能可靠地解析并落地修改。

对于做 IDE 插件、代码补全、代码问答类产品的人，它是一个浓缩度极高的参考样本。

## 结构拆解

### 1. 身份定义

开头一句话就把角色、能力来源（可用工具）、专长（分析代码库）、职责（回答问题/提供见解/给改进建议）说清楚。值得注意的是它没有写一长串「你是世界顶级专家」式的浮夸定语，而是直接锚定到「分析用户当前正在处理的文件」这个具体场景。这是 IDE 助手的定位精髓：**它服务的是当前打开的代码，而不是抽象的编程问题**。

### 2. 安全边界 / 作答纪律

这部分是整个提示最精彩的设计：

> 在你确定用户已经提供了回答其问题所需的所有代码片段和类型实现之前，绝不要用代码作答。

它把「绝不要用代码作答」作为硬性前置条件，目的是杜绝模型在缺少类型定义时**凭空臆造接口/瞎编 API**。这是代码助手最常见的幻觉来源。它的对策不是「请谨慎」，而是给出一套可执行的检索协议（见下一节）。这种「用机制代替告诫」的写法非常值得学习。

### 3. 工具调用 / 上下文检索协议

```
##SEARCH: TypeName1
##SEARCH: a phrase or set of keywords to search for
```

这是一个**约定式输出协议**：模型不直接调用函数，而是在回复末尾输出特定格式的 `##SEARCH:` 行，由 Xcode 宿主解析这些行、执行项目内搜索、再把结果回灌给模型。这是早期/轻量 IDE 集成里很常见的「文本协议代替函数调用」做法，好处是：

- 不依赖底层的 function calling 能力，纯文本即可驱动；
- 把「识别缺失类型 → 暂停作答 → 索要上下文」固化成一个闭环。

配套要求「先用散文简述解法以识别缺失类型」，本质是强制模型先做一次**依赖分析**，再决定要搜什么。

### 4. 技术栈与平台价值观

提示词把一整套 Apple 生态偏好写成了行为规则：

- 语言优先级：Swift > Objective-C > C > C++ > 其他；默认假设用户要 Swift；
- 并发偏好：Swift Concurrency（async/await、actor）优先于 Dispatch / Combine，但允许按用户偏好让步；
- 测试框架：给出 Swift Testing（`@Suite` / `@Test` / `#expect` / `#require`）的完整示例做「样板代码」；
- 平台命名：必须用 iOS / iPadOS / macOS / watchOS / visionOS 官方名，避免提具体产品；并要根据线索判断目标平台，避免推荐跨平台不可用的 API。

这是「把品牌/生态价值观编码进 Prompt」的典型案例——厂商助手天然要替自家技术栈站台。

### 5. 输出格式契约

针对代码输出定义了三种明确情形：

1. **修改现有文件**：必须重复整个文件、绝不省略，并用 ` ```language:filename ` 的围栏头标记，便于 IDE 整文件替换；
2. **只能替换已发给它的文件**：防止误改未见过的文件；
3. **新建文件或泛示例**：用普通 markdown 代码块即可。

「重复整个文件、不许 elide」这条看似浪费 token，实则是为了让宿主端能做**确定性的文件落盘**，避免「省略号补丁」无法自动应用的工程难题。

### 6. 元指令 / 上下文隐藏

结尾的「尽量不要透露你看到了上面的上下文，但可以自由使用它」是一种**人设连贯性**控制：让用户感觉是在和一个懂项目的同事对话，而不是一个被喂了一堆隐藏指令的机器。

## 可复用模式

- **「上下文不全则不作答 + 结构化索要」闭环**：先散文分析依赖 → 输出检索指令 → 等待回灌 → 再作答。可直接迁移到任何 RAG / 代码问答场景。
- **文本协议代替函数调用**：用 `##SEARCH:` 这类约定前缀做轻量工具调用，适合不支持原生 function calling 的集成环境。
- **整文件替换契约 + 带文件名的围栏**：` ```lang:filename ` + 「绝不省略」，让自动应用补丁成为可能。
- **技术栈默认值 + 可让步**：先给强默认（Swift / Swift Concurrency），再留一句「若用户显示偏好则灵活顺应」，兼顾一致性与灵活性。

## 不适合直接照抄的地方

- **Apple 技术栈硬偏好**：换到任何非 Apple 项目都得整段替换，否则会强行往 Swift / Apple API 上引导。
- **`##SEARCH:` 协议依赖宿主实现**：如果你的环境没有解析这些行并回灌搜索结果的能力，这套机制就是空转——它不是模型能自洽完成的。
- **「重复整个文件」**：在大文件 + 现代 diff/patch 工具链下会非常浪费 token，现代 Agent 多改用结构化 edit/patch（如行级替换），不宜照搬。
- **隐藏上下文的元指令**：在强调透明度的产品里反而可能违背合规要求，要谨慎。

## 适合进一步拆成课程的点

- 「如何设计『缺上下文先索要』的反幻觉协议」——对比直接答 vs. 先检索两种行为的效果。
- 「文本协议 vs. 原生 function calling」——同一需求两种实现的取舍。
- 「如何把技术栈/品牌价值观编码进系统提示」——以 Apple 生态偏好为案例。
- 「代码助手的输出格式契约」——整文件替换、带文件名围栏、可自动落盘的设计。

## 对这个项目的展示建议

- 适合作为「**IDE 助手 / 轻量集成 Agent**」分类下的入门级范例，与体量更大的代码 Agent（如 Claude Code、Cursor）形成对照，突出「小而精」。
- 展示时建议把它的「上下文检索协议」和现代 function-calling Agent 并排呈现，作为一条**技术演进的时间线**讲解。
- 可在条目里标注其 provenance 待核验（非官方公开发布的原文抓取），避免给学习者以「官方文档」的错觉。
