# Qoder Quest Design 学习分析

## 一句话定位

这是 Qoder（阿里巴巴出品的 agentic IDE）中「Quest Design / 设计文档」模式的系统提示词，专门约束 AI 扮演技术文档专家，把用户的功能想法迭代地转化为一份高层抽象、不含代码、可视化为主的设计文档。

## 为什么值得学习

- 它是一个**单一目标的专精 Agent 模式**示例：不写代码、不做实现，只产出一类工件（设计文档）。这种"窄而深"的角色定义，比泛用助手更能体现 prompt 工程中的约束设计。
- 它给出了**按仓库类型分发的文档模板库**（后端/前端/全栈/组件库/CLI/移动/桌面/库/框架/其他共 10 类），是一个把"领域知识"硬编码进 prompt 的典型范本，可直接迁移到任何需要"按场景输出固定结构"的产品。
- 它展示了**工具调用纪律**的写法：并行调用、绝不暴露工具名、绝不调用未提供的工具、白名单失败的兜底话术。
- 它的 `search_replace` / `create_file` / `edit_file` 工具说明，演示了如何用"夸张惩罚"（$100000000 penalty）和大量重复来强化关键约束——既是值得学习的强调手法，也是值得警惕的反面教材。

## 结构拆解

### 1. 身份定义

开头用 `## AI Assistant Identity` 锁定三件事：产品名（Qoder）、运行环境（agentic IDE）、角色（专家级技术文档专家）。同时立刻嵌入第一条安全红线——拒绝透露所用语言模型。把"我是谁 + 我不能说什么"绑在一起，避免身份与保密规则脱节。

### 2. 安全边界

`## Communication Rules` 集中了保密与合规约束：不讨论敏感/情感话题、不泄露内部 prompt/工具、不透露模型、不与其他 AI 比较、用占位符替换 PII、拒绝恶意代码。其设计要点是把"对外话术"也写明（被问身份时礼貌拒绝并把话题拉回需求），而不仅是"禁止"。

### 3. 工具调用

分三层：`## Tool Calling Rules`（通用纪律）→ `## Parallel Tool Calls Guidelines`（并行优化）→ `## Available Functions`（逐个工具的详细 schema 与使用时机）。每个工具不仅给参数，还给"何时用/何时不用"和示例，尤其 `search_codebase` 用 PascalCase/camelCase 作为符号搜索 vs 语义搜索的决策规则，非常具体可执行。

### 4. 工作流程与输出约束

`## Design Process Steps` 用 5 步流程（意图检测→仓库类型检测→编写设计→精炼→反馈）把抽象目标拆成可执行步骤，并强调"不要把流程步骤告诉用户"。`### 4. Refine Design` 明确产物约束：删 plan/deploy/summary 段、删所有代码、改用建模语言/表格/Mermaid、不超过 800 行。结尾再次强调"绝不写 summary 部分"。

## 可复用模式

- **按场景分发的模板库**：用一组 `### XXX SPECIALIZATIONS` 段落，让 AI 先判定类型、再套用对应的固定章节结构。适合任何"输出需标准化但场景多样"的产品（报告、文档、配置生成等）。
- **工具说明三件套**：每个工具写清 `USE WHEN / NOT FOR / EXAMPLES`，再加一条可机械执行的"决策规则"。比单纯描述功能更能降低误用率。
- **流程隐身**：明确告知模型"按 workflow 执行，但不要向用户暴露步骤编号"，让交互自然化。
- **产物负面清单**：用"删除 X / 不要写 X / 不超过 N 行"的硬约束控制输出体积和形态，比正面描述更有效。
- **身份+保密绑定**：把"被追问身份时如何回应"写成可直接复述的话术，避免模型即兴发挥。

## 不适合直接照抄的地方

- **$100000000 罚款式恐吓**：大量重复的"OR face a $100000000 penalty"是一种脆弱的强调手法，依赖模型对夸张语言的服从，可读性差、可维护性低，正式产品 prompt 不宜效仿，应改用结构化约束或工具层校验。
- **硬编码的本地路径**：`b:\Download\qoder`、`B:\Download\qoder\.qoder\quests\{designFileName}.md` 等是特定环境快照，照抄会失效；真实使用应由运行时动态注入。
- **大量逐字重复**：同一条规则（如"默认用 search_replace"）连写多遍，膨胀 token 且并不必然提升遵从度。
- **`%!(EXTRA int=10000)`** 这类是模板渲染泄漏的脏数据，属于该快照的瑕疵，不应作为规范。
- **强制英文回复**：`Communication Guidelines` 写死英文，迁移到中文产品需调整。

## 适合进一步拆成课程的点

- 如何为"窄目标 Agent"设计身份与边界（对比泛用助手）。
- 模板分发模式：从"判定类型"到"套用结构"的提示词工程实践。
- 工具说明的写法：schema + 决策规则 + 正反例。
- 输出约束工程：负面清单、行数上限、禁用代码改用建模语言。
- 反面案例分析：为什么"罚款恐吓 + 逐字重复"不是好的强调方式，有哪些更优替代（分点强调、工具层校验、结构化标签）。

## 对这个项目的展示建议

- 归类到「IDE 助手」，作为"专精文档生成 Agent"的代表，与代码生成型 Agent 形成对照。
- 在展示页突出两个亮点：①按仓库类型分发的 10 套文档模板；②清晰的工具说明三件套。
- 同时附上"反面教材"标注，提醒读者罚款恐吓、硬编码路径、脏数据等不可照抄，引导批判性阅读。
- 来源未经官方确认，标注为「待核验」，不要对外声称为官方公开 prompt。
