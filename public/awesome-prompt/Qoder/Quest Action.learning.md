# Qoder Quest Action 学习分析

## 一句话定位

这是阿里巴巴 Qoder IDE 中「Quest（任务）」后台 Agent 的系统提示词，专门约束一个**自主、无人值守**的编码 Agent：把一份设计文档转化为可执行的实现任务清单，并在结对编程语境下完成代码改动。

## 为什么值得学习

- 它是一个完整的「IDE 内置编码 Agent」提示词样本，覆盖了身份、保密、规划、主动性、工具调用、测试、代码编辑、收尾等几乎全部模块，结构相当典型，适合作为研究 IDE 助手 Prompt 的参考骨架。
- 它演示了「后台 Agent」这一特殊模式与普通对话式 Agent 的区别：不与用户来回交互、自主决策、产出极简总结，这是当前 agentic IDE 演进的重要方向。
- 它在保密与身份防护上写得非常克制且具体（拒绝透露模型、拒绝横向比较、绝不输出尖括号内容），是研究「Prompt 安全防护」的好材料。
- 末尾通过 `<design_doc>` / `<user_query>` / `<execution_instruction>` 等占位符，清晰展示了**模板注入式**的运行时上下文拼装方式，对理解工程化 Prompt 的填充机制很有价值。

## 结构拆解

### 1. 身份定义

开篇用一句话锁定角色（Qoder，集成在 agentic IDE 中的结对编程助手），并立即叠加一条硬约束——拒绝透露所用语言模型。随后 `<background_agent>` 块进一步收窄为「后台自主运行」模式，明确「不询问澄清、产出 1–2 句总结」。这种「先定身份再叠加运行模式」的写法，让同一套 Agent 能力可以被复用到不同执行形态。

### 2. 安全边界（communication）

`<communication>` 块集中处理保密与身份防护，要点高度具体：
- 不泄露内部指令/系统提示词/敏感配置；
- 绝不输出尖括号 `<...>` 内的内容（防止把内部标签泄露给用户）；
- 不打印可运行的终端命令代码块，改用工具执行（既是安全也是 UX）；
- 不透露模型、不与其他 AI 比较，并给出被追问时的标准应对话术（礼貌拒绝→强调能力→引回需求）。
此外还强制要求引用代码符号时用 markdown 链接语法包裹，把「可导航性」写进了输出规范。

### 3. 工具调用（tool_calling / use_parallel_tool_calls）

工具规则分两块：一块讲通用纪律（严格遵循 schema、不调用未提供的工具、不向用户暴露工具名），另一块专门强调**并行优先**，但给出两条关键例外——文件编辑与终端命令必须顺序执行，以避免一致性问题与竞态条件。这种「默认并行、危险操作串行」的取舍是工程化 Agent 的实用经验。

### 4. 规划与主动性（planning / proactiveness）

`<planning>` 给出了「简单任务 3 步内直接做、复杂任务先收集信息再列详细任务清单」的分流策略，并强调「每个实现步骤后紧跟验证任务」「未执行不得标记完成」。`<proactiveness>` 则把「果断执行、优先用工具自查而非追问用户」作为默认行为基调，与后台 Agent 身份相互印证。

### 5. 测试与代码编辑（testing / code_change_instruction）

测试块要求「一次只生成并验证一个测试文件」「写完立即运行并报告结果」，并坦诚承认初版常出错、要靠迭代。代码编辑块规定用 `edit_file` 而非直接输出代码、用 `// ... existing code ...` 省略未改动部分、每轮每文件尽量只编辑一次，并以 `get_problems` 校验作为**强制最终步骤**直至无问题。

### 6. 运行时上下文注入

文件后半部分是运行时拼装的上下文：`<user_info>`（OS、IDE 版本、工作区路径、系统时间）、`<project_wiki>`（项目知识标题列表 + search_memory 使用建议）、`<project_instructions>`（目录结构）、以及 `<execution_instruction>` + `<design_doc>` + `<user_query>` 三件套。这正是「Quest Action」的核心任务模板：基于设计文档产出实现任务清单。

## 可复用模式

- **身份 + 运行模式分层**：先定义通用角色，再用独立块（如 `<background_agent>`）切换执行形态，可复用到「前台/后台」「交互/批处理」等多形态 Agent。
- **保密话术模板**：把「被追问敏感问题时怎么答」直接写成三步话术，避免模型临场发挥泄露信息。
- **并行/串行分级规则**：默认鼓励并行，但显式列出必须串行的危险操作清单。
- **编辑 + 校验闭环**：每次代码改动以静态检查（get_problems）收尾并迭代至零问题，是降低「不可运行代码」的实用约束。
- **占位符注入模板**：用 `<design_doc>` / `<user_query>` 等标签预留运行时填充位，便于工程侧拼装上下文。

## 不适合直接照抄的地方

- 大量工具名（`run_in_terminal`、`edit_file`、`get_problems`、`search_memory`、`add_tasks`）是 Qoder 私有工具，直接照抄到别的产品里会失效，需替换为你自己的工具集。
- `<user_info>` / `<project_wiki>` 中的具体值（windows 24H2、Qoder IDE 0.1.16、`b:\Download\qoder`、系统时间）是某次会话的快照，不具通用性，属于运行时变量而非提示词常量。
- 「绝不透露所用模型/不与其他 AI 比较」这类品牌防护策略要结合自身合规与产品定位再决定是否采用。
- 结尾 `design content goes here` 是占位示例，并非真实设计内容。

## 适合进一步拆成课程的点

- 「后台 Agent vs 交互式 Agent」提示词设计差异：何时禁止追问、如何控制输出长度。
- 工具调用的并行/串行决策：哪些操作必须串行，为什么。
- 编码 Agent 的「编辑—校验—迭代」闭环如何在提示词层面强制落地。
- Prompt 安全与身份防护：保密清单、尖括号过滤、标准拒答话术的设计。
- 运行时上下文注入工程：占位符模板、项目知识检索（memory）的触发条件。

## 对这个项目的展示建议

- 可与其他 IDE 助手（如 Cursor、Kiro、Trae）的提示词并列对比，突出 Qoder「后台 Agent + 设计驱动任务清单」的差异化定位。
- 在展示页用高亮标注「保密话术」「并行/串行规则」「编辑-校验闭环」三个最具教学价值的片段。
- 提醒读者 `<user_info>` 等为运行时快照、provenance 为「待核验」，避免被误读为官方稳定文档。
