# Qoder System Prompt 学习分析

## 一句话定位

这是一份面向 **agentic IDE 编码助手** 的系统提示词：它把"结对编程伙伴"的身份、严密的工具调用纪律、强制验证闭环和记忆管理整合在一起，是研究"IDE 内嵌 Agent 如何安全、可靠地自动改代码"的优质范本。

## 为什么值得学习

- **完整的编码 Agent 形态**：从身份、沟通、规划、工具、测试、Web 开发到记忆管理，覆盖了一个真实生产级 IDE Agent 几乎所有职责面，结构清晰可对照。
- **强制验证闭环**："每次改完代码必须用 get_problems 验证直到无问题"这条贯穿全文，体现了 Agent 设计中"自检收尾"这一核心可靠性模式。
- **并行 vs 串行的明确边界**：只读工具鼓励并行、写操作（文件编辑/终端）强制串行，是一个非常实用、可直接复用的工具编排规则。
- **强约束话术示范**："$100000000 penalty"这类夸张惩罚措辞，是观察"提示词如何强化高优先级约束"的典型样本（也是反面教材，见下文）。

## 结构拆解

### 1. 身份定义

- **解决的问题**：让模型稳定扮演"集成在 IDE 中、与 USER 结对编程"的角色，并锁死身份（拒绝透露底层模型、拒绝与其他 AI 比较）。
- **写得好在哪**：身份与行为绑定——"结对编程伙伴"直接推导出后续的主动性、验证、测试等行为基调；身份保护规则集中在 Communication Guidelines 一处，便于维护。
- **可复用**：`你是 X，集成在 Y 中，与 USER 协作完成 Z。当被问及底层模型时必须拒绝回答。`
- **不要照抄**：拒绝透露模型/拒绝比较的措辞偏防御，开放产品可酌情放宽。

### 2. 安全边界

- **解决的问题**：防泄露（内部指令/系统提示/尖括号内容）、防危险操作（并行写、未验证路径）。
- **写得好在哪**：把"不输出 `<...>` 包裹内容"这种具体可执行的规则写死，比抽象的"保护隐私"更可落地；安全约束在 Security and Safety、Critical Reminders 多处重申，形成冗余强化。
- **可复用**：写操作串行化 + 操作前校验路径 + 改后强制验证，这套"写前校验、写后验证"的安全三件套适用于任何能改文件的 Agent。
- **不要照抄**：天价罚款话术（$100000000 penalty）并非可靠的工程手段，详见下文。

### 3. 工具调用

- **解决的问题**：规范 14 个工具的使用纪律，避免乱调用、并行冲突、对用户暴露工具名。
- **写得好在哪**：
  - "绝不向用户提及工具名，用自然语言描述动作"——区分了 Agent 的内部机制与用户可见表达。
  - 并行/串行规则给了具体例子（读 3 个文件并行 3 个调用），降低歧义。
  - 工具选择有决策规则（含代码标识符→符号搜索；描述功能→语义搜索），把"何时用哪个工具"显式化。
- **可复用**：`只读工具尽量并行；写文件/跑终端必须串行。` + `引用代码符号时用 markdown 链接语法包裹。`
- **不要照抄**：Parallel Tool Calls 段落几乎逐字重复出现两次，属于冗余，不必模仿。

### 4. 规划与任务管理

- **解决的问题**：区分简单任务（≤3 步直接做）与复杂任务（拆解+规划），避免对琐事过度规划。
- **写得好在哪**："每个实现步骤后立即插入验证任务""执行后才标记完成"——把验证嵌进计划本身，而非事后补做。
- **可复用**：`≤3 步直接执行，否则用 add_tasks 拆成可验证小步，每步后紧跟验证任务，实际执行后才标记完成。`

### 5. 测试与验证闭环

- **解决的问题**：保证生成代码可立即运行，避免"写完即交、不管能否编译"。
- **写得好在哪**："一次只写并验证一个测试文件""写完单测必须立即执行并报告结果"——把不可靠的批量生成拆成可控的单文件循环；坦承"初始实现常出错，要迭代到通过"是很务实的自我设定。
- **可复用**：`改代码 → get_problems 验证 → 修复 → 再验证，直到无问题` 这一闭环模板。

### 6. 记忆管理

- **解决的问题**：跨会话沉淀用户偏好、项目信息、规范、经验教训。
- **写得好在哪**：用四类（user_prefer / project_info / project_specification / experience_lessons）+ 两级 scope（workspace / global）把"记什么、记到哪"结构化，便于检索复用。
- **可复用**：这套"分类 + 作用域"的记忆模型可直接迁移到任何长期协作型 Agent。

### 7. 代码改动与输出风格

- **解决的问题**：约束改动方式（默认 search_replace、`// ... existing code ...` 占位）和行数上限（600 行）。
- **写得好在哪**：用占位符表达"只改必要部分"，既省 token 又减少误伤；明确 import/依赖必须补全，保障可运行性。
- **可复用**：`默认用精确替换工具改动，用占位符省略未改代码，补全所有依赖确保可立即运行。`

## 可复用模式

- **写前校验、写后验证三件套**：操作前验路径 → 串行写 → 改后强制 get_problems 闭环。
- **并行/串行二分规则**：只读并行、写操作串行，并配具体示例消歧。
- **任务分级**：≤3 步直接做，复杂任务才拆解规划，验证步骤内嵌。
- **记忆分类模型**：四类内容 × 两级作用域。
- **身份与机制隔离**：不向用户暴露工具名/内部标签，只用自然语言描述动作。

## 不适合直接照抄的地方

- **天价罚款话术（$100000000 penalty）**：靠"威胁"强化约束既不可靠也显廉价，工程上应改为清晰的规则 + 失败后果说明，而非情绪化恐吓。
- **大段重复内容**：Parallel Tool Calls / Use Parallel Tool Calls 两段几乎完全重复，应合并去冗余。
- **硬编码的 600 行限制**：与具体工具实现强耦合，迁移到别的系统需替换为自身的真实约束。
- **过度防御的身份保护**：拒绝比较、拒绝透露模型对开放产品可能过严。

## 适合进一步拆成课程的点

- 「编码 Agent 的验证闭环」：以 get_problems 强制循环为例，讲自检收尾如何提升可靠性。
- 「工具编排：什么时候并行、什么时候串行」：以只读/写操作二分为例。
- 「Agent 的记忆系统设计」：四类 × 两级作用域的分类法。
- 「约束话术的正确写法」：用本提示词的罚款话术作反面案例，对比"规则 + 后果"的工程化写法。
- 「任务规划粒度」：简单/复杂二分与验证内嵌。

## 对这个项目的展示建议

- 把它与其他 IDE 类 Agent（如 Cursor、Kiro）并列，专门对比"验证闭环""工具并行策略""记忆模型"三个维度。
- 在详情页高亮"写前校验/写后验证三件套"和"并行 vs 串行"两张可复用规则卡片。
- 用 `$100000000 penalty` 作为"约束话术反面教材"的展示锚点，引导学习者思考更好的写法。
- 翻译版与学习分析并排展示，方便读者对照英文原文逐段理解设计意图。
