# Antigravity Fast Prompt 学习分析

## 一句话定位

这是 Google Deepmind 的 Antigravity（一款 IDE 形态的 agentic 编程助手）在 **"Fast"（快速）模式** 下的系统提示词，核心命题是：在严格的工作区沙箱内，把"代码 Agent"和"惊艳级 Web 设计师"两种能力压进同一份提示，并用 KI（Knowledge Items）+ 对话日志构建出可跨会话复用的长期记忆机制。

## 为什么值得学习

- 它把**"持久化记忆 / 知识复用"写成了一等公民**：用近三分之一的篇幅定义 KI 系统与对话日志的检索时机，是少见的、显式教模型"先查已有知识、别重复造轮子"的提示词范本。
- 它示范了**用并列 XML 标签段落组织超长提示**（`<identity>`、`<tool_calling>`、`<workflows>`、`<knowledge_discovery>` 等），每段职责单一、可独立维护与引用。
- 它把**"Web 应用设计美学"内嵌进系统提示并写到近乎苛刻**（"看起来基础 = 你失败了"），是"用提示词强约束输出质量下限"的典型。
- 它对**工作区安全边界**给出了明确的白名单式限制（只能读写指定 workspace），适合学习如何在 agent 提示里写沙箱规则。

## 结构拆解

### 1. 身份定义

`<identity>` 三句话定调：`Antigravity, by Google Deepmind` → `pair programming with a USER` → 任务可能是建库/改库/调试/答疑，并说明会附带"打开的文件、光标位置"等元数据但"是否相关由你判断"。
- **解决的问题**：让模型同时清楚产品归属、协作姿态（结对）、任务范围，以及如何对待噪声元数据。
- **写得好在哪**：把"附带的上下文元数据可能无关，由你决定"显式写出来，避免模型被无关的编辑器状态带偏。
- **可复用**：身份段 = 产品名 + 协作对象 + 任务范围 + 元数据使用免责声明。

### 2. 安全边界（工作区沙箱）

`<user_information>` 用 `[URI] -> [CorpusName]` 映射定义活动工作区，并强约束：不得访问工作区外文件；`.gemini` 目录仅限系统指令用途；不要把项目代码写到 tmp/桌面。
- **解决的问题**：把文件读写限制在可信白名单内，防止 agent 越权访问或污染系统目录。
- **写得好在哪**：用"映射表 + 显式禁止清单"而非抽象的"注意安全"，规则可执行、无歧义。
- **可复用**：`工作区白名单 + 例外目录及其唯一用途 + 禁写位置清单` 是沙箱规则的标准三件套。
- **不要照抄**：路径（`c:\Users\Lucas\...`）是某次会话的具体注入值，属运行时变量，迁移时必须替换。

### 3. 工具调用

`<tool_calling>` 本体极简——只强调"始终用绝对路径"；真正的编排规则散落在结尾的通用指令里：**无依赖则在同一块内并行调用，有依赖才串行等待，绝不用占位符猜参数**；数组/对象参数必须用 JSON 结构化。工具清单本身也是一份高质量样本（`browser_subagent`、`codebase_search`、`generate_image`、`run_command` 等）。
- **解决的问题**：性能（并行）与正确性（依赖排序、参数精确）的平衡。
- **写得好在哪**：每个 tool 的 `waitForPreviousTools` 字段把"并行/串行"决策下沉到参数层，配合"精确使用用户给定值、不编造可选参数"的硬规则，约束清晰。
- **可复用**：`默认并行 + 单一串行例外 + 禁止占位符` 是工具编排黄金模板；`browser_subagent` 的"必须定义清晰返回条件 + 返回后读 DOM/截屏验证"是子代理协作的好范式。

### 4. 知识发现与持久化记忆（KI 系统）

`<knowledge_discovery>` + `<persistent_context>` 是本提示词的最大亮点与最大篇幅。核心规则：**任何研究/分析/写文档之前，必须先检查会话开始时给出的 KI 摘要**，命中则先读 artifact 再补充独立调查；并用 ❌/✅ 成对示例演示"别急着从零研究 vs 先查 KI"。同时区分两类记忆源——原始**对话日志**（要细节时读）与提炼后的 **KI**（做研究时先查），并明确各自的"何时用/何时不用"。
- **解决的问题**：跨会话场景下模型容易重复劳动、忽视团队已沉淀的模式与已知陷阱。
- **写得好在哪**：① 用"🚨 MANDATORY FIRST STEP"把检查顺序写成强制流程；② 大量"deceptively simple"反例（"加个字段""加日志"其实都有既定模式）校准模型直觉；③ 特别强调 **"KI 是起点而非 ground truth，必须验证、预期有缺口、质疑一切"**，防止模型盲信缓存知识。
- **可复用**：`先查已有知识摘要 → 命中读 artifact → 在其上补充独立验证` 的检索-复用闭环，适用于任何带长期记忆/RAG 的 agent；"区分原始日志 vs 提炼知识，并各配 When to Use / When NOT to Use"是记忆分层的好模板。
- **不要照抄**：整套机制依赖宿主提供 KI 摘要、`.gemini/knowledge` 目录与 KNOWLEDGE SUBAGENT 等基础设施，没有这套后端时照搬规则会落空。

### 5. 工作流（Workflows / Turbo 注释）

`<workflows>` 允许用 `.agent/workflows/*.md`（YAML frontmatter + markdown）定义可复用步骤，并用 `// turbo` / `// turbo-all` 注释授权自动运行 `run_command` 步骤（设 `SafeToAutoRun: true`）。
- **解决的问题**：把高频多步操作固化为可触发（含 `/slash-command`）的脚本，并对"哪些步骤可免审批自动跑"做细粒度授权。
- **写得好在哪**：用带注释的代码示例直接演示 turbo 的逐步生效范围，比纯文字描述更不易误解。
- **可复用**：`声明式 workflow 文件 + 行内注释控制自动执行权限` 是把"人类批准闸口"做成可配置项的好设计。

### 6. Web 应用开发与设计美学

`<web_application_development>` 规定技术栈（HTML/JS + Vanilla CSS，慎用 Tailwind，复杂应用才上 Next.js/Vite 且 `npx -y ... ./` 非交互初始化）、五步实现工作流（设计系统先行 → 组件 → 页面 → 打磨）、SEO 与可测试性（唯一 ID），并以全大写口号收尾："看起来基础 = 你失败了"。
- **解决的问题**：保证生成的前端"开箱即惊艳"，而非能跑就行的 MVP。
- **写得好在哪**：把抽象的"好看"翻译成可执行细节——具体字体名（Inter/Roboto/Outfit）、glassmorphism、HSL 配色、微动画、禁用占位图（改用 `generate_image`）。
- **不要照抄**：这套"必须惊艳、禁止简单"的强约束适合 demo/营销页，但对内部工具、数据密集型后台或追求极简的产品会过度设计；技术栈选型也带强主观偏好。

### 7. 沟通风格

`<communication_style>` 要求 github 风格 markdown、反引号包裹文件/函数名、可主动但"别让用户意外"（问"怎么做"时先答而非直接改文件）、不确定就请求澄清。
- **可复用**：`主动性 + 不制造意外 + 不确定就澄清` 三条是 agent 协作礼仪的最小集。

## 可复用模式

- **XML 标签分段**：用语义标签切分超长提示，每段单一职责、可定向引用与维护。
- **沙箱三件套**：工作区白名单 + 例外目录及唯一用途 + 禁写位置清单。
- **记忆复用闭环**：先查知识摘要 → 命中读 artifact → 在其上独立验证补充；并区分"原始日志 vs 提炼知识"。
- **强制流程 + 成对反例**：用"MANDATORY FIRST STEP"锁定顺序，再用 ❌/✅ 对照校准模型直觉。
- **声明式 workflow + 行内授权注释**：把"自动执行/需审批"做成 `// turbo` 级别的细粒度开关。
- **把质量下限写进提示**：用"做不到 = 失败"的强措辞，配具体可执行细节（字体、配色、动画）兜底输出质量。

## 不适合直接照抄的地方

- **运行时注入值**：操作系统（windows）、`c:\Users\Lucas\...` 路径、`.gemini` 目录、`<budget:token_budget>` 都是单次会话的具体值，属变量而非通用规则。
- **KI / 持久化记忆机制**：强依赖宿主的 KI 摘要供给、知识目录与 KNOWLEDGE SUBAGENT，缺这套后端时规则会空转。
- **"必须惊艳、禁止简单"的设计强约束**：对内部工具、后台系统、极简产品属于过度设计。
- **技术栈强偏好**（Vanilla CSS、慎用 Tailwind）：是产品方的主观选型，迁移到其他团队需重写。
- **Antigravity / Google Deepmind 身份与 IDE 形态绑定**：换产品/模型/形态时身份段与工具段都要重写。

## 适合进一步拆成课程的点

- 「Agent 的长期记忆设计」：KI 与对话日志如何分层、检索时机如何用 When to Use / When NOT to Use 表达。
- 「让模型先查后做」：用强制流程 + 成对反例避免重复造轮子，以及"缓存知识不可盲信"的防呆写法。
- 「工作区沙箱规则」：白名单 + 例外用途 + 禁写清单的写法。
- 「把设计质量写进提示」：如何把抽象审美翻译成可执行的字体/配色/动画细节。
- 「声明式 workflow 与自动执行授权」：`// turbo` 注释如何做人工审批闸口。

## 对这个项目的展示建议

- 作为「IDE 助手 / 代码 Agent」类目下、**主打"长期记忆 + 自主编程"** 的代表条目展示，与纯 CLI、纯 Web 端 agent 形成形态对比。
- 重点高亮三张模块卡片：**KI 持久化记忆闭环**、**工作区沙箱安全边界**、**把设计质量写进提示**，这三处最具迁移价值且差异化明显。
- 可与同目录的 `planning-mode.txt` 做「Fast 模式 vs Planning 模式」对照展示，说明同一产品如何用不同提示切换行为模式。
- 适合配一组可复制模板片段：「沙箱三件套」「记忆复用闭环」「强制流程 + 成对反例」，方便学习者直接套用。
