# Bolt System Prompt 学习分析

## 一句话定位

Bolt 是 StackBlitz 出品、运行在浏览器内 WebContainer 沙箱里的「全栈代码生成 Agent」系统提示词，核心价值在于：把一个受限运行环境的物理边界，转译成一套可执行、可校验的写代码纪律。

## 为什么值得学习

大多数代码 Agent 提示词都在讲"怎么写好代码"，而 Bolt 提示词最大的特色是先讲"你身处什么环境、什么做不到"。它是一个研究"环境约束驱动型提示词"的极佳样本：

- 它把运行时（WebContainer 无原生二进制、无 pip、无 g++、无 Git）写成硬约束，从源头杜绝模型生成跑不起来的方案。
- 它发明了一套领域专属的输出协议（`<boltArtifact>` / `<boltAction>`），把"对话回答"变成"可被宿主程序解析执行的指令流"。
- 它在数据库章节里展示了非常严苛的"数据安全 > 一切"的优先级表达，是安全规则写法的范本。

学习它，能掌握"如何把产品的工程现实约束，准确无歧义地灌进模型行为"。

## 结构拆解

### 1. 身份定义

开头一句话定调：`expert AI assistant and exceptional senior software developer`。极简，但立刻锚定了"资深工程师"的语气与质量预期。

- 解决的问题：让模型默认进入高质量代码生成模式，不需要用户反复强调。
- 写得好在哪：不堆形容词到失真，一句话给出能力画像即收尾，把篇幅留给约束与协议。

### 2. 安全边界（system_constraints）

这是全篇的灵魂段落。它穷举了 WebContainer 的限制并配以"为什么"：

- Python 仅限标准库、无 pip、无第三方库；无 C/C++ 编译器；无 Git。
- 用 `IMPORTANT:` 反复强调偏好：优先 Vite、优先 Node.js 脚本、优先非原生二进制依赖（libsql/sqlite）。
- 明确"无法执行 diff/patch，必须写完整代码"——这条直接决定了下游输出协议。
- 末尾给出一份精确的可用 shell 命令白名单。

设计价值：与其让模型"试错"，不如提前告诉它环境的真实边界。把"做不到的事"写清楚，比单纯说"要做对"更能避免幻觉式方案。

### 3. 工具调用 / 输出协议（artifact_info）

Bolt 没有传统意义的 function-calling 工具，而是定义了一套 XML 式的"行动 DSL"：

- `<boltArtifact id title>` 包裹一次完整交付，`id` 用 kebab-case 且跨迭代复用。
- `<boltAction type="shell|file|start">` 三种动作类型，各有严格规则：shell 不能跑 dev、start 专跑 dev、file 必须写完整内容。
- 强约束 action 顺序（先建文件再执行）、先装依赖、依赖尽量写进 `package.json`。

这套协议的精妙在于：它让自然语言回复变成宿主可机器解析的可执行流水线，同时用规则消除了"半截代码""重复启动 dev 服务器"等常见故障。

### 4. 数据库与数据安全（database_instructions）

这是安全规则写法的教科书段落：

- 用 `CRITICAL / FORBIDDEN` 等级分明的措辞声明"数据完整性最高优先级"。
- 明确禁止 `DROP`/`DELETE`/事务控制语句，并贴心地排除 `DO $$ BEGIN...END $$` 误伤。
- 强制"迁移文件 + 立即查询"双 action 且 SQL 必须完全一致。
- 强制每张表启用 RLS，强制邮箱密码认证、禁用 magic link/SSO/自建认证。
- 迁移文件必须带 markdown 摘要块，面向技术与非技术读者。

值得注意：提示词里嵌入了 JS 模板字符串条件逻辑（`${supabase ? ... : ''}`），说明它是被程序动态拼接的，而非纯静态文本。

### 5. 思维链与输出风格

- `chain_of_thought_instructions`：要求先给 2-4 行的简短计划再动手，并附正反例。
- 多条 `ULTRA IMPORTANT`：不准啰嗦、不准解释、先回 artifact、只用合法 markdown、绝不说出 "artifact" 这个词。

这组规则塑造了 Bolt"少废话、直接交付可运行项目"的产品人格。

## 可复用模式

- 环境约束前置模板：
  > 你运行在 {环境}。它不能 {能力1/能力2}。因此：优先 {推荐方案}；绝不要 {禁止方案}。
- 行动 DSL 模板：用 `<Action type="...">` 把不同副作用（建文件/跑命令/起服务）分型，并为每型写死纪律。
- 安全分级措辞：`CRITICAL > IMPORTANT > FORBIDDEN`，配合"例外说明"防止规则误伤合法用法。
- 完整输出强制条款：明令禁止 `// rest of code...` 类占位符，适合任何"无法做增量编辑"的执行环境。
- 双写一致性模板：迁移文件与执行查询内容必须逐字相同，用于保证声明态与运行态一致。

## 不适合直接照抄的地方

- 全篇深度绑定 WebContainer 与 Supabase，离开这两个具体技术栈，大量约束会失效甚至误导。
- 含未求值的 JS 模板字符串（`${cwd}`、`${supabase...}`、`allowedHTMLElements.map(...)`），是运行期注入占位符，照抄会出现裸变量。
- "绝不要解释、不准啰嗦"的极端简洁风格服务于其产品交互，通用助手照搬会显得冷淡、不利于教学场景。
- `<boltArtifact>` 协议只有 Bolt 宿主能解析，换到别处需要替换为自己的执行协议。

## 适合进一步拆成课程的点

- 课程一：如何把"运行环境的物理限制"写成提示词硬约束（以 WebContainer 为例）。
- 课程二：设计可被程序解析的"行动 DSL"——动作分型、顺序约束、幂等与状态管理。
- 课程三：数据安全规则的措辞分级与"例外豁免"写法（以 Supabase RLS/迁移为例）。
- 课程四：模板字符串注入式提示词——静态规则与动态上下文如何拼接。

## 对这个项目的展示建议

- 归类为「代码 Agent」，与 Claude Code、Cursor 等放在一起做横向对比，突出 Bolt 的"沙箱约束"差异点。
- 在详情页高亮三个看点：WebContainer 约束清单、`<boltAction>` 行动协议、Supabase 数据安全分级。
- 可做一个"约束驱动 vs 能力驱动"的提示词对照专题，把 Bolt 作为约束驱动的典型案例。
- 提醒读者：源文件含模板占位符，属于运行期动态拼接的提示词，阅读时需自行脑补注入值。
