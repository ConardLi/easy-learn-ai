# Roo Code System Prompt 学习分析

## 一句话定位

Roo Code 是一款多模式（Code / Architect / Ask / Debug / Boomerang）的 VS Code 编程 Agent，其 System Prompt 用纯文本约定 + XML 工具协议，把"单步工具调用 + 用户确认"的安全循环写进了每一次交互。

## 为什么值得学习

- 它是**不依赖原生 function calling**、靠 XML 标签自定义工具协议的典范，适合在任何裸 LLM 上复刻 Agent 能力。
- 它把**模式（mode）**作为一等概念：同一套工具，通过不同角色定义与文件编辑限制衍生出多种专业 Agent，是"少量提示词撬动多种人格"的好样本。
- 安全机制非常克制：每次只用一个工具、必须等用户确认结果、`attempt_completion` 前要自检——这是一套可直接迁移的"防失控"模板。
- 工具描述里大量内嵌**正例示范**（多个 `<read_file>` / `<apply_diff>` 例子），是教科书级的"用示例消除歧义"。

## 结构拆解

### 1. 身份定义

开篇只有两句："技术高超的软件工程师" + "以最少的代码改动完成任务，专注可维护性"。

- 解决的问题：给模型一个稳定、克制的工程师人格，避免大改、过度发挥。
- 写得好在哪：身份与价值观（minimal changes、maintainability）一句话绑定，比堆砌形容词有效。
- 可复用：`你是 {角色}，你以 {核心约束} 完成任务，专注于 {质量目标}。`
- 别照抄：它的"最少改动"价值观是为编辑现有代码库设计的，做"从零生成"类产品时反而要鼓励完整产出。

### 2. 安全边界（RULES + 工具确认循环）

这是整个 Prompt 的重心，分散在 `Tool Use Guidelines`、`RULES`、`attempt_completion` 描述里：

- **单步执行**："每条消息只能使用一个工具" + "必须等待用户确认再继续"——把不可逆操作拆成可审查的最小单元。
- **目录沙箱**："不能 cd 出工作区"、"所有路径相对基目录"、"不要用 ~ / $HOME"——锁死文件系统作用域。
- **完整写入铁律**：`write_to_file` 必须给出完整文件、禁止 `// rest unchanged` 占位符——避免模型偷懒导致文件损坏。
- **自检机制**：`attempt_completion` 前必须在 `<thinking>` 中自问"先前工具是否已确认成功"。
- 可复用模板：`一次一个工具 → 等结果 → 再决策`；`写整文件时禁止省略占位符`。
- 别照抄：强制人工确认会牺牲自动化吞吐，全自动 Agent 场景需要换成自动回灌结果。

### 3. 工具调用（XML 协议）

- 用 `<tool_name><param>value</param></tool_name>` 的 XML 约定取代原生 tools API。
- 每个工具都遵循统一文档结构：**描述 + 参数表 + Usage 模板 + 多个 Examples**。
- `apply_diff` 用 `SEARCH/REPLACE` + 行号块；`read_file` 支持行范围流式读取大文件；这些都是为"省 token + 精准"设计的。
- 可复用：工具文档"四段式"（说明/参数/用法/示例）是降低模型调用错误率的最佳实践。
- 别照抄：XML 协议是在没有 function calling 时的折中；若底层模型支持原生工具调用，应优先用原生方式，解析更稳。

### 4. 多模式与编排（MODES + switch_mode + new_task）

- 五种内置模式共享工具，但角色定义和**可编辑文件正则**不同（如 architect 只能编辑 `\.md$`）。
- `switch_mode` 让 Agent 自请求换角色，`new_task` 衍生子任务，`Boomerang` 模式专门做任务编排委派。
- 可复用：用"角色定义 + 文件/工具权限矩阵"做轻量级多智能体，无需为每个 Agent 重写整套工具。

### 5. 上下文与可扩展性（environment_details + .roo/rules + MCP）

- `environment_details`（文件树、运行中的终端）自动注入，但提示词明确要求"不要当成用户请求"。
- 自定义指令可从 `.roo/rules/`、`.roo/rules-code/` 分层加载，并可用 `.roo/system-prompt-code` 整体覆盖。
- MCP（Model Context Protocol）作为外部工具/资源扩展点，分本地 Stdio 与远程 SSE 两类。
- 可复用：把"自动注入的环境上下文"和"用户真实意图"在提示词里显式区分，能减少模型误把噪声当指令。

### 6. 输出风格

- 禁止以 "Great/Certainly/Okay/Sure" 开头；要求直接、技术化、不寒暄。
- `attempt_completion` 结尾**禁止反问**，必须是终态结论。
- 可复用模板：`严禁以 {寒暄词} 开头；结论必须终态化，不得以问题结尾。`

## 可复用模式

- **工具文档四段式**：描述 → 参数（标注必填/可选）→ Usage 模板 → 多 Examples。
- **单步安全循环**：一次一个工具 + 等确认 + 完成前自检。
- **角色 × 权限矩阵**：同一工具集 + 不同 role 定义 + 文件编辑白名单 = 多专业 Agent。
- **环境与意图分离**：自动注入上下文要显式声明"非用户直接请求"。
- **完整写入约束**：整文件写入禁止占位符省略。

## 不适合直接照抄的地方

- **XML 工具协议**：仅在没有原生 function calling 时才有价值；有原生 tools 的平台应改用原生 schema。
- **强制人工逐步确认**：会显著拖慢全自动流水线，需按产品形态调整为自动回灌。
- **硬编码的 Windows 路径与 `c:/Projects/JustGains-Admin`**：这是某次会话快照的环境值，复用时必须参数化。
- **"最少改动"价值观**：偏向维护现有代码库，生成型场景应反向设定。

## 适合进一步拆成课程的点

1. 不靠原生 function calling，如何用 XML/Markdown 自定义一套可解析的工具协议。
2. 工具文档怎么写才能把模型的调用错误率压到最低（参数标注 + 反例/正例）。
3. 用"角色 + 权限矩阵"实现轻量多智能体编排（Boomerang 模式案例）。
4. Agent 安全设计：单步执行、目录沙箱、完整写入、完成前自检。
5. 分层指令加载机制（全局 rules → 模式 rules → 整体覆盖）的工程实现。

## 对这个项目的展示建议

- 在卡片高亮里突出"XML 工具协议 + 多模式 + 单步安全循环"三个差异点，与原生 function calling 的 Agent 形成对照。
- 配套展示中英对照，重点圈出 `Tool Use Guidelines` 与 `RULES` 两段，它们最能体现安全设计取舍。
- 可与同库中的 Claude Code / Cline 等同类编程 Agent 并列，做"工具协议风格"横向对比的教学切片。
- 标注 `sourceStatus: 待核验`，并提示其中路径/系统信息为某次会话快照，仅供学习结构、不可当配置直接使用。
