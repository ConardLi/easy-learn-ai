# Manus Modules 学习分析

## 一句话定位

这是 Manus 通用 AI agent 的「模块化系统提示词」全文，把一个能在 Linux 沙箱里自主操作的 agent 拆成十几个用 XML 标签界定的规则模块（事件流、agent loop、planner、knowledge、datasource、各类操作规则），是「如何用提示词把一个复杂自主体系结构化封装」的范本。

## 为什么值得学习

它和单纯的「角色设定 + 几条规则」型提示词不同，呈现了一个**生产级自主 agent 的完整心智模型**：

- 它先定义 agent 感知世界的方式（事件流的 7 类事件），再定义 agent 行动的节奏（agent loop 的 6 步），最后才逐域下达具体操作约束。这种「感知—循环—领域规则」的分层是大型 agent 提示词的通用骨架。
- 它显式引入了外部系统模块（Planner / Knowledge / Datasource），并明确告诉模型「这些事件是系统生成的，你不需要回复」，展示了提示词如何与多模块运行时协同，而不只是孤立地约束一个 LLM。
- 它对每一类工具（shell、browser、file、deploy、message）都给出了相互独立、可单独阅读的规则块，便于维护、扩展和 A/B 调整。
- 大量约束背后都有可推断的产品动机（成本、可访问性、可追溯性、安全），是学习「为什么这么写」的好素材。

## 结构拆解

### 1. 身份与能力定义（intro / system_capability）
开篇用一句话定身份（Manus，由 Manus 团队创建），紧接着用 `<intro>` 列出 6 类擅长任务、用 `<system_capability>` 列出可调用的环境能力。**能力清单与身份分离**：前者面向用户表达「我能做什么」，后者面向模型说明「我手上有哪些杠杆」。

### 2. 感知层：事件流（event_stream）
定义了 agent 唯一的输入通道——一个按时间排序、可能被截断的事件流，并枚举 Message / Action / Observation / Plan / Knowledge / Datasource / 其他共 7 类事件。这是整份提示词的认知基座：后面所有模块都建立在「我通过读事件流理解世界」之上。

### 3. 行动层：agent loop（agent_loop）
用 6 步固定循环（分析事件 → 选工具 → 等执行 → 迭代 → 提交 → 待机）约束 agent 的节奏，并强制「每次迭代只选一个工具调用」。这把一个开放式自主体收敛成可预测、可调试的单步循环。

### 4. 外部模块协议（planner / knowledge / datasource）
三个模块块本质是「与运行时其他系统对话的协议说明」：planner 提供带编号伪代码的计划，knowledge 提供有适用范围的最佳实践，datasource 提供权威数据 API。datasource 还附了一段 `weather.py` 代码示例，明确「数据 API 只能用 Python 调用、不能当工具、结果要存文件」。

### 5. 领域操作规则（todo / message / file / info / browser / shell / coding / deploy / writing / error_handling）
这是提示词的主体，每个 XML 块管一类操作：
- **todo_rules**：用 todo.md 做可见的进度清单，且规定「计划优先于 todo.md，但 todo.md 更细」。
- **message_rules**：区分 notify（非阻塞）与 ask（阻塞），强制首条回复只确认收到、交付物以附件提供。
- **info_rules**：明确信息优先级（数据 API > 网络搜索 > 内部知识），强调搜索摘要不算来源、必须访问原页。
- **browser_rules**：定义元素索引格式 `index[:]<tag>text</tag>`、视口限制、Markdown 提取的不完整性。
- **shell / coding / deploy_rules**：偏运维细节（-y/-f 自动确认、代码必须落盘再执行、服务必须监听 0.0.0.0）。
- **writing_rules**：强约束长文写作风格（散文优先、最少数千字、分草稿再追加、最终不得删减）。

### 6. 边界与元规则（error_handling / sandbox_environment / tool_use_rules）
收尾给出错误恢复阶梯（核实参数 → 按报错修 → 换方法 → 求助用户）、沙箱环境的硬事实（Ubuntu 22.04、Python/Node 版本），以及关于「如何使用工具」的元规则（必须用函数调用、不许向用户提工具名、不许捏造工具）。

## 可复用模式

- **「感知 → 循环 → 领域规则」三段式骨架**：任何需要多工具、多轮自主执行的 agent 都可以套用：先定义输入（事件流/上下文格式），再定义单步循环，最后逐域下规则。
- **每次只调用一个工具**：用一句硬约束把并发与混乱挡在门外，极大提升可调试性，是新手 agent 设计最值得抄的一条。
- **XML 标签分块**：每个领域一个语义标签（`<shell_rules>`、`<browser_rules>`…），让规则可定位、可独立增删、可被其他提示词 `@引用`。
- **通信分级（notify / ask）**：把「打扰用户」抽象成显式的两档动作，并默认偏向非阻塞，是平衡自主性与用户体验的好模板。
- **信息来源优先级显式化**：用一条 `权威数据 > 搜索 > 内部知识` 的排序治理幻觉，并补「摘要不算来源、必须访问原页」。
- **进度可视化（todo.md）+ 计划优先级声明**：让长任务有外部可见、可纠偏的状态载体，同时说清 todo.md 与上游计划谁说了算。
- **错误恢复阶梯**：核实 → 修复 → 换路 → 求助，给模型一个明确的失败处理顺序而不是无限重试。

## 不适合直接照抄的地方

- **强绑定 Manus 自己的运行时**：planner / knowledge / datasource 三个模块、`/opt/.manus/.sandbox-runtime`、expose port 工具、`data_api.ApiClient` 都是 Manus 专有基础设施，没有同款运行时照抄会指向不存在的能力。
- **沙箱硬事实会过期**：Ubuntu 22.04、Python 3.10.12、Node 20.18.0 是写死的环境快照，换环境必须同步更新，否则误导模型。
- **写作规则过于激进**：「最少数千字」「最终长度必须超过所有草稿之和」「几乎全程禁列表」是 Manus 产品调性的选择，对大多数场景（尤其要简洁的助手）反而是反模式。
- **「不向用户提工具名」是产品策略**：在需要透明披露工具行为的场景下不适用。
- **数据 API「成本由系统承担、无需登录」**：这是 Manus 的商业封装假设，自建系统通常需要自行处理鉴权与配额。

## 适合进一步拆成课程的点

- **课程一：agent 的感知层设计**——什么是事件流，为什么把 Message/Action/Observation 显式分类，截断与省略如何处理。
- **课程二：把自主体收敛成单步循环**——agent loop 6 步拆解，「每次只调一个工具」带来的可调试性收益。
- **课程三：提示词如何与外部模块协同**——planner/knowledge/datasource 协议设计，以及「系统事件无需回复」这类协同约定。
- **课程四：领域规则的 XML 分块工程化**——如何按工具/操作切分规则块，做到可维护、可引用、可灰度。
- **课程五：用提示词治理幻觉与失败**——信息来源优先级、搜索摘要不算来源、错误恢复阶梯。
- **课程六：人机通信分级**——notify vs ask 的设计哲学与适用边界。

## 对这个项目的展示建议

- 本仓库已有 Manus 的「Prompt」与「Agent loop」条目，建议把本「Modules」条目定位为**「Manus 规则模块全景」**：前两者讲身份与循环主干，本条目补齐「逐域操作规则」的细节层，三者形成由粗到细的递进，可在卡片简介中互相提示「配合阅读」。
- 在 highlights 中重点突出三段式骨架、单步循环、XML 分块、notify/ask 分级这几条最具迁移性的设计，引导读者从「看一份具体提示词」上升到「学一套 agent 提示词的组织方法」。
- 在 cautions 中显著标注「强绑定 Manus 运行时」与「沙箱版本会过期」，避免读者把专有基础设施当通用模板照搬。
- 可在课程化展示时配一张「感知层 / 行动层 / 模块协议 / 领域规则 / 元规则」的分层结构图，作为理解整份提示词的导航。
