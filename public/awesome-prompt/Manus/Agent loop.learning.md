# Manus Agent Loop 学习分析

## 一句话定位

这是 Manus 系统提示词的开篇片段，用「身份 + 能力清单 + 工作语言规则 + 系统能力 + agent loop 六步循环」搭建起一个通用计算机 agent 的运行骨架，核心看点是把 agent 的执行节奏显式写成一个可循环的状态机。

## 为什么值得学习

- 它示范了如何用极短的篇幅（30 余行）就把一个通用 agent 的「我是谁、我能做什么、我怎么一步步做」讲清楚，密度很高。
- 「agent loop 六步」是这段提示词最有价值的部分：它把抽象的「自主执行」拆解成分析事件 → 选工具 → 等执行 → 迭代 → 提交 → 待命的闭环，这是构建任何 ReAct/事件驱动 agent 都能借鉴的心智模型。
- 「每次迭代只选择一次工具调用」是一条克制而关键的约束，避免 agent 一次性盲目堆叠动作、丢失对中间观察结果的反馈，体现了对单步可观测性的重视。
- 它明确区分了「能力（capabilities）」与「执行流程（loop）」两个层面，结构分层清晰，适合作为提示词写作的范本。

## 结构拆解

### 1. 身份定义

开头一句话点明身份（Manus，由 Manus 团队创建的 AI agent），紧接一张 6 条的「擅长任务」清单，把 agent 的能力域从信息收集、数据分析、写作、建站到通用编程逐层铺开。这种「身份 + 擅长领域」的写法，既给模型设定角色，又隐性划定了任务边界。

### 2. 工作语言规则

中间一段专门规定语言行为：默认英语、用户指定则切换、思考与回应都用工作语言、工具调用中的自然语言参数也要用工作语言，并特别要求「避免纯列表与项目符号格式」。这是对多语言、跨地区产品很实用的一致性约束，也透露出对输出风格的偏好。

### 3. 系统能力（System capabilities）

用一张清单枚举 agent 实际拥有的执行手段：消息工具、带网络的 Linux 沙箱、shell/编辑器/浏览器、多语言编程、自主装依赖、部署公开访问、必要时建议用户接管浏览器做敏感操作。这一段把「抽象能力」落到「具体工具与环境」，让模型清楚自己手里有哪些牌。其中「建议用户临时接管浏览器以进行敏感操作」是一个值得注意的安全协作设计。

### 4. Agent Loop 六步循环

核心部分：把任务执行定义为一个迭代循环——分析事件、选择工具、等待执行、迭代、提交结果、进入待命。它强调以事件流（event stream）为输入，以单次工具调用为最小动作单元，以耐心重复为执行纪律，最终以消息附件交付成果并回到空闲态等待新任务。这是一个完整的状态机描述。

## 可复用模式

- **能力清单 + 系统能力双层结构**：上层写「我擅长解决什么问题」（面向用户价值），下层写「我有哪些具体工具与环境」（面向执行），二者分开能让角色设定和工具映射都更清晰。
- **显式 agent loop 模板**：可直接套用「分析事件 → 选择工具 → 等待执行 → 迭代 → 提交结果 → 进入待命」作为任意自主 agent 的执行骨架。
- **单步动作纪律**：「每次迭代只选择一次工具调用」可作为约束写入任何工具型 agent，强制其在每一步后吸收观察结果再决策。
- **工作语言一致性规则**：默认语言 + 跟随用户 + 思考/回应/工具参数全部统一语言，是多语言产品提示词的现成模板。

## 不适合直接照抄的地方

- 身份与团队名（Manus）是品牌专属，复用时必须替换为自己的产品。
- 「Linux 沙箱、可部署公开访问、自主安装依赖」描述的是 Manus 特定的运行环境，若你的 agent 没有这些底层能力，照抄会造成能力幻觉。
- 这只是完整系统提示词的开篇片段，缺少工具的具体调用规范、规划/记忆模块、错误恢复与安全细则，不能当成可独立运行的完整提示词使用。
- 「避免纯列表与项目符号格式」是 Manus 的风格偏好，未必符合你的产品输出需求。

## 适合进一步拆成课程的点

- 「如何为通用 agent 设计一个可循环的执行状态机」——以这六步为案例，对比 ReAct、Plan-and-Execute 等范式。
- 「能力描述的两个层次：用户价值 vs 执行工具」——讲提示词中身份与工具映射的分层设计。
- 「单步工具调用纪律」——讨论为什么限制每轮一个动作有利于可观测性与可控性。
- 「多语言 agent 的工作语言一致性」——一个小而实用的提示词工程专题。
- 「敏感操作的人机协作」——以「建议用户接管浏览器」为引子讲 agent 的安全交接设计。
