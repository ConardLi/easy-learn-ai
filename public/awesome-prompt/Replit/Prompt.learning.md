# Replit System Prompt 学习分析

## 一句话定位

这是一份面向「在线 IDE 内嵌编程助手」的 system prompt：它不直接执行操作，而是通过一套结构化的 XML 提案标签，把"想做什么"转译成 IDE 可自动应用的"建议动作"。

## 为什么值得学习

- 它示范了**"提议而非执行"**的协作范式：Agent 不拥有最终写入权，只产出可被 IDE 解析、用户确认的结构化提案。这对任何"人在回路（human-in-the-loop）"的产品都极具参考价值。
- 它用**意图分类**把模糊的用户请求路由到正确的处理通道（改文件 / 跑命令 / 纯问答 / 引导到其他工具），是一份很好的"请求分诊"模板。
- 它把**产品边界写进了 prompt**：哪些事情 Assistant 不该做（如处理 secrets、部署），而要"让位"给专门的工作区工具。这是大型产品中"职责划分"的典型写法。
- 篇幅短、密度高，适合作为入门者理解"prompt 即产品契约"的范例。

## 结构拆解

### 1. 身份定义（identity）

- **解决的问题**：用一句话锚定角色与场域——"Replit Assistant"+"在 Replit 在线 IDE 中协助编码"。
- **写法亮点**：极简，不堆砌人格修饰，只交代名称与职责，把空间留给后面的能力与协议。
- **可复用**：`你是 {产品名}，职责是在 {环境} 中完成 {任务域}`。

### 2. 安全边界（behavioral_rules + 危险命令标记）

- **解决的问题**：防止 Agent 过度发挥、擅自重构或执行破坏性操作。
- **写法亮点**：
  - `behavioral_rules` 只有两条，却覆盖了"聚焦请求 + 遵循既有模式 + 不做创造性扩展"这一编程助手最常翻车的点。
  - 危险命令通过 `is_dangerous` 布尔属性显式标注（`rm -rf *`、`killall python` 等举例），把"危险判断"前置为结构化字段，而非靠下游猜测。
- **可复用**：用一个布尔/枚举字段把"风险等级"编码进动作提案，交由 UI 决定是否二次确认。

### 3. 工具调用 / 提案协议（response_protocol）

这是全篇的核心，把"动作"标准化为多种 XML 标签：

| 标签 | 用途 | 关键约束 |
|------|------|----------|
| `<proposed_file_replace_substring>` | 局部替换 | `<old_str>` 必须在文件中唯一，否则失败 |
| `<proposed_file_replace>` | 整文件替换/创建 | — |
| `<proposed_file_insert>` | 按行号插入/建新文件 | 行号缺失则追加到末尾 |
| `<proposed_shell_command>` | 执行命令 | 标注 `is_dangerous`；不得用于起服务 |
| `<proposed_package_install>` | 装包 | 指定 language + package_list |
| `<proposed_workflow_configuration>` | 配置长时运行命令 | 不得直接改 `.replit` |
| `<proposed_deployment_configuration>` | 部署 build/run | 不得直接改 `.replit` |
| `<proposed_actions>` | 末尾汇总 | summary ≤ 58 字符 |

- **解决的问题**：让 LLM 的自然语言输出可被 IDE 确定性解析并自动落地，同时保留每个动作的语义（编辑 vs 替换 vs 部署）。
- **写法亮点**：
  - 为不同动作设计**不同标签**而非一个万能标签，使下游解析与 UI 呈现都更精确。
  - 反复强调**不要直接编辑 `.replit`**，而走专用提案——把"配置文件的唯一写入通道"收敛，避免配置漂移。
  - 用**唯一性约束**（`<old_str>` 必须唯一）规避替换歧义，并直接告诉模型"否则会失败"，把约束的后果讲清楚。

### 4. 意图分诊与工具让位（capabilities + workspace tool nudge）

- **解决的问题**：把用户请求分为四类——改文件、跑命令、纯问答、引导到其他工具——并为每类配大量正例。
- **写法亮点**：
  - 每种能力都附 4-5 个**具体查询示例**，用 few-shot 锚定边界，比抽象描述更可执行。
  - 明确"涉及 secrets/环境变量 → 引导到 Secrets 工具""部署/发布 → 引导到 Deployments 工具"，并提醒"用户可能用 publish 等别名"——体现对真实用户语言的预判。
- **可复用**：`当请求涉及 {X}，绝不要自行处理，改为引导到 {专门工具}`。

### 5. 输出风格约束

- 末尾 `<proposed_actions>` 的 `summary` 限定 **≤ 58 字符**，是典型的"为 UI 留位"约束——prompt 直接服务于界面展示尺寸。

## 可复用模式

1. **提案标签协议**：把每类副作用动作封装成带属性的 XML/JSON 标签，让模型输出"可被程序消费"，而不是只给人读。
2. **请求分诊 + few-shot 边界**：用具体示例划定"该做 / 该让位"的界限。
3. **危险动作显式编码**：`is_dangerous: true|false`，把风险判断交给模型标注、由 UI 兜底确认。
4. **单一写入通道**：对关键配置（`.replit`）规定只能通过专用动作修改，禁止旁路编辑。
5. **UI 感知约束**：摘要字数、Run 按钮、下拉菜单等界面元素直接写入 prompt。

## 不适合直接照抄的地方

- 所有 `<proposed_*>` 标签都是 **Replit 专有协议**，依赖其 IDE 后端解析；脱离该环境无任何意义，必须替换为你自己产品的动作 schema。
- `.replit`、Nix、Workflow、Deployments、Secrets 等都是 **Replit 平台概念**，不可平移。
- 58 字符摘要、Run 按钮等约束绑定具体 UI，需按自己界面重新定义。
- 行为规则只有两条，对更复杂的安全/合规场景（注入防护、越权、隐私）覆盖不足，照抄会留下盲区。

## 适合进一步拆成课程的点

- **课程一：结构化动作协议设计**——如何为 Agent 设计可被程序解析的动作标签，及"一动作一标签 vs 万能标签"的取舍。
- **课程二：human-in-the-loop 的提案模式**——"提议-确认-应用"三段式，与直接执行型 Agent 的对比。
- **课程三：请求分诊与工具让位**——用 few-shot 划分职责边界，避免 Agent"越权"。
- **课程四：把 UI 约束写进 prompt**——字数限制、按钮语义等界面契约如何反向约束模型输出。

## 对这个项目的展示建议

- 作为「IDE 助手 / 代码 Agent」类目的**轻量入门样本**：篇幅短、结构清晰，适合首屏展示，配上"提案而非执行"的一句话核心卖点。
- 在翻译页旁高亮 `response_protocol` 的标签表格，作为"结构化动作协议"的直观教学锚点。
- 与执行型代码 Agent（如直接读写文件的 Claude Code 类）并列对比，凸显"提议型 vs 执行型"两种权限模型的差异。
- 标注 `sourceStatus: 待核验`，提醒读者这是社区流传快照，非官方发布版本。
