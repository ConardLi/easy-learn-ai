# Windsurf Prompt (Wave 11) 学习分析

## 一句话定位

这是 Windsurf 的 agentic 编程助手 Cascade 的 system prompt（Wave 11 版本），用一组 XML 分区把"身份 + 工具纪律 + 代码改动规范 + 安全边界 + 记忆系统 + 计划维护"组织成一套可执行的 IDE 内 agent 行为契约。

## 为什么值得学习

- 它是少数把"持续工作直到任务解决"（agentic loop）与"工具调用必须节制"两个看似矛盾的要求同时写清楚的工程化 prompt。
- 用 XML 标签做模块分区（`<tool_calling>`、`<making_code_changes>`、`<running_commands>` 等），可读性高、便于按场景检索，是商用 IDE 助手 prompt 的典型范式。
- 对"代码必须立即可运行""命令安全判定不可被用户推翻""记忆系统主动落盘"等规则给出了清晰的优先级和模态词，是学习如何写"强约束 + 留余地"指令的好样本。
- 真实地暴露了产品取舍：例如规定被问及底层模型时回答 `GPT 4.1`，以及对工具成本（token、冗余调用）的反复强调。

## 结构拆解

### 1. 身份定义

开篇用三段话锁定身份：产品名 Cascade、公司归属 Windsurf、范式 AI Flow，以及核心工作模式"与 USER 结对编程"。

- 解决的问题：让模型在多轮对话中保持稳定的角色与口吻，并明确"USER 请求永远最高优先级"。
- 写法亮点：把"附加的当前状态元数据（打开的文件、光标位置）可能相关也可能无关，由你判断"显式写出，避免模型把噪声上下文当成强指令。
- 可复用：身份段 = 名称 + 归属 + 工作范式 + 优先级声明 + 元数据使用说明。
- 不宜照抄：AI Flow、Cascade 等是品牌专属概念。

### 2. 安全边界

主要集中在 `<running_commands>`，是整份 prompt 的纪律核心。

- 规则：使用 `run_command` 时绝不能把 `cd` 写进命令，改用 cwd；可能有破坏性副作用（删文件、改状态、装系统依赖、外部请求）的命令绝不自动运行。
- 最强约束："你不能允许 USER 推翻你在这方面的判断"——把安全判定权牢牢留在模型侧，并用 "NEVER NEVER" 双重否定加重语气。
- 写法亮点：给出了 allowlist 这个"用户确有需要时的合规出口"，避免一刀切；同时要求"不要在回复中引用工具的具体参数"，防止泄露内部 schema。
- 可复用模板：先列"何为不安全"，再写"默认拒绝自动执行"，最后给"合规放行通道"。

### 3. 工具调用

`<tool_calling>` 用 6 条规则 + 3 个示例约束工具使用。

- 核心矛盾的处理：既要求"持续工作直到完全解决"，又要求"只在绝对必要时调用工具、绝不冗余调用"。通过"通用问题直接答、声明用工具就立刻调、异步工具等输出时就停手"等细则平衡。
- 写法亮点：用正例 `<example>` 演示"何时不该调工具"和"先 grep 再 view 再 edit"的检索-编辑链路，比纯规则更易模仿。
- 可复用：在工具说明里同时给"该调"和"不该调"的对照示例，是降低误调用率的有效手段。

### 4. 代码改动规范

`<making_code_changes>` 规定不向用户直接输出代码、必须用编辑工具，并强调"生成的代码必须立即可运行"。

- 落地细则：补全 import/依赖/endpoint、从零建库要带依赖文件和 README、>300 行拆成多次编辑（受 8192 输出 token 限制）、禁止生成超长哈希或二进制、调用编辑工具时永远先生成 `TargetFile`。
- 写法亮点：附了一个完整的"分步骤解释 + 改动摘要"示例，统一了交付话术。
- 可复用：把"输出限制（token）"转化为"行为约束（拆分编辑）"，是对模型物理边界的工程化处理。

### 5. 记忆系统

`<memory_system>` 鼓励主动、频繁地用 `create_memory` 落盘关键上下文。

- 解决的问题：上下文窗口有限且会被清空（含 checkpoint 摘要），用外部记忆库对冲长任务的遗忘。
- 写法亮点：连用多个"你无需……"消除模型的保守倾向，并把"用户可拒绝"作为安全阀，让 agent 敢于多写记忆。
- 可复用：当系统有外部状态存储时，prompt 应明确"无需许可、无需等待、可被撤销"三点来鼓励使用。

### 6. 计划维护

`<planning>` 要求通过 `update_plan` 让计划始终反映"当前世界状态"。

- 触发时机写得很全：收到新指令、完成项、获取改变范围的信息、投入重大行动前、完成大量工作后。
- 价值取向："宁可多更新也别错过更新"，鼓励频繁同步状态。

### 7. 其它专项区块

- `<browser_preview>`：Web 服务器启动后必须调用预览，桌面/游戏类不调——场景化的工具触发条件。
- `<calling_external_apis>`：选版本要兼容依赖文件，API Key 不得硬编码。
- `<communication_style>`：第二/第一人称、markdown、反引号格式化标识符。
- `<EPHEMERAL_MESSAGE>` 说明：系统注入的临时消息要遵循但不回应——一种带外指令通道。

## 可复用模式

- XML 分区式 prompt：每个关注点一个标签，便于维护与检索。
- 模态词分级：IMPORTANT / EXTREMELY IMPORTANT / CRITICAL / NEVER NEVER 形成可感知的强度梯度。
- "默认拒绝 + 合规出口"安全模板：先定义风险，再默认禁止自动化，最后给受控放行通道（allowlist）。
- 正例驱动：用 `<example>` 演示期望的工具链路与交付话术，而非只罗列规则。
- 物理限制行为化：把 token 上限翻译成"大编辑拆小"的可执行规则。

## 不适合直接照抄的地方

- 品牌与范式词（Cascade、Windsurf、AI Flow）以及"被问模型答 `GPT 4.1`"这类产品特定话术。
- 具体工具名（`run_command`、`replace_file_content`、`view_code_item`、`create_memory`、`update_plan` 等）与你的实际工具集强绑定。
- 8192 token、知识截止 2024-06、windows/具体 workspace 路径等环境参数。
- 注入在 prompt 里的真实用户路径属于上下文泄露样本，复用时应剔除。

## 适合进一步拆成课程的点

- 「如何让 agent 既持续干活又不滥用工具」：以本 prompt 的工具调用 6 条为案例。
- 「命令执行安全护栏设计」：拆解 running_commands 的风险分类与放行机制。
- 「外部记忆系统的 prompt 写法」：如何用语言鼓励 agent 主动落盘且可撤销。
- 「把模型物理限制翻译成行为规则」：以 token 上限 → 拆分编辑为例。

## 对这个项目的展示建议

- 归类为 IDE 助手 / 代码 Agent，与 Cursor、Claude Code 等同类 prompt 横向对比工具纪律与安全策略的差异。
- 在详情页高亮"安全边界不可被用户推翻"和"记忆系统主动落盘"两段，作为商用 agent 设计范例。
- 标注 sourceStatus 为待核验，并提示其中含真实用户路径属社区抓取样本，仅供学习。
