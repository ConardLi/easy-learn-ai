# Codex CLI System Prompt 学习分析

## 一句话定位

这是一份面向"已部署、可运行代码"的终端编码 Agent 系统提示词，核心在于把"在沙箱里自主改代码并交付"这件事，拆成清晰的能力声明、编码准则和收尾自检流程。

## 为什么值得学习

它和聊天型助手最大的区别在于：默认假设"我有一个可写、可运行、基于 git 的真实工作区"。提示词围绕这个前提，把代理行为约束成工程团队里一个负责任的远程队友——既给足自主权（持续推进直到解决），又用编码准则和收尾清单兜住质量底线。对于想做"能落地交付"的编码 Agent，这是一份很典型的范本。

## 结构拆解

### 1. 身份定义

开篇明确"你在 Codex CLI 内部运行"，并列出可执行能力（接收文件、发起函数调用、应用补丁、运行命令、沙箱回滚、遥测回放）。特别加了一段澄清：不要把自己和旧版 Codex 语言模型混淆。这种"主动消歧"是大模型提示词里很实用的一招，避免模型基于训练记忆产生身份错觉。

### 2. 能力边界与自主性

"你是一个 Agent —— 请持续推进直到任务彻底解决"是 agentic loop 的灵魂句：明确告诉模型不要中途把球踢回给用户。同时配一句"没把握就用工具读文件，绝不要猜"，把自主性和可靠性绑在一起，防止 Agent 自信地编造。

### 3. 安全/许可声明

用一组"允许做什么"的白名单（可在专有仓库工作、可分析漏洞、可展示代码与工具调用细节）来主动解除模型的过度谨慎。这是反向安全设计——很多模型会因为版权或安全顾虑而拒绝合理请求，提示词通过显式授权把这些顾虑提前消解。

### 4. 编码准则（CODING GUIDELINES）

这是全篇最有复用价值的部分：根因修复优于表面补丁、避免不必要复杂性、忽略无关 bug、保持代码风格一致、改动最小化、不擅自加版权头、用 git log/blame 补上下文。每一条都是资深工程师代码评审里的高频要求，被显式写成 Agent 行为规范。

### 5. 收尾自检流程

完成编码后必须执行的检查清单：git status 健全性检查、用 git diff 移除自己添加的内联注释、检查误加的版权头、运行 pre-commit、按任务复杂度分级输出总结。这套"交付前自检"是把人类工程师的肌肉记忆固化成提示词的好例子。

### 6. 输出与交互礼仪

区分"需要改文件"与"只是回答问题"两种模式：前者交付式、要求总结改动；后者用友好队友语气。还有一条很实用的细节——已经用 apply_patch 改过的文件，不要再让用户"自己保存"或贴大文件内容，避免画蛇添足的交互。

## 可复用模式

- **白名单式授权**：与其写"注意安全"，不如显式列出"允许做 X、Y、Z"来解除模型过度拒绝。
- **持续推进 + 不许猜**：把自主性（keep going）和可靠性（don't guess, use tools）成对写入，是 agentic 提示词的稳定句式。
- **编码准则清单化**：把代码评审标准（根因修复、最小改动、风格一致、无关 bug 不管）写成可执行 bullet，比泛泛的"写好代码"有效得多。
- **交付前自检清单**：git status / git diff / 移除注释 / pre-commit，可直接迁移到任何 git 工作区类 Agent。
- **分级输出**：小任务简短要点、大任务高层描述 + 评审者关心的细节，让总结粒度匹配任务规模。

## 不适合直接照抄的地方

- `apply_patch` 的具体补丁格式、`codex --help`、遥测/沙箱回滚等都强绑定 Codex CLI 平台形态，迁移到其他工具需替换。
- "网络访问已被禁用""不需要 git commit，会自动完成"是该产品运行时的特定假设，换平台不成立。
- "尽量移除所有内联注释"这条偏激进，反映的是该产品的代码风格取向，未必适合所有团队（有些团队鼓励必要注释）。
- pre-commit 的处理逻辑依赖目标仓库存在该配置，属于条件性规则。

## 适合进一步拆成课程的点

- 如何用"白名单授权"消除大模型的过度拒绝。
- agentic loop 中"持续推进"与"不许编造"如何协同设计。
- 把代码评审标准转写成 Agent 可执行的编码准则。
- 交付前自检清单的设计方法论。

## 对这个项目的展示建议

可以和 Claude Code 系统提示词并排展示，作为"两家顶级厂商的终端编码 Agent"对比案例：Claude Code 偏重安全边界与极简响应风格，Codex CLI 偏重编码准则与交付自检。两者对照能很好地讲清"代码 Agent 提示词到底在约束什么"。
