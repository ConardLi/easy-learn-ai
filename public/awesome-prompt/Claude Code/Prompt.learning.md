# Claude Code System Prompt 学习分析

## 一句话定位

这是一份面向软件工程任务的 CLI 代码 Agent 系统提示词。它不是把模型包装成“懂编程的聊天助手”，而是把模型约束成一个能读代码、调用工具、修改文件、运行验证、同时保持极简输出的命令行协作工具。

## 为什么值得学习

Claude Code 这份提示词的价值在于，它同时处理了四个产品级问题：

1. Agent 应该怎样定义自己的身份和边界。
2. 一个能修改代码、运行命令的 Agent 应该如何处理安全风险。
3. 在 CLI 这种高密度界面里，回答风格应该怎样压缩。
4. 如何用任务管理和工具策略，把开放式编程请求变成可追踪的执行流程。

它适合学习“执行型 Agent”的系统提示词设计，而不是学习普通聊天机器人的提示词写法。

## 结构拆解

### 1. 身份定义

原文开头把助手定义为“interactive CLI tool”，并且明确任务域是“software engineering tasks”。

这比“你是一个资深软件工程师”更具体。它同时交代了三个信息：

- 产品形态：CLI 工具
- 工作对象：软件工程任务
- 协作方式：使用下方指令和可用工具协助用户

可学习写法：

```text
你是一个运行在 [产品界面/环境] 中的 [工具类型]，用于帮助用户完成 [任务范围]。你应结合 [可用工具/上下文] 协助用户。
```

设计启发：身份定义不要只写人格或职业，要写清楚“运行在哪里、服务什么任务、通过什么方式工作”。

### 2. 安全边界

这份提示词没有泛泛写“不要做危险的事”，而是把安全边界拆成允许和拒绝：

- 拒绝创建、修改或改进可能被恶意使用的代码
- 允许安全分析、检测规则、漏洞解释、防御性工具、安全文档

这是一种非常实用的安全写法。它降低了模型过度拒绝的概率，也减少了模型在灰区任务上的自由发挥。

可学习写法：

```text
拒绝：[明确禁止的行为]
允许：[边界内仍可帮助的任务]
当不确定时：[采取的保守策略]
```

设计启发：好的安全规则要给模型“可执行的边界”，而不是只给价值观口号。

### 3. 产品信息查询策略

当用户询问 Claude Code 自身能力时，提示词要求先用 WebFetch 到官方文档查询。

这条规则解决的是“产品知识会过期”的问题。系统提示词不要求模型凭记忆回答，而是把动态信息交给外部文档。

可学习写法：

```text
当用户询问 [产品自身能力/价格/限制/配置] 时，先查询 [权威来源]，再回答。
```

设计启发：凡是会变化的信息，都应该绑定权威来源，而不是写死在系统提示词里。

### 4. 响应风格

这份提示词对输出风格的约束非常强：简短、直接、少于 4 行、不要前言后语、一个词能回答就用一个词。

这不是单纯的“少说话”，而是服务于 CLI 产品界面。命令行用户通常更关心下一步操作和结果，不希望看到长篇解释。

值得注意的是，它还通过多个示例强化了“短回答”的形态。这比只写一句“保持简洁”更有效。

可学习写法：

```text
由于你的输出会显示在 [界面类型]，你的回答应 [风格要求]。以下示例展示了理想回答长度和形态。
```

设计启发：风格规则最好绑定产品界面。不同界面的“好回答”并不一样。

### 5. 主动性边界

提示词允许 Agent 主动，但限定在“用户要求做事时”。如果用户只是问方案，应先回答方案，而不是直接实施。

这条规则解决的是 Agent 产品里很常见的问题：模型过度行动。尤其是代码工具，如果用户只是想讨论，Agent 却开始改文件，会造成不信任。

可学习写法：

```text
你可以主动执行后续步骤，但只在用户明确要求你完成任务时。若用户在询问方案或权衡，应先回答问题，不要直接修改环境。
```

设计启发：主动性不是越高越好。好的 Agent 要知道什么时候行动，什么时候只是帮助用户思考。

### 6. 代码库约定

“Following conventions” 是这份 prompt 的核心工程质量模块。它要求：

- 修改前先理解已有代码约定
- 不假设库可用
- 新建组件前先看已有组件
- 编辑代码前先看 imports 和上下文
- 避免泄露密钥

这组规则的本质是让 Agent 不要把代码库当作空白项目，而要把它当作已有系统来协作。

可学习写法：

```text
在修改前，先观察相邻代码、依赖声明、命名方式和已有模式。优先沿用当前项目的约定，而不是引入新的抽象或库。
```

设计启发：代码 Agent 的关键能力不是“能写代码”，而是“能写进当前代码库”。

### 7. 任务管理

提示词强制使用 TodoWrite 来规划和追踪任务，并要求完成后立即标记 completed。

这不是为了形式化列清单，而是为了让长任务具备状态可见性。对复杂工程任务来说，Agent 很容易忘记步骤、重复处理或中途偏航。Todo 工具把任务从自然语言变成可追踪对象。

可学习写法：

```text
当任务包含多个步骤、可能跨文件、需要验证或存在不确定性时，使用任务列表拆解，并在每一步完成后立即更新状态。
```

设计启发：如果 Agent 要做长任务，就应该有外部化的进度状态。

### 8. 执行任务流程

“Doing tasks” 把软件工程任务拆成稳定流程：

1. 必要时规划任务
2. 搜索理解代码库
3. 实现方案
4. 运行测试验证
5. 完成后运行 lint/typecheck
6. 未经明确要求不提交 commit

这是一套很完整的代码 Agent 默认工作流。它既给了 Agent 自主性，也设置了停止边界。

可学习写法：

```text
处理 [任务类型] 时，按 [理解上下文] -> [实施] -> [验证] -> [汇报] 的流程执行。除非用户明确要求，不要进行 [高影响动作]。
```

设计启发：工作流规则最好把“该做什么”和“不要越界做什么”放在一起。

### 9. 工具使用策略

工具策略强调两个点：

- 文件搜索时优先使用更节省上下文的 Task 工具
- 多个独立信息请求应并行调用工具

这说明系统提示词不仅可以约束“是否用工具”，还可以约束“怎样高效用工具”。

可学习写法：

```text
当多个工具调用彼此独立时，优先批量并行执行。当某类工具能减少上下文占用或提高准确性时，优先使用它。
```

设计启发：工具调用规则应该关注成本、顺序、依赖关系和失败处理，而不只是列出可用工具。

### 10. 环境注入

提示词末尾注入运行环境、模型、知识截止日期、git 状态、分支和最近提交。

这部分不是永久规则，而是“会话上下文”。它告诉 Agent 当前工作环境是什么，避免 Agent 反复查询基础状态。

设计启发：系统提示词可以拆成两层：

- 稳定规则：身份、安全、风格、工作流
- 动态上下文：日期、目录、平台、git 状态、模型信息

后者适合运行时生成，不适合人工写死。

## 可复用模式

### 模式一：身份 + 产品形态 + 任务范围

适合所有 Agent 产品。

```text
你是一个 [产品形态] 中的 [Agent 类型]，帮助用户完成 [任务范围]。你应结合 [可用工具/上下文] 完成任务。
```

### 模式二：允许/拒绝双边界

适合安全、金融、医疗、代码执行等高风险领域。

```text
拒绝 [禁止行为]。允许 [合规帮助范围]。当任务落入灰区时，优先 [保守动作]。
```

### 模式三：界面驱动的回答风格

适合 CLI、IDE、浏览器侧边栏、移动端等强界面约束产品。

```text
由于你的回答会显示在 [界面] 中，请保持 [长度/格式/语气]。除非用户要求，不要 [不适合该界面的输出形态]。
```

### 模式四：行动边界

适合能修改文件、发请求、执行命令的 Agent。

```text
当用户明确要求实施时，主动执行必要步骤。当用户询问方案、解释或权衡时，先回答问题，不要直接改变环境。
```

### 模式五：工程任务闭环

适合代码 Agent。

```text
处理工程任务时，先理解现有代码和约定，再实施改动；完成后运行可用验证命令。除非用户明确要求，不要提交代码或执行高影响操作。
```

## 不适合直接照抄的地方

1. 产品链接和反馈入口属于 Claude Code，应替换为自己的产品文档和反馈渠道。
2. “少于 4 行”的极简输出适合 CLI，但不适合教学型网站、知识库或内容生成产品。
3. TodoWrite、WebFetch、Task 等工具名依赖具体运行环境，复用时应替换为自己的工具系统。
4. 模型名、日期、git 状态属于运行时上下文，不应该静态写死。
5. “绝不要添加注释”是 Claude Code 的产品选择，不一定适合所有代码生成场景。

## 适合进一步拆成课程的点

- 如何写 Agent 的身份定义
- 如何写工具调用策略
- 如何给代码 Agent 设计安全边界
- 如何根据产品界面设计响应风格
- 如何把开放式任务变成可追踪任务流
- 如何区分稳定系统规则和动态运行上下文

## 对这个项目的展示建议

在详情页里，这份 prompt 可以重点展示三块：

1. 左侧结构目录：身份、安全、风格、约定、任务管理、工具策略、环境上下文。
2. 中间原文/译文切换：用户可以对照阅读。
3. 右侧学习卡片：每个章节给“解决什么问题”“可复用写法”“不建议照抄”。

这样用户不会只是复制 Claude Code 的系统提示词，而是能学会如何设计自己的代码 Agent。
