# Warp.dev System Prompt 学习分析

## 一句话定位

这是 Warp（AI 终端）中 Agent Mode 的系统提示词：一个**以终端/CLI 为唯一交互界面**的编码型 Agent，核心目标是用安全、克制、可控的方式在终端里完成软件开发任务。

## 为什么值得学习

- **场景边界极其清晰**：开篇就把自己钉死在“终端环境、无浏览器、只能用提供的工具”上。这种对运行环境的明确声明，能有效抑制模型幻想出不存在的能力。
- **“问题 vs 任务”二分法**：在动手之前先判断用户是“问怎么做”还是“让你去做”，这是一个非常实用的意图分流模式，避免 Agent 在用户只想了解时擅自执行命令。
- **安全规则密度高且具体**：从密钥处理、不安全命令、分页器（pager）陷阱到“不要用 cat 读文件”，都给出了可执行的细则，而非空泛的“要安全”。
- **工具使用规则贴近真实终端痛点**：例如禁止交互式/全屏命令、强制 `--no-pager`、避免 `cd`、用 `read_files` 而非 `cat`——这些都是终端 Agent 在生产中真实踩过的坑。

## 结构拆解

### 1. 身份定义

开头一句话定义身份（Agent Mode @ Warp），紧接着用两条 `IMPORTANT` 划出红线：拒绝恶意意图、交互界面是终端且没有浏览器。
**设计价值**：身份 + 环境约束 + 能力边界三件事在前 6 行就说清楚，让模型在后续所有推理中都带着“我在终端里、我没浏览器”的前提。

### 2. 安全边界

安全规则分散且贯穿全文，但都很具体：

- **恶意/有害任务**：直接拒绝（多处重复强调，体现优先级）。
- **不安全命令**：默认强烈规避，仅在用户明确要求且场景安全（如本地开发数据库）时才放行——给出了“例外的判断标准”而非一刀切。
- **密钥处理**：要求用环境变量中转（`API_KEY=$(secret_manager ...)`），绝不 `echo` 读取明文；遇到星号脱敏输入时用 `{{secret_name}}` 占位并提示用户替换。
  这是整篇里最精彩的可复用片段。

### 3. 工具调用

为每个工具（`run_command`、`read_files`、`grep`、`file_glob`、`edit_files`）单独列出使用规则，而不是笼统说“合理使用工具”：

- `run_command`：禁交互/全屏、强制非分页、避免 `cd`、URL 需安全。
- `read_files`：合并相邻行范围、单请求包含多段、5000 行分块策略。
- `grep`：查询必须是 ERE 并转义特殊字符。
- `edit_files`：精确字符串 search/replace、禁止 `// ... existing code ...` 省略、禁止行号、移动代码用两个块。

**设计价值**：把工具的“失败模式”前置写进规则（比如 edit 用省略号会失败、cat 会破坏上下文），相当于把踩坑经验固化成约束。

### 4. 输出风格与文件路径

- 输出必须是纯文本，除 `<citations>` 外不允许 XML 标签。
- 引用文件需遵循相对/绝对路径规则，并给出大量正反示例。
- 引用外部上下文或用户规则时必须用 `<citations>` schema 标注来源。
  **设计价值**：引用机制让 Agent 的结论可追溯，适合需要审计的工程场景。

### 5. 工作流与任务完成

- **意图分流**：问题→只给指引并询问是否代为执行；任务→按复杂度分简单/复杂处理。
- **任务完成纪律**：精确完成所请求的事，不擅自 commit/push、不擅自构建；同时又强调“倾向于行动，不要事事先确认”。
  **设计价值**：这是对“过度主动”与“过度保守”两个极端之间的精细平衡，用具体例子（修完 bug 别自动 push）落地。

## 可复用模式

1. **环境能力声明前置**：在 prompt 开头明确“我能用什么、不能用什么”，抑制工具幻觉。
2. **意图二分流**：先判断“咨询”还是“执行”，再决定是否动手。
3. **逐工具规则化**：为每个工具写明使用边界与失败模式，而不是泛泛要求。
4. **密钥环境变量中转模板**：`SECRET=$(secret_manager ...)` + `{{secret_name}}` 占位，可直接搬到任何 CLI Agent。
5. **分页器规避**：终端 Agent 通用——强制 `--no-pager` 或管道接 `cat`。
6. **引用可追溯**：`<citations>` 强制标注外部上下文来源。

## 不适合直接照抄的地方

- **工具名是 Warp 私有的**（`run_command`、`read_files`、`edit_files`、`file_glob` 等），换平台需替换为自己的工具命名。
- **硬编码的工作目录** `C:\Users\jmoya\Desktop` 明显是某次快照泄漏出的具体环境值，照抄会误导模型，必须改成动态注入。
- **“5000 行分块”策略**绑定了 Warp 自身工具的返回上限，不通用。
- **“绝不用 cat 读文件”**这条是 Warp 上下文管理机制的特例，其他 Agent 若没有等价的 `read_files` 机制不必照搬。
- 部分规则有笔误（如 `retieve_file`），说明这是真实泄漏的快照而非官方精修版，引用时应保持怀疑。

## 适合进一步拆成课程的点

- **「问题 vs 任务」意图分流**：如何用 prompt 让 Agent 在执行前正确分流。
- **CLI Agent 的密钥安全范式**：环境变量中转 + 脱敏占位的完整实践。
- **工具规则编写法**：以“失败模式驱动”的方式为每个工具写约束。
- **终端 Agent 的分页器/交互式命令陷阱**：为什么必须 `--no-pager`、禁交互。
- **任务完成的主动性平衡**：在“别擅自做多”和“别事事确认”之间如何拿捏。

## 对这个项目的展示建议

- 作为**「代码 Agent / 终端 Agent」类目的代表案例**展示，与 IDE 助手、浏览器 Agent 形成对照，凸显“运行环境如何塑造 prompt 设计”。
- 重点高亮**密钥处理段落**和**逐工具规则段落**，它们是最具教学价值、可直接迁移的片段。
- 标注本文为**未注明日期的泄漏快照（含硬编码路径与笔误）**，提醒读者它体现的是设计思路而非官方权威文本，引用细节需自行核验。
