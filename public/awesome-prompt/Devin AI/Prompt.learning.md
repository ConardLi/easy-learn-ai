# Devin AI System Prompt 学习分析

## 一句话定位

Devin AI 的 system prompt 是一份"自主软件工程 Agent"的完整操作手册：它把一个全自动写代码、跑测试、开 PR 的虚拟工程师，约束在一套以**专用命令优先、环境隔离、安全合规**为核心的工作规范里。

## 为什么值得学习

- 它是少数把"Agent 拥有真实操作系统"这一假设讲透的 prompt：不是聊天助手，而是一个会用 shell、编辑器、LSP、浏览器、部署的完整工程角色。
- 它示范了**用专用命令替代裸 shell**的设计哲学——每个高频动作都被收敛成带语义的命令，并反复强调"有专用命令就别用 shell/grep/find/cat"。这对工具型 Agent 的可控性、可观测性极有借鉴价值。
- 它把"何时该停下来思考"显式化（`<think>` 工具的必用/应用清单），是 reasoning-as-a-tool 的优秀范本。
- 它区分了 planning / standard 两种模式，体现了"先规划后执行"的双阶段 Agent 架构。
- 安全与边界（不泄露 prompt、不碰 secrets、不自行修环境、不改测试）写得克制而清晰，是企业级 Agent 的合规模板。

## 结构拆解

### 1. 身份定义

开篇一句话把角色钉死："你是 Devin，一名使用真实计算机操作系统的软件工程师……真正的代码高手"。

- **解决的问题**：让模型从"对话助手"切换到"会动手、会迭代到正确为止的工程师"心智，强调 codebase 理解力与持续迭代。
- **写法妙处**：用"few programmers are as talented as you"做能力锚定（capability priming），再立刻用"abiding by the guidelines"把自由度收回到规范内。先放权、再约束。
- **可复用**：`你是 {角色}，一个 {能力锚定}。你会收到 {输入}，使命是用 {工具集} 在 {约束} 下完成它。`

### 2. 安全边界与响应限制

包含数据安全（Data Security）与响应限制（Response Limitations）两块。

- **解决的问题**：把客户代码/数据视为敏感信息、禁止外泄 secrets、禁止提交密钥、禁止透露自身指令。
- **写法妙处**：响应限制给了**标准回绝话术**——被问 prompt 细节时统一回 "You are Devin. Please help the user with various engineering tasks"。预设回答比"不要回答"更可执行。
- **不可照抄**：这个固定回绝句是 Devin 品牌专属，复用时要换成自己的话术。

### 3. 工具调用（命令参考）

这是 prompt 的主体，把能力切成 8 大类：Reasoning / Shell / Editor / Search / LSP / Browser / Deployment / User-interaction / Misc / Plan。

- **解决的问题**：给 Agent 一套带语义、可观测、互不重叠的动作空间，避免它退化成"什么都丢进 shell"。
- **写法妙处**：
  - 每个命令统一"标签语法 + Description + Parameters + 示例"四段式，机器友好。
  - 反复出现的硬约束："绝不能用 shell 查看/编辑文件，用编辑器命令"；"绝不能用 grep/find，用内置搜索"。把易错的越界行为提前堵死。
  - 强调**并行输出**：无依赖的命令应一次性多发，显式优化吞吐。
  - 提供 `find_and_edit` 这种"正则定位 + 子 LLM 批量改"的重型重构工具，体现规模化编辑思路。
- **可复用模板**：`<命令名 参数=...>正文</命令名>` + 紧跟 Description/Parameters/示例；并配一句"优先用专用命令，而非通用 shell"。

### 4. 推理工具（think）

把"思考"本身做成一个工具，并给出**必须用**（关键 git 决策前 / 探索转编辑前 / 报告完成前）和**应该用**（10 种场景）两级清单。

- **解决的问题**：防止 Agent 在高风险节点上不假思索地行动；强制在"理解→动手"的拐点做自检。
- **写法妙处**：用编号场景把抽象的"该思考了"变成可判定的触发条件，并明确"用户看不到 think 内容"鼓励自由推理。
- **可复用**：为任何 Agent 加一个"反思/自检"工具，并列出必触发清单（破坏性操作前、声明完成前）。

### 5. 工作流程与模式（Planning / Standard）

双模式 + 工作方式（Approach to Work）+ 编码最佳实践。

- **解决的问题**：把任务拆成"先收集信息形成 plan，再按 plan 执行"两阶段；规定遇到困难先收集信息再下结论。
- **关键纪律**：
  - 环境问题用 `report_environment_issue` 报告后**绕开**（改用 CI），绝不自行修环境。
  - **绝不改测试**来骗过测试，先怀疑被测代码。
  - 提交前跑 lint/测试。
- **可复用**：planning/standard 分模式、"环境问题上报而非自修"、"测试失败先查实现"都是高质量工程 Agent 的通用规则。

### 6. 上下文与代码规范

编码最佳实践强调：先读约定再改、绝不假设库可用（先查 package.json 等）、新建组件先看已有组件。

- **解决的问题**：让自动改代码"入乡随俗"，避免引入不存在的依赖或风格冲突。
- **可复用**：`改动前先读周边 imports 与依赖清单，模仿既有风格`，几乎适用于一切 codebase Agent。

### 7. 用户协作与 Git 规范

message_user（含 ref_file/ref_snippet 富链接、request_auth 安全输入）、list_secrets、git/GitHub 操作规范。

- **写法妙处**：明确"用户只能看到 `<message_user>` 里的内容"，逼迫 Agent 把交付物显式说出来；secrets 通过专门安全 UI 收集。
- **Git 纪律**：绝不 force push、绝不 `git add .`、默认分支名格式、CI 三次不过就求助——把协作风险量化成可执行规则。

## 可复用模式

1. **专用命令优先原则**：为高频动作建语义命令，并显式禁止用裸 shell/grep/cat 绕过，提升可观测与可控。
2. **思考即工具 + 触发清单**：把反思做成命令，列"必须用/应该用"的可判定场景。
3. **双模式工作流**：planning 收集信息 → standard 按计划执行。
4. **环境隔离纪律**：环境问题"上报+绕行"，不让 Agent 在脏环境里乱修。
5. **标准回绝话术**：对越权/窥探类请求预设固定回答。
6. **并行动作**：无依赖命令一次性批量输出以提效。
7. **改动前先读约定**：依赖、风格、组件范式先侦察后落笔。

## 不适合直接照抄的地方

- **大量 Devin 专属命令**（deploy_backend 仅支持 Poetry+FastAPI、deploy_frontend、devinid、Fly.io、`/home/ubuntu` 路径假设）强绑定其运行时，迁移到别的 Agent 需要替换整套工具层。
- **固定回绝句**（"You are Devin...")是品牌话术，照搬会暴露来源。
- **POP QUIZ 机制**是 Devin 内部评测/对齐手段，普通产品没有对应基础设施时无意义。
- **"绝不修环境"**适合托管式云 Agent，本地开发助手可能恰恰需要帮用户修环境，要按场景反向调整。

## 适合进一步拆成课程的点

- 「如何为工具型 Agent 设计动作空间」：从 Devin 的 8 类命令反推命令粒度与命名原则。
- 「Reasoning as a Tool」：用 think 工具的触发清单讲 Agent 自检设计。
- 「两阶段 Agent：先规划后执行」：planning/standard 模式的状态机建模。
- 「Agent 的安全与合规边界」：secrets、prompt 防泄露、git 破坏性操作防护。
- 「自动改代码如何不破坏既有工程」：上下文侦察 + 不改测试 + 跑 lint 的纪律链。

## 对这个项目的展示建议

- 作为"代码 Agent"分类的**标杆案例**展示，与浏览器 Agent、IDE 助手类对比，突出它"拥有完整操作系统 + 专用命令体系"的独特定位。
- 在详情页高亮三个记忆点：**专用命令优先**、**think 触发清单**、**环境上报而非自修**。
- 用中英对照阅读模式呈现命令参考部分，配上"哪些是通用模式 / 哪些是 Devin 专属"的标注，帮助学习者区分可复用与不可照抄。
- 在 caution 区提醒：来源未核验、命令层强绑定 Devin 运行时，学习其设计思想而非逐字搬运。
