# Cline System Prompt 学习分析

## 一句话定位

一份“VSCode/IDE 内执行型代码 Agent”的系统提示词：用**强约束的工具调用协议（XML + 一次一工具 + 等待确认）**、**文件编辑两段式策略（write_to_file / replace_in_file）**与**运行环境边界（cwd、禁止 cd、environment_details）**，把模型行为固定成可控的工程工作流。

## 为什么值得学习

1. **把工具调用做成“可解析协议”**：用 XML 标签定义工具调用格式，天然适配解析器/中间层，避免“模型自由发挥”导致的执行失败。
2. **把“可靠执行”写进流程**：强调一次一工具、等待用户确认、每一步由上一步结果驱动，能显著降低连锁错误与不可逆操作风险。
3. **把文件修改规范化**：明确 write_to_file（整文件）与 replace_in_file（局部）各自适用场景，并给出精确的 SEARCH/REPLACE 规则，等于把“补丁策略”固化为 Prompt 模块。
4. **把“运行环境约束”外显**：cwd、禁止 cd、环境信息、活跃终端等都写入规则，属于 IDE Agent 在现实工程环境里最常见的坑位。

## 结构拆解

### 1. 身份定义：把模型定位成“工程师”而不是聊天助手

开头一句 “highly skilled software engineer” 是典型的“角色能力锚定”，为后续的工程化规则提供合理性基础：用户期待你像工程师一样行动，而不是泛泛解释。

### 2. 工具调用协议：格式、节奏、确认机制三件套

这一部分有三个关键点：

1) **格式（可解析）**：强制 XML 风格 `<tool><param>...</param></tool>`，并多次强调“Always adhere”。

2) **节奏（一次一工具）**：明确“one tool per message”，并要求 step-by-step；这能降低：
- 串行依赖丢失（中间一步失败导致后续误执行）
- 工具输出不可预测带来的级联错误

3) **确认（等待用户结果）**：强调“receive the result in the user's response”“ALWAYS wait for user confirmation”，把人类确认作为安全阀。

可复用模板（适合抽成你站点的“模式卡”）：

```text
工具调用节奏：
- 一次只调用一个工具
- 每一步必须基于上一步结果
- 未得到用户确认前，不假设成功
```

### 3. 工具清单设计：把“能力边界”写在 schema 里

Prompt 对每个工具的描述并不只是说明功能，而是顺带写入了“策略”：

- execute_command：用 requires_approval 区分高风险与低风险动作（安装/删除/网络 vs 读取/构建/启动 dev server）
- read_file：强调适用文本/PDF/DOCX，不适合二进制
- write_to_file：强调必须提供完整文件内容（防止截断导致文件损坏）
- replace_in_file：强调 EXACT 匹配、只替换首个命中、按文件顺序列出多块、不得截断行
- search_files / list_files / list_code_definition_names：提供“先结构后细节”的探索路径

学习点在于：**工具描述本身就是行为策略的载体**。你在做索引/课程时，可以把这些“工具策略句子”单独摘出来作为可复用片段。

### 4. 浏览器能力（条件注入）：把能力做成可插拔模块

`supportsComputerUse ? ... : ""` 这种写法很值得学：同一份系统提示词可以在不同产品形态（是否支持浏览器/截图）下启用不同模块。

这对你的项目有启发：可以在学习分析里引导读者理解“模块化提示词”的实现方式：

- 固定核心：工具协议、编辑策略、规则、目标
- 条件增强：browser_action 的坐标点击规则、启动/关闭约束、浏览器独占期（active 时不能用其他工具）

### 5. 文件编辑策略：把“改代码”变成两类动作

Prompt 明确告诉 Agent：

- 默认用 replace_in_file 做局部修改（更安全）
- 大改/新建用 write_to_file（一次性重写）
- 并且专门补充“自动格式化”会改变文件状态 → 后续 SEARCH 必须以最终文件为准

这是典型的“失败恢复/稳健性”设计：不是只告诉你怎么改，而是告诉你 **为什么会改失败**（格式化、匹配不准、块太大）以及怎么避免。

可复用模板：

```text
改文件优先级：
1) 能局部改就局部改（replace_in_file）
2) 需要重构/生成模板才整文件写（write_to_file）
3) 注意格式化后的最终状态作为下一次 SEARCH 基准
```

### 6. 模式切换：PLAN/ACT 双阶段

这份 Prompt 把“思考/规划”和“执行/落地”分开：

- PLAN MODE：允许交流、画图、澄清、出计划
- ACT MODE：工具驱动、产出结果、attempt_completion 收口

这种设计特别适合 IDE/工程任务：能防止模型在不确定时直接动手改代码，也能降低用户参与成本（先看方案再执行）。

### 7. 规则与输出风格：降低产品体验噪音

有几条很“产品化”的规则：

- 禁止以 “Great / Certainly / Okay / Sure” 开头（统一语气、避免套话）
- attempt_completion 结果不以问题结尾（避免无休止对话）
- “看不到命令输出就假设成功” + 必要时让用户复制粘贴（针对 IDE 终端输出不稳定的现实妥协）

这些规则不是模型能力问题，而是“交互成本与可靠性”问题，值得在学习模块里重点标记为“输出风格 / 用户协作 / 失败恢复”。

## 可复用模式

### 模式 1：工具调用的可解析 DSL

```xml
<tool_name>
<param>value</param>
</tool_name>
```

要点：格式统一 → 工具层易解析 → 更容易做权限、审计、回放。

### 模式 2：风险分级的 requires_approval

```text
低风险：读文件、列目录、构建、启动 dev server → requires_approval=false
高风险：安装卸载、删改文件、网络操作、系统配置 → requires_approval=true
```

### 模式 3：局部替换的“精确匹配”约束

```text
SEARCH 必须逐字符匹配（含空格/缩进/换行）
每块只替换第一次命中
多块按文件顺序排列
SEARCH 不能是半行
```

### 模式 4：浏览器工具的“会话独占期”

```text
浏览器启动 → 期间只能 browser_action → 关闭 → 才能用其他工具
```

适合任何需要“状态机式交互”的工具（浏览器、远程桌面、交互式终端等）。

## 不适合直接照抄的地方

1. **占位符与模板表达式**：`${cwd.toPosix()}`、`${supportsComputerUse ? ... : ""}`、`${mcpHub...}` 依赖具体宿主环境。
2. **工具集合与命名**：execute_command / replace_in_file / attempt_completion 等是 Cline 的工具 API，不是通用标准。
3. **对终端输出的假设**：“看不到输出也当作成功”是产品妥协，换平台可能会造成误判。
4. **坐标点击策略**：browser_action 强依赖截图与分辨率，换成 DOM ref 模型或不同自动化框架需要重写。

## 适合进一步拆成课程的点

1. “一次一工具 + 等待确认”如何显著提升 Agent 稳定性（含反例演示）。
2. replace_in_file 的精确匹配为什么难用、如何写出稳健的 SEARCH 块。
3. PLAN/ACT 双阶段在人机协作中的作用：减少误改、提升可控性。
4. 工具描述即策略：怎么把“产品安全/权限/审计”写进工具 schema。

## 对这个项目的展示建议

1. 在页面里把 Prompt 的核心模块做锚点：**Tool Use Formatting / Tools / Tool Use Guidelines / Editing Files / Act vs Plan / Rules / Objective**。
2. 标签与筛选建议：`工具调用`、`IDE Agent`、`补丁策略`、`一次一工具`、`权限确认`、`浏览器（可选）`、`MCP`。
3. 结合你站点的“对照阅读”能力：把 `${...}` 模板段落高亮为“平台绑定内容”，帮助读者快速识别“可复用 vs 需替换”。
