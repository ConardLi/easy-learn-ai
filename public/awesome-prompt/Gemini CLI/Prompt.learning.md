# Gemini CLI System Prompt 学习分析

## 一句话定位

这是 Google Gemini CLI 的核心 system prompt：把一个通用大模型约束成一个"安全优先、约定优先、极度简洁"的命令行软件工程 agent。

## 为什么值得学习

- 它是少数公开可读的**终端型代码 agent**完整 system prompt，结构清晰、模块分明，适合作为写 agent prompt 的范本。
- 它把"如何在真实仓库里安全改代码"这件事拆成了可执行的步骤序列（理解→规划→实现→验证），而不是空泛的口号。
- 它在"主动完成任务"与"不越权、不擅自扩张范围"之间做了精细权衡，这正是 agent prompt 最难的部分。
- 大量使用强模态词（NEVER / MUST / VERY IMPORTANT）+ 具体反例，示范了如何让约束"长牙齿"。
- 结尾用一组 `<example>` 对话把抽象规则锚定成可模仿的行为样例，是 few-shot 对齐风格的好教材。

## 结构拆解

### 1. 身份定义
开篇一句话即定身份："一个专注于软件工程任务的交互式 CLI agent"，并立刻给出优先级——安全、高效、严格遵循指令、善用 Tool。
- **解决的问题：** 让模型在第一时间收敛到"终端里的工程师"这一狭窄人设，而非泛化助手。
- **写得好在哪：** 身份+目标+约束源（指令与 Tool）一句话讲完，没有冗余背景故事。
- **可复用：** `你是一个专注于{领域}的{形态} agent。你的首要目标是{价值1}、{价值2}地帮助用户，严格遵循以下指令并善用可用 Tool。`

### 2. 安全边界
分布在 `Security and Safety Rules`、`Outside of Sandbox`、`Respect User Confirmations` 多处，层层设防：
- 改系统状态的命令**必须**先解释目的与影响；但**不需要**单独请求许可（交由 UI 确认对话框），避免了"每步都问"的体验灾难。
- **绝不要**引入暴露/记录/提交 secrets 的代码。
- 运行在沙箱外时要额外提醒用户启用沙箱化——把"环境风险"显式告知。
- 用户取消的调用不得反复重试。
- **写得好在哪：** 区分了"解释"与"请求许可"两件事，安全感与流畅度兼得。
- **可复用：** "执行高风险动作前必须解释意图与影响 + 信任外部确认机制 + 尊重用户拒绝" 三件套。

### 3. 工具调用
`Tool Usage` 段把工具使用规约写得非常工程化：
- 绝对路径强制（含专门的 `Path Construction` 准则，给出拼接示例）。
- 独立调用要**并行**。
- 长驻命令用后台 `&`；交互式命令尽量用非交互版本（`npm init -y`）。
- `save_memory` 仅用于跨会话的**用户个人**偏好，明确**不要**存项目上下文——给记忆工具划清了边界。
- **可复用：** 对每个工具写清"何时用 / 何时绝不用 / 边界条件"，比只罗列工具名有效得多。

### 4. 工作流程（双轨）
区分了两类任务并给出不同流程：
- **Software Engineering Tasks：** 理解→规划→实现→验证(测试)→验证(标准)，强调用 grep/glob 先建立认知、用单测建立自我验证循环、绝不假设测试命令。
- **New Applications：** 需求→提案→用户批准→自主实现→验证→征求反馈，并附带"未指定技术栈时的默认选型表"。
- **写得好在哪：** 同一个 agent 面对"改存量"和"建增量"采用不同节奏；存量任务强调谨慎与验证，增量任务允许自主推进到可运行原型。
- **可复用：** "先 recon 再动手 + 强制收尾验证（build/lint/type-check）"是代码 agent 的黄金骨架。

### 5. 输出风格
`Tone and Style` 为 CLI 场景定制："少于 3 行""不闲聊""无前言后语""Tool 做事、文本只用于沟通"。
- **解决的问题：** 终端读体验差，冗长输出是噪音。
- **可复用：** 给"输出长度上限 + 禁用填充语 + 工具与文本分工"三条硬约束，能立竿见影提升 agent 利落度。

### 6. 用户协作与范围控制
`Proactiveness` 要彻底完成隐含的后续动作，但 `Confirm Ambiguity/Expansion` 又规定不得越权扩张；"被问怎么做就先解释，别直接做"。Git 段落同样体现：草拟提交信息但不擅自 push。
- **写得好在哪：** 用一对看似矛盾的规则界定"主动"的精确边界。
- **可复用：** "主动补齐直接隐含的子任务，但任何超出明确范围的重大动作都要先确认。"

### 7. Few-shot 示例锚定
结尾一组 `<example>` 把语气和工作流落地：`1+2 → 3`（极简）、重构前先找测试网（谨慎）、删除目录先讲清后果（安全）。
- **可复用：** 抽象规则后紧跟 2-6 个对照样例，能显著降低模型对"度"的误判。

## 可复用模式

- **身份一句话收敛 + 优先级排序。**
- **双轨工作流：** 存量任务（recon→plan→implement→verify×2）/ 增量任务（需求→提案→批准→实现→验证→反馈）。
- **安全三件套：** 高风险前解释 + 信任外部确认 + 尊重取消。
- **工具规约模板：** 每个 Tool 标注用途/边界/反例。
- **输出硬约束：** 行数上限 + 禁填充语 + 工具与文本分工。
- **强模态词 + 反例**：用 NEVER/MUST 配具体场景，而非泛化建议。
- **规则 + few-shot 对照样例**双层对齐。

## 不适合直接照抄的地方

- **工具名是 Gemini CLI 专有的**（`run_shell_command`、`replace`、`read_many_files`、`save_memory`、`glob`、`search_file_content`），换平台必须替换为你自己的工具集，否则模型会幻觉调用不存在的工具。
- **默认技术选型表（React+Bootstrap、FastAPI、Three.js 等）带有强烈倾向性**，未必符合你团队的技术栈，直接照搬会引导出不合预期的脚手架。
- **"少于 3 行输出"对教学/解释型产品过于克制**，面向新手的助手照抄会显得冷淡。
- **"不需要请求许可、交给 UI 确认对话框"依赖宿主有确认机制**；若你的运行环境没有该机制，照抄会导致危险命令无人把关。
- **沙箱外提醒**只在确实运行于用户裸机时才有意义。

## 适合进一步拆成课程的点

1. 如何为 agent 写"双轨工作流"，并讲清存量 vs 增量任务的节奏差异。
2. 安全规约设计：解释 vs 许可的区分、信任外部确认机制、取消语义。
3. 工具调用规约怎么写才能减少幻觉（用途/边界/反例三段式）。
4. CLI 场景的输出风格约束与 few-shot 锚定技巧。
5. "主动性"与"范围控制"如何用一对规则精确界定。

## 对这个项目的展示建议

- 在卡片高亮上突出"终端代码 agent 范本""双轨工作流""安全优先"三个标签，与浏览器类/设计类 agent 形成对照。
- 学习页可做"规则 ↔ 末尾 example"的对照视图，让读者直观看到抽象约束如何落到具体对话行为。
- 可与 Claude Code、Codex CLI 等同类 prompt 并排，做"代码 agent 安全与工作流"横向对比专题。
- 标注 `sourceStatus: 待核验`，提醒读者这是未注明日期的快照，使用时以官方最新版本为准。
