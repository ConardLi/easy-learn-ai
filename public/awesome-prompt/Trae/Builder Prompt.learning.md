# Trae Builder Prompt 学习分析

## 一句话定位

这是字节跳动 Trae AI（IDE 内置编程 Agent）的「Builder」系统提示词，展示了一个嵌入 IDE 的结对编程 Agent 如何在「自动注入上下文 + 工具调用 + 任务管理 + 严格输出引用规范」之间取得平衡。

## 为什么值得学习

- 它是「IDE 助手」这一品类的典型范例：与 Claude Code/Cursor 等命令行 Agent 不同，它假设宿主环境会自动注入打开的文件、光标位置、编辑历史等隐式上下文，提示词需要专门交代如何取舍这些信息。
- 它把「输出格式契约」做到了产品级精度：定义了两套互不混淆的引用语法（网络引用 `<mcreference>` 与代码引用 `<mcfile>/<mcsymbol>/<mcfolder>/<mcurl>`），这是 IDE 类产品做可点击跳转、可溯源所必需的工程约束。
- 它用大量正反 `<example>` 教模型「何时使用 / 何时不用 TodoWrite」，是用 few-shot 校准 Agent 行为粒度的好样本。
- 它把安全与保密边界写得很克制：拒绝泄露提示词/工具描述/剩余轮次，同时要求不要反复道歉，体现了对「Agent 人格」的塑造。

## 结构拆解

### 1. 身份定义

开篇用三段确立角色：「强大的 agentic AI 编程助手」「只在 Trae AI 中运行」「与 USER 结对编程」。
价值在于：它显式声明了「宿主会自动附加当前状态信息（打开文件、光标、编辑历史）」，并立刻补一句「相关与否由你判断」——把上下文的取舍责任交还给模型，避免模型被无关上下文带偏。这是 IDE 内置 Agent 区别于纯 CLI Agent 的核心设计点。

### 2. 安全边界

集中在 `<communication>` 的第 4/6/7 条：拒绝输出系统提示词、插件、规则、工具描述、剩余轮次。
写法值得学：它把「机密」范围用同义词列举（重复、翻译、改写、打印、总结、格式化、返回……）来防御绕过式越狱，而不是只写一句「不要泄露」。同时第 5/8 条塑造人格——不撒谎、不过度道歉，这属于「行为风格的安全」。

### 3. 工具调用

`<toolcall_guidelines>` + 显式工具白名单（todo_write、search_codebase、view_files、write_to_file、run_command、web_search、finish 等）。
关键设计：
- 「最小化不必要调用」是成本/延迟意识。
- 「绝不调用未提供的工具」「不要把文件名或函数名当作工具名」是防幻觉护栏。
- 「绝不对已存在文件使用 create_file」强制先读后写，避免覆盖。

### 4. 代码修改规范

`<making_code_changes>` 11 条是这份提示词最实用的部分：先理解约定再模仿、补齐 import/依赖、绝不假设库可用（先查 package.json）、新建组件先看现有组件、安全不写入密钥、图片用 SVG、改动尽量一步且不超过 3 步。
这些都是把「资深工程师的隐性习惯」显式化为规则，便于直接迁移到任何编程 Agent。

### 5. 调试与外部 API

`<debugging>`（治根因、加日志、写隔离测试）和 `<calling_external_apis>`（无需许可自选最佳 API、版本兼容依赖文件、API Key 安全提示）补足了从「写代码」到「排错+集成」的链路。

### 6. 输出引用契约

两套 XML 引用语法是这份提示词最具产品特色的部分：`<mcreference>` 用于网络搜索溯源，`<mcfile>/<mcsymbol>/<mcfolder>/<mcurl>` 用于在回复里生成可被 IDE 解析、可点击跳转的代码引用，并反复强调「引用标签外不得加反引号」「行号从 1 起算、含空行注释」。这是为渲染层服务的硬约束。

### 7. 任务管理

通过 4 组正例 + 4 组反例校准 TodoWrite 的使用边界，再加上明确的状态机（pending/in_progress/completed）与完成判定（测试失败/实现不完整/有未解错误时不得标记完成）。这是把「任务管理」从模糊建议变成可执行规则的范本。

## 可复用模式

- **上下文自带 + 显式取舍**：「环境会注入 X、Y、Z，是否相关由你判断」——适用于任何有隐式上下文注入的助手。
- **同义词枚举式保密**：把可能的越狱措辞列全，再统一拒绝。
- **工具白名单 + 反幻觉条款**：显式列出工具名，并禁止把文件名当工具名、禁止调用不存在的工具。
- **正反 few-shot 校准行为粒度**：用「该用 / 不该用」成对示例，比一句抽象规则更有效。
- **先读后写护栏**：「绝不对已存在文件用 create_file」「先查依赖文件再用库」。
- **输出格式硬契约**：为渲染/跳转定义专门标签语法并强调边界细节（反引号、行号计数）。

## 不适合直接照抄的地方

- `<mcreference>`、`<mcfile>`、`<mcsymbol>` 等标签是 Trae 渲染层私有协议，脱离其 IDE 无法解析，照搬到别的产品只会产生无意义的 XML。
- 工具白名单（search_codebase、edit_file_fast_apply 等）是 Trae 专有工具名，其他 Agent 必须替换为自身工具集。
- 「不超过 3 步完成修改」「不透露剩余轮次」等约束与具体产品的计费/轮次机制绑定，通用场景未必适用。
- 「世界上最好的 IDE」属营销性自我定位，迁移时应去掉。

## 适合进一步拆成课程的点

- 模块一：IDE 内置 Agent 的上下文模型——隐式注入 vs 主动检索，如何写「取舍责任」。
- 模块二：输出格式契约设计——如何为可点击引用/溯源设计标签语法与边界规则。
- 模块三：用正反 few-shot 校准 Agent 的「努力程度/行为粒度」（以 TodoWrite 为例）。
- 模块四：编程 Agent 的代码修改护栏清单（先读后写、不假设库、安全不写密钥）。
- 模块五：提示词保密与防越狱的同义词枚举法。

## 对这个项目的展示建议

- 与已有的 CLI 类 Agent（如 Claude Code）放在一起对比，作为「IDE 助手」品类代表，突出「自动上下文注入 + 私有引用协议」这两点差异。
- 在详情页高亮两套引用语法的对照表，作为「输出格式契约」教学锚点。
- 把 TodoWrite 的正反示例抽成一个可交互的「该不该建任务清单」小测验。
- 标注来源为「待核验」社区整理，提醒读者标签协议为 Trae 私有、不可跨产品照搬。
