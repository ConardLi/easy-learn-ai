# Trae Chat Prompt 学习分析

## 一句话定位

Trae Chat Prompt 是字节跳动 Trae IDE 中"对话/编程助手"的系统提示词，核心是定义一个运行在 agentic IDE 内、采用 AI Flow 范式、与用户结对编程的 AI 编程助手，并用大量"输出格式契约"约束它如何回复代码与命令。

## 为什么值得学习

- 它是一个**典型的 IDE 内嵌编程助手提示词**：和通用聊天助手不同，它假设自己有"宿主环境"（IDE 会替它执行工具），从而把执行能力和决策能力解耦。
- 它把**输出格式当作硬契约**来写。比如统一用 `// ... existing code ...` 占位符表示未改动代码，这是 IDE 做 diff/局部应用补丁的关键约定，非常能体现"提示词为下游解析器服务"的工程思维。
- 它展示了一个**工具列表为空时的优雅降级**写法：即使当前没有可用工具，仍保留完整的工具调用规范，方便后续注入工具。
- 它把安全、版权、隐私、范围限制等边界条件以编号清单方式集中表达，便于学习者对照自己的产品做"边界 checklist"。

## 结构拆解

整个提示词用 XML 风格的标签分块（`<identity>` / `<purpose>` / `<tool_instruction>` / `<guidelines>`），这种分块本身就是一种值得学的组织方式：每一块职责单一，便于维护和被程序裁剪。

### 1. 身份定义（identity / purpose）

- `identity` 用三句话锚定身份：产品名（Trae AI）、运行环境（agentic IDE）、协作模式（与用户结对编程），并点明任务范围（建库 / 改库调试 / 答疑）。
- `purpose` 补充了一个关键的**控制流约定**：助手要先决定"是否需要工具"，再据此设置标志位，然后要么输出工具参数、要么输出回复文本。这其实是把一个"路由决策"写进了人格设定里。
- 学习点：身份段不只是"你是谁"，更是在交代"你在什么系统里、按什么循环工作"。

### 2. 安全边界（reply_guideline 中的若干条）

- 不撒谎、看不到上下文就请用户提供（第 2 条）——抑制幻觉。
- 不输出侵权内容（第 13 条）。
- 政治敏感与个人隐私话题直接拒答（第 14 条）。
- 范围限制：只做软件开发相关话题，无关问题就提醒"我是 AI 编程助手"（第 17 条）。
- 不擅自修改用户已有注释（第 9 条）——尊重用户既有代码。
- 学习点：这些边界都写得"可执行"（给出了具体动作），而不是空泛的价值观宣言。

### 3. 工具调用（tool_instruction）

- 三段式：`tool_list`（当前为空）+ `toolcall_guideline`（调用纪律）+ `tool_parameter_guideline`（参数纪律）。
- 调用纪律强调：严格遵循 schema、绝不调用不存在的工具、不向用户暴露工具内部信息、调用前先盘点并比较候选工具再择优。
- 参数纪律强调：不编造、不追问可选参数、用户给的精确值原样使用、从描述性词语里推断必填参数。
- 学习点：把"工具治理"拆成"能不能调用"和"参数怎么填"两层，是很清晰的责任划分。

### 4. 输出格式契约（reply_guideline + code_reference_guideline）

- 最硬的约定是 `// ... existing code ...` 占位符：必须精确、唯一、不随语言变化。这是为了让 IDE 能可靠地把模型输出 diff 回真实文件。
- 代码块要标注语言 ID 和文件路径；现有文件还要重述所属方法/类，并给绝对/相对路径回退策略。
- 终端命令默认面向 Windows、每块只一条命令、注意路径分隔符——典型的"减少歧义、便于直接执行"。
- 学习点：当提示词的输出会被机器消费时，格式规则往往比内容规则更重要。

### 5. 引用规范（web_citation / code_reference）

- 区分两套引用体系：网页搜索结果引用 vs. 代码/符号/文件/文件夹引用，并反复强调"两者完全分开、不要混用"。
- 学习点：当系统里存在多个相似机制时，提示词需要显式划清边界，否则模型容易串用。（注：原文这部分因模板渲染丢失了具体 XML 标签示例，呈现得较为残缺。）

## 可复用模式

- **XML 标签分块**：用 `<identity>`、`<tool_instruction>` 等语义标签切分提示词，方便程序按块注入/裁剪。
- **占位符 diff 契约**：定义一个唯一、不可变的占位符来表示"未改动代码"，让下游能做局部补丁应用。
- **工具治理双层模型**：`是否调用`（toolcall_guideline）与 `参数如何填`（tool_parameter_guideline）分开写。
- **空工具优雅降级**：保留工具规范但声明"当前无工具，勿生成 toolcall"。
- **可执行的边界清单**：每条安全/范围限制都附带明确动作（拒答 / 提醒 / 请用户提供上下文）。

## 不适合直接照抄的地方

- **默认输出 Windows 命令**：这是 Trae 特定的产品假设，多数项目应改为根据实际操作系统判断。
- **`// ... existing code ...` 占位符**：只有当你的客户端真的实现了对应的补丁应用逻辑时才有意义，否则只是无效约定。
- **AI Flow / agentic IDE 等品牌化措辞**：属于 Trae 专属叙事，迁移到别的产品要替换。
- **残缺的引用示例**：原文 `web_citation_guideline` 与 `code_reference_guideline` 的具体格式示例已丢失，不能照抄，需要补全实际标签。

## 适合进一步拆成课程的点

- 「如何为 IDE 编程助手设计 diff 输出契约」——以占位符规则为例。
- 「身份段如何兼任控制流路由」——以 purpose 的"先判断是否用工具"为例。
- 「工具调用纪律：schema 严格性、防幻觉、防越权暴露」。
- 「可执行的安全边界清单怎么写」——对比空泛价值观 vs. 带动作的规则。
- 「多套引用机制如何防止混用」。
