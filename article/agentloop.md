你跟 ChatGPT、DeepSeek 聊天的时候，交互模式是这样的：
你说一句 → 它回一句 → 结束
一问一答，单次往返。这叫 Single-pass（单轮响应）。
绝大多数聊天场景下，够用了。你问"量子纠缠是什么"，它给你解释清楚，完事儿。
但有些事情，一个来回搞不定。比如你说：
"帮我查一下最近三个月 A 股涨幅最大的 10 只股票，分析它们的行业分布，然后做一张可视化图表。"
这里面有搜索、有筛选、有数据处理、有图表生成。
中间任何一步的结果都会影响下一步怎么做。出了错还得回头调整。
单轮响应干不了这事。
Agent Loop 就是为了解决这个问题而存在的。

---
Agent Loop 是什么
Agent Loop 是 AI Agent 的迭代执行循环。
LLM 在循环中反复推理、调用工具、观察结果，直到任务完成或达到终止条件。
听起来抽象？拆成伪代码就六行：
while not done:
    response = call_llm(messages)
    if response has tool_calls:
        results = execute_tools(response.tool_calls)
        messages.append(results)
    else:
        done = True
return response
就是一个 while 循环。
LLM 每跑一轮，要么决定调用某个工具继续干活，要么判断任务完成、输出最终结果。循环会一直转，直到它认为"够了"或者触发了你设置的停止条件（比如最大迭代次数、token 预算）。
Anthropic 的工程团队有个说法挺到位的："Agent 通常就是 LLM 在一个循环里基于环境反馈使用工具。"

---
循环里到底在发生什么
每一轮循环，内部跑五个阶段：
[图片]
1. 感知（Perceive）
Agent 接收输入。可能是用户的原始指令，可能是上一轮工具返回的结果，也可能是一个报错信息。
这些都会被塞进上下文（Context），送给 LLM。

---
2. 推理（Reason）
LLM 拿到当前所有上下文后，开始思考：现在什么状况？目标达成了吗？下一步该干什么？
这一步是整个循环的大脑。

---
3. 规划（Plan）
对于复杂任务，LLM 会先做任务拆解——把大目标分成几个子步骤，排好优先级。简单任务可能直接跳到执行。

---
4. 行动（Act）
Agent 执行具体操作：调一个 API、查一次数据库、跑一段代码、发一个 HTTP 请求。
这些操作都通过"工具"（Tools）来完成。
工具的定义包含名称、描述和参数 schema，LLM 决定调哪个、传什么参数。

---
5. 观察（Observe）
工具返回结果后，Agent 检查：成功了吗？数据对不对？够不够？任务完成了吗？
没完成的话，把结果追加到上下文里，回到第 1 步，继续转。
五个阶段，不断循环。这就是 Agent Loop。

---
看一个具体的例子
任务：找到 2026 年引用量最高的 Agent Memory 论文，总结核心发现。
第一轮：
- 推理：我需要先搜论文
- 行动：调用学术搜索 API，关键词 "agent memory 2026"
- 观察：返回 15 篇论文及引用数

---
第二轮：
- 推理：引用最高的那篇 340 次，我需要拿到全文
- 行动：调用文档检索工具
- 观察：成功拿到摘要和核心章节

---
第三轮：
- 推理：信息够了，可以写总结了
- 行动：不需要调工具，直接生成
- 观察：输出最终答案，退出循环

---
三轮。三次工具调用。一个完整答案。
用 Single-pass 根本没法做 — 你不可能在一次 LLM 调用里同时搜索、检索、总结。

---
三种主流的循环模式
Agent Loop 是统称。具体到"LLM 在循环里怎么思考和行动"，业界演化出了几种路子。
ReAct：边想边做
全称 Reasoning + Acting。2022 年普林斯顿和谷歌联合提出。
机制很直觉：每一步先想（Thought），再做（Action），然后看结果（Observation）。
想完做，做完看，看完再想。
Thought: 我需要查询 A 公司的市值
Action: search("A 公司市值")
Observation: A 公司市值 5000 亿
Thought: 现在我需要对比 B 公司……
优点： 灵活。每一步都可以根据上一步的结果调整策略。碰到意外能随时应变。
缺点： 每轮都调 LLM 做推理，token 消耗高，延迟也高。步骤越多越费钱。
ReAct 在 ALFWorld 基准测试上比纯推理方式高了 34%，WebShop 上高了 10%。效果好，但成本也摆在那儿。
适合： 探索性任务、路径不确定的任务、需要随时纠偏的场景。

---
Plan-and-Execute：先想好再做
这个模式把"规划"和"执行"彻底拆开。
先让一个 Planner 生成完整的任务计划，再让一个 Executor 按计划逐步走。走完所有步骤，任务结束。
Plan:
1. 搜索 A 公司市值
2. 搜索 B 公司市值
3. 计算差值
4. 生成对比报告

Execute: 按顺序执行 1→2→3→4
优点： LLM 调用次数少。规划只做一次，执行阶段每步不需要重新推理。ICML 2024 上的 LLMCompiler 论文报告，这种模式比 ReAct 快 3.6 倍。
缺点： 如果计划本身有问题，或者执行过程中环境变了，整个流程容易跑偏。有些实现加了"重新规划"节点来缓解，但灵活性天然弱于 ReAct。
适合： 步骤清晰、环境稳定、对成本敏感的任务。

---
Reflexion：做完回头看
2023 年普林斯顿的 Noah Shinn 等人在 NeurIPS 上提出。
机制： 在 ReAct 的基础上加了一层"反思"。内部有三个组件协作：

---
1. Actor（执行者）：干活的 Agent，按 Thought → Action → Observation 循环执行任务
2. Evaluator（评估器）：任务做完后打分——成功还是失败，过了几个测试用例
3. Self-Reflection（反思器）：拿到评估结果后，LLM 对自己的执行过程做复盘——哪一步判断错了？为什么失败？下次怎么避免？反思结论存进 episodic memory（情景记忆）
下次遇到类似任务时，Agent 从记忆里取出之前的反思笔记，作为上下文的一部分送给 LLM。
相当于给 Agent 配了一本"错题本"。

---
用代码生成的例子来说明：

任务：写一个 Python 函数，字符串去重并保持原始顺序。
第 1 次： Agent 用 set() 去重。跑测试，失败——set() 不保证顺序。
评估器： 失败。3 个测试用例过了 1 个。
反思： "我用了 set() 来去重，但 set 不保证元素顺序。下次应该用 dict.fromkeys() 或手动遍历追踪已出现字符。"
第 2 次： Agent 带着这段反思笔记重新写，用了 dict.fromkeys()。测试全过。
两次尝试，一次反思，搞定。没有反思的 Agent 可能在同一个错误上反复撞墙。
优点： 能从失败中学习。不改模型权重，只靠自然语言反馈就能让 Agent 在后续尝试中持续进步。
缺点： 每多一次尝试就多一轮完整的 Agent Loop，加上反思本身也是一次 LLM 调用。Token 消耗是三种模式里最高的。而且它依赖明确的成功/失败信号——如果任务没有清晰的评估标准，反思就无从做起。

---
生产环境的两大硬约束
本地跑个 demo 验证 Agent Loop，很简单。但上生产，有两个问题必须提前想好。
成本
Agent 每转一轮循环就是一次 LLM 调用。
Anthropic 的数据显示，Agent 的 token 消耗大约是普通聊天的 4 倍。Multi-Agent 场景下能飙到 15 倍。
SWE-Bench 和 GAIA 基准测试里，表现好的 Agent 每个任务消耗的 token 是普通对话的 10 到 50 倍。
一个不受约束的 Agent，单次任务成本能到 5-8 美元。
控制成本几个实操手段：
- 能用 Plan-and-Execute 就别用 ReAct——规划一次比每步推理便宜得多
- 给每个 Agent 运行设 token 预算上限
- 缓存工具调用结果，避免重复查询
- Prompt Cache 可以砍掉约 90% 的重复输入成本

---
可观测性
普通聊天就一个来回，出了问题一目了然。
Agent 可能跑了 15 轮循环，调了 8 个工具，中间还有分支判断。结果不对的时候，你得知道：第几轮开始跑偏的？哪个工具返回了错误数据？LLM 的推理在哪一步出了岔子？
没有结构化日志和追踪，Agent 就是黑盒。出了问题无从下手。
生产环境需要的可观测能力：
- 每一轮的推理过程、工具选择、参数、返回值，全部记录
- 支持按时间线回放整个执行过程
- 异常检测：连续多轮没进展时自动告警

---
停止条件：容易被忽视的关键设计
没有停止条件的 Agent Loop 是危险的。
Oracle 技术博客里提到过一个案例：某团队部署了一个网页爬虫 Agent。目标网站改了页面结构，爬虫工具开始返回空数据。Agent 的 prompt 写着"重试直到拿到数据"。于是它在 5 分钟内循环了 400 次，烧掉几千个 token，直到撞上平台速率限制才停下来。
如果设了最大迭代次数（哪怕只是 3 次），这事根本不会发生。

---
生产环境推荐的停止条件：
- 最大迭代次数：比如 10 轮。到了就停，不管有没有完成
- Token/成本预算：超阈值强制终止
- 无进展检测：连续 N 轮没产生新信息，退出
- 目标达成检查：每轮结束时评估任务是否完成
这几个条件叠着用，互为保底。

---
从单循环到多循环
当任务复杂度继续上升，一个 Agent Loop 也不够用了。
微软的 Magentic-One 用了双循环架构：外层循环做战略规划，内层循环做具体执行。
内层卡住了，外层可以推翻整个策略，从头再来。
Anthropic 的 Claude Research 用了编排-工作者模式：一个主 Agent 做任务分配，多个子 Agent 并行执行不同子任务。他们的内部评估显示，多 Agent 比单 Agent 表现高了 90.2%。
多循环架构的代价也很直观：Agent 越多，循环越多，token 消耗往上飙，协调复杂度也大幅增加。所以业界的共识是：先用单循环把事办了，确实不够再加。

---
Agent Loop 和传统编程的区别
初次接触 Agent Loop 的人经常会问：这跟我写个 for 循环调 API 有啥区别？
区别在控制权。
传统编程里，循环做什么、怎么做、何时停，全部由你的代码决定。逻辑是硬编码的。
Agent Loop 里，每一轮做什么、调哪个工具、传什么参数，是 LLM 在运行时动态决定的。
你写的代码只提供循环框架和工具集。具体走哪条路，是 Agent 自己"想"出来的。
所以 Agent 能处理开放性任务——你不需要提前写好所有的 if-else 分支。
代价是 Agent 也会出错——毕竟决策权交给了一个概率模型。
